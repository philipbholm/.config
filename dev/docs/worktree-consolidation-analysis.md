# Worktree Consolidation Analysis

Comparing the dotfiles approach (`dev.sh` + `gwc.sh`) with PR #1894's in-repo approach.

**Date**: 2026-04-15  
**PR**: https://github.com/ledidi-as/ledidi-monorepo/pull/1894

---

## Compatibility Issues

### 1. Port Offset (Easy fix)

| Approach | Offset Formula |
|----------|----------------|
| Dotfiles | `slot × 100` |
| PR #1894 | `worktree × 10` |

**Fix**: Change to `10×` in `dev.sh`.

Note: PR uses different base ports for some services:

| Service | Main compose | Dotfiles override base | PR base |
|---------|--------------|------------------------|---------|
| Shell | 3010 | (not included) | 3010 |
| Frontend | 3003 | 3003 | 3003 |
| Registries GQL | 4006 | 4006 | 4006 |
| Registries gRPC | 50051 | 50006 | 50006 |

The gRPC port difference (50051 vs 50006) needs verification.

### 2. Database Strategy (Incompatible)

| Approach | Strategy |
|----------|----------|
| Dotfiles | Separate postgres container + volume per worktree (`database_data_wt_{slot}`) |
| PR #1894 | Shared postgres, separate databases (`registries_wt1`, `registries-test-wt1`) |

**Decision needed**:
- Keep dotfiles approach (safer, heavier) → ignore PR's `REGISTRIES_DATABASE` pattern
- Adopt PR approach (lighter) → remove per-worktree postgres from override

### 3. Network Strategy (Incompatible)

| Approach | Strategy |
|----------|----------|
| Dotfiles | Isolated network per worktree (`default-network-wt-{slot}`) |
| PR #1894 | Shared `default-network` (external, from main compose) |

The PR's `docker-compose.worktree.yml` won't work with isolated networks — it expects to join main's network to reach admin-mock, localstack, etc.

**Decision needed**:
- Keep dotfiles approach → PR's worktree compose is unusable
- Adopt PR approach → lose network isolation, require main compose running

### 4. Compose File Strategy

| Approach | Strategy |
|----------|----------|
| Dotfiles | Generate `docker-compose.stack.yml` at runtime |
| PR #1894 | Static `docker-compose.worktree.yml` + `.env.local` with variables |

These can coexist if you choose which to use based on context.

### 5. Missing from Dotfiles Approach

- `VITE_WORKTREE_ID` for browser tab prefix (`WT1: Registries`)
- Shell service port allocation
- Storybook ports (`STORYBOOK_REGISTRIES_PORT`, `STORYBOOK_MAIN_PORT`)

### 6. Seeding Issue

`run_seed` in `dev.sh` hardcodes database name `registries`:

```bash
POSTGRES_URL="postgresql://...@postgres:5432/registries" npm run seed-icd10
```

If adopting per-worktree databases, this needs to use `registries_wt{N}`.

## Consolidation Options

### Option A: Adopt PR conventions in dotfiles tooling (Recommended)

Keep full-isolation model but align naming/offsets with the PR.

**Changes required**:

1. Change offset to `10×` in `dev.sh`
2. Add `VITE_WORKTREE_ID` to override generation
3. Add Shell/Storybook ports to override
4. Generate `.env.local` alongside `CLAUDE.local.md` so repo scripts work
5. Update seed commands to use dynamic database name
6. Ignore `docker-compose.worktree.yml` — dotfiles approach supersedes it

**Benefits**: Full isolation + compatibility with repo tooling that reads `.env.local`.

### Option B: Switch to shared infrastructure

Abandon per-worktree postgres/network, use PR's model.

**Changes required**:

1. Require main compose running
2. Use `docker-compose.worktree.yml` instead of generated override
3. Let `dev up` just run the worktree compose with correct `.env.local`

**Loses**: Isolation, automatic slot management, ability to run without main compose.

### Option C: Support both modes

Add a flag: `dev up --shared` vs `dev up --isolated` (default).

## Implementation Notes for Option A

### Change offset calculation

```bash
# In dev.sh, change:
offset=$(( resolved_slot * 100 ))
# To:
offset=$(( resolved_slot * 10 ))
```

### Add worktree ID to override

In `generate_worktree_override()`, add to `registries-frontend` environment:

```yaml
- VITE_WORKTREE_ID=${s}
```

### Add Shell and Storybook ports

```yaml
shell:
  ports: !override
    - "$(( 3010 + offset )):3010"
```

### Generate .env.local

Add to `sync_context_files()` or create new function:

```bash
cat > "$repo_root/.env.local" <<EOF
# Auto-generated by dev.sh — worktree slot $s
WORKTREE_NUM=$s
FRONTEND_PORT=$(( 3003 + offset ))
REGISTRIES_GQL_PORT=$(( 4006 + offset ))
REGISTRIES_GRPC_PORT=$(( 50006 + offset ))
SHELL_PORT=$(( 3010 + offset ))
STORYBOOK_REGISTRIES_PORT=$(( 6006 + offset ))
STORYBOOK_MAIN_PORT=$(( 6007 + offset ))
REGISTRIES_DATABASE=registries
POSTGRES_URL=postgresql://postgres:postgres@localhost:$(( 5432 + offset ))/registries-test
EOF
```

### Fix seeding

Update `run_seed()` to use the correct database based on slot:

```bash
# For main (slot 0): registries
# For worktrees: still use registries (separate postgres instance)
```

Since dotfiles approach uses separate postgres per worktree, the database name can stay `registries` — it's isolated by container, not by database name.
