# Learnings from PR #1250

This PR fixed the treatment overview page by replacing multiple separate GraphQL queries (patient, visit entries, treatment, visits) with a single consolidated `getTreatmentAssignment` query that returns all needed data.

---

## 1. Consolidate Related GraphQL Queries into a Single Endpoint

The old page made four separate queries (`useGetPatientQuery`, `useGetPatientVisitEntriesQuery`, `useGetTreatmentQuery`, `useGetTreatmentVisitsQuery`) with dependencies between them (one query needed data from another via `skip`). A reviewer praised the refactor as "much more concise with this query than many small ones." Multiple chained queries create loading waterfalls, increase complexity with conditional `skip` logic, and make error handling harder.

```graphql
# Before: 4 separate queries with skip-chains
useGetPatientQuery -> useGetTreatmentQuery(skip: !patient) -> useGetTreatmentVisitsQuery(skip: !treatment)

# After: 1 query returning everything needed
query GetTreatmentAssignment($input: GetTreatmentAssignmentInput!) {
  getTreatmentAssignment(input: $input) {
    treatmentAssignment { id patientId treatmentId startDate endDate }
    treatment { id name }
    visits { id name order repeatable }
    visitEntries { id patientId visitId registryId createdAt }
  }
}
```

**Guideline:** When a page needs related data from multiple entities, create a single backend query that returns everything the page needs rather than chaining multiple queries with skip logic on the frontend.

---

## 2. Design Backend Queries Around Frontend Use Cases

The original code fetched a full patient object just to find a treatment assignment ID, then made further queries based on that. The fix created a purpose-built `getTreatmentAssignment` endpoint scoped to exactly what the page needed. Generic "get everything" queries push filtering and assembly logic to the frontend, where it is harder to optimize and test.

**Guideline:** Design GraphQL queries to serve specific UI views rather than forcing the frontend to assemble data from multiple generic endpoints.

---

## 3. Avoid Sequential Query Dependencies with Skip Logic

The old code used `skip: !treatmentAssignment` and `skip: !treatment` to create a chain of dependent queries. This pattern causes loading waterfalls where each query must complete before the next can start, degrading perceived performance and complicating loading state management.

**Guideline:** If you find yourself using `skip` to chain GraphQL queries where one depends on another's result, consider merging them into a single query on the backend.
