# Learnings from PR #1442

Adding a "filled date" field to form data entries, allowing users to track when a form was actually filled out separately from creation/update timestamps.

---

## 1. Clarify Ambiguous Date Semantics Up Front

When adding a date field, consider whether the name clearly communicates what the date represents. In this PR, "filled date" was ambiguous -- it could mean "the date the form was digitally filled in" or "the date the original paper form was completed." When users input data retroactively (e.g., entering last week's paper forms today), the system needs to distinguish between "when data was entered" and "when the event occurred." Ambiguous naming leads to users misinterpreting the field and entering wrong data.

**Guideline:** When a domain model has multiple timestamps with overlapping semantics, name them explicitly (e.g., `dataEntryDate` vs `observationDate`) and document the distinction in the schema or UI tooltip.

---

## 2. Make Optional Parameters Non-Optional When Possible

The original code had `entryId` as optional, leading to awkward null-checking throughout the component. The refactor changed `ensureFormDataEntryExists` to always return a `string` (throwing on failure) rather than `string | undefined`. This eliminated defensive checks downstream and made error handling clearer -- the caller either gets a valid ID or an exception propagates.

```typescript
// Before: returns undefined on failure, caller must check
const ensureFormDataEntryExists = async (): Promise<string | undefined> => { ... }

// After: throws on failure, caller gets clean string
const ensureFormDataEntryExists = async (): Promise<string> => {
  if (currentFormDataEntryId) return currentFormDataEntryId;
  const result = await createFormDataEntry({ ... });
  if (!result.data) throw new Error("Failed to create form data entry");
  return result.data.createFormDataEntry.formDataEntry.id;
};
```

**Guideline:** Prefer throwing errors over returning `undefined` from helper functions when the caller cannot meaningfully continue without a result.

---

## 3. Validate Domain Event Payloads With Strict Schemas

The event schema accepted `z.string().nullable()` for `filledDate` without format validation. If an invalid date string slips through, it causes runtime errors when converting to `Date` in the projection handler. Adding a regex or format validator at the schema level catches bad data before it enters the event store.

```typescript
// Loose: any string passes
filledDate: z.string().nullable()

// Strict: only YYYY-MM-DD format passes
filledDate: z.string().regex(/^\d{4}-\d{2}-\d{2}$/).nullable()
```

**Guideline:** Validate date formats at the Zod schema level in domain events so invalid data never enters the event store.

---

## 4. Question Whether a Feature Belongs in the Current Entity

The reviewer asked "How do we differ between the date something was filled and the associated date?" -- surfacing that the new field may need a companion concept or may be solving the wrong problem. Before adding a field, verify that it addresses the actual user need and that the domain model is the right place for it, rather than building a partial solution that creates confusion later.

**Guideline:** Before adding a new field to a domain entity, validate with stakeholders that the field's semantics match the user's mental model of the data.
