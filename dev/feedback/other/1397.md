# Learnings from PR #1397

This PR added medication management functionality to the registry, enabling clinicians to assign medications to patients with ATC code tracking, dosage info, preparation forms, and administration routes.

---

## 1. Limit Database Migrations to One Per Pull Request

The PR contained multiple migrations, making it harder to reason about the schema evolution and roll back if needed. A single migration per PR keeps the migration history clean and each migration directly tied to a reviewable set of changes.

**Guideline:** Squash database migrations into a single migration per PR, and plan to reset the migration history before going live if multiple migrations accumulated during development.

---

## 2. Use Precise Enum Values Rather Than Ambiguous Ones

The "end reason" for medications included a generic "Change dosage" option. Reviewers pointed out this should be split into "Increase dosage" and "Decrease dosage" because the direction of change matters clinically and for analysis. Ambiguous enum values lead to lost information.

**Guideline:** When defining enum values that represent user actions or clinical decisions, prefer specific values (increase/decrease) over vague ones (change) to preserve the intent.

---

## 3. Document the Origin of Domain Constants and Enum Lists

The GraphQL schema included medication-specific enums (preparation forms, administration routes, dosage units) but reviewers asked where these values came from -- whether they followed FHIR or another standard. Without documentation, future developers cannot evaluate correctness or know when updates are needed.

**Guideline:** When adding domain-specific enum values or constants, include a comment noting their source (e.g., "from FHIR", "provided by clinical team") so future maintainers can trace the origin.

---

## 4. Ensure All Schema Fields Are Wired Through the Full Stack

The GraphQL schema exposed `preparationForm` and `administrationRoute` on mutations, but the resolver did not forward these fields to the use case, and the response mapper did not include them. This means the fields were silently dropped -- the API accepted them but never stored or returned them. This is a common issue in full-stack features.

**Guideline:** After adding fields to a GraphQL schema, verify they are forwarded in the resolver, stored by the use case, persisted in the database, and returned in the response mapper -- test the full round-trip.
