# Learnings from PR #1193

Fixed radio button label click targeting, date input placeholder color on uncontrolled components, and radio disabled state styling in the shared component library.

---

## 1. Use React useId for dynamically rendered form element IDs

The PR replaced a domain-specific `formElementId` with React's `useId()` hook for generating the `id` attribute connecting radio group items to their labels. A reviewer commented approvingly ("almost `use`"), noting the approach was close to the React 19 `use` hook pattern. Using `useId()` ensures unique, stable IDs that work correctly with SSR and avoid collisions when multiple instances of the same component render on a page.

**Guideline:** Use React's `useId()` hook for HTML id attributes on dynamically rendered form elements rather than using domain IDs that may collide.

---

## 2. Clean up event listeners with useEffect instead of ref callbacks

The initial implementation added an `input` event listener inside a ref callback for date inputs, which would leak listeners on re-renders. The fix was to split the logic: use a `useCallback` ref to store the node reference, then use a `useEffect` to attach/detach the listener with proper cleanup. This is the correct React pattern for imperative DOM event management.

```typescript
const combinedRef = React.useCallback(
  (node: HTMLInputElement | null) => {
    inputRef.current = node;
    if (typeof ref === "function") ref(node);
    else if (ref) ref.current = node;
  },
  [ref],
);

React.useEffect(() => {
  const node = inputRef.current;
  if (!node || !isDateInput) return;
  const updateMutedClass = () => {
    node.classList.toggle("text-muted-foreground", !node.value);
  };
  updateMutedClass();
  node.addEventListener("input", updateMutedClass);
  return () => node.removeEventListener("input", updateMutedClass);
}, [isDateInput]);
```

**Guideline:** Never add DOM event listeners inside ref callbacks; use useEffect with cleanup functions to manage imperative event listeners.

## 3. Apply cursor and disabled styles at the correct component layer

The PR applied `cursor-pointer` conditionally on the RadioGroupItem (only when not disabled) and moved disabled styling (`disabled:cursor-not-allowed`, `disabled:opacity-50`, `disabled:border-baseline-500`) into the base `radio-group.tsx` component using Tailwind's `disabled:` variant. Styling disabled states at the primitive component level ensures consistency across all consumers, while interactive cursor styles should be conditional on the enabled state.

**Guideline:** Define disabled visual states (opacity, cursor, border color) on the base primitive component, and conditionally apply interactive styles (cursor-pointer) only when the component is enabled.

---

## 4. Use Tailwind group modifiers for parent-state-dependent child styling

The PR added the `group` class to the radio item container and used `group-disabled:fill-baseline-500` on the inner circle icon to change its fill color when the parent radio is disabled. This avoids needing JavaScript state to propagate disabled styling to nested elements.

```tsx
<RadioGroupPrimitive.Item className={cn("group ...")} ...>
  <CircleIcon className="fill-primary-700 group-disabled:fill-baseline-500 ..." />
</RadioGroupPrimitive.Item>
```

**Guideline:** Use Tailwind's `group` / `group-disabled:` modifiers to style child elements based on parent state rather than passing disabled props down through multiple layers.
