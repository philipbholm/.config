# Learnings from PR #948

Initial form implementation for registries, including creating forms and adding text elements via an event-sourced backend with a React frontend.

---

## 1. Name Database Migrations Descriptively

Unnamed migrations (e.g., `20251008091359_/migration.sql`) make it hard to understand what changed when scanning the migration history. When every migration is an anonymous timestamp, debugging schema issues requires opening each file individually.

**Guideline:** Give each Prisma migration a descriptive name, and aim for one migration per PR to keep changes atomic and reviewable.

---

## 2. Remove Auto-Generated Noise Comments from Code

Generated comments (e.g., boilerplate JSDoc or ORM-generated annotations) that add no information should be removed before committing. They clutter the codebase and train readers to skip comments entirely, including the ones that matter.

**Guideline:** Strip auto-generated comments that restate the obvious, and configure generators to suppress them when possible.

---

## 3. Add Authorization Checks to Every New Endpoint

New query/mutation handlers that skip authorization checks create security gaps that are easy to miss in review. Even in early-stage features, missing auth means any authenticated user can access any resource.

**Guideline:** Every new backend handler must include an `authorize()` call before performing any data access.

---

## 4. Derive Types from Your Schema Instead of Redefining Them

Manually redefining types that already exist in Prisma (or another schema source) creates drift risk. When the schema changes, the hand-written type will silently diverge.

**Guideline:** Use `Pick` or utility types from your ORM/schema layer to derive application-level types, keeping a single source of truth.

---

## 5. Standardize UI Copy Capitalization

Inconsistent capitalization in UI strings (e.g., "Create Form" vs "Create form") looks sloppy and complicates i18n. Establishing a convention early avoids accumulating inconsistencies that are tedious to fix later.

**Guideline:** Only capitalize the first word in UI labels and dictionary entries (sentence case), unless it is a proper noun.

---

## 6. Use Specific Error Types for Domain Failures

Throwing a generic authorization error when the real problem is "resource not found" misleads both developers debugging issues and API consumers handling errors. Distinct error types enable correct client-side handling and clearer logging.

**Guideline:** Throw `NotFoundError` for missing resources and `NotAuthorizedError` for permission failures -- never conflate the two.

---

## 7. Write Tests for Every New Query/Mutation

Shipping a new endpoint without tests means regressions go undetected until production. When a reviewer asks "No tests for this one?" that is a signal the feature is incomplete, not that the reviewer is being pedantic.

**Guideline:** Every new backend operation must ship with at least one integration test covering the happy path and one covering the primary error case.

---

## 8. Make Tests Verify Distinct Behavior

Duplicate test cases that exercise the same code path provide false confidence. Each test should verify a distinct scenario -- if two tests are identical, one should be replaced with a more interesting assertion (e.g., testing ordering when adding multiple elements at the same index).

**Guideline:** Before writing a new test case, confirm it exercises behavior not already covered by existing tests.

---

## 9. Remove Unused Imports and Dead Code Before Review

Leftover unused imports (e.g., `'useRegistryId' is defined but never used`) signal that the code was not cleaned up before opening the PR. They add confusion about intent and will eventually trigger linter warnings in CI.

**Guideline:** Run the linter locally and resolve all warnings before marking a PR as ready for review.

---

## 10. Treat Console Statements as Errors in Production Code

Bare `console.log` calls left in production code are debug artifacts. Use `console.error` for genuinely unexpected situations that warrant alerting, and remove everything else. Static analysis tools will flag these, but catching them in review is faster.

**Guideline:** Replace `console.log` with proper error reporting (`console.error` or an alerting service) or remove it entirely before merging.
