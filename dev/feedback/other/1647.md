# Learnings from PR #1647

This PR adds the ability to copy form data from a previous patient event entry to a new one within repeatable registry events.

---

## 1. Validate Authorization on Both Source and Target in Copy/Transfer Operations

When an endpoint copies or moves data between two entities, the authorization check must cover both the source (read) and the target (write). In this PR the authorize step only checked write permission on the target registry, meaning a user could potentially copy form data from a registry they should not have read access to, simply by knowing the source entry ID.

**Guideline:** Any operation that reads from one entity and writes to another must enforce permissions on both sides -- read on the source, write on the target.

---

## 2. Wrap Multi-Step Mutations in a Transaction

The copy operation dispatched a `PATIENT_EVENT_ENTRY_CREATED` event and then a `FORM_DATA_ENTRIES_COPIED` event as separate event-store calls. If the second call fails, the system is left in an inconsistent state: an empty event entry exists without the copied form data. Wrapping these in a single transaction ensures atomicity.

```typescript
// Problem: two separate createAndPublishEvent calls
await dependencies.eventStore.createAndPublishEvent({ ... }); // creates entry
await dependencies.eventStore.createAndPublishEvent({ ... }); // copies forms
// If second fails, orphaned entry exists
```

**Guideline:** When a single user action triggers multiple state-changing events, wrap them in a transaction so the operation either fully succeeds or fully rolls back.

---

## 3. Limit Domain Events to One Per Endpoint Call

Dispatching multiple domain events from a single endpoint makes downstream projections harder to reason about and can introduce ordering dependencies. One reviewer raised concern about dispatching both `PATIENT_EVENT_ENTRY_CREATED` and `FORM_DATA_ENTRIES_COPIED` from a single endpoint. A single composite event (e.g., `FORM_DATA_ENTRY_COPIED_WITH_NEW_EVENT_ENTRY`) can be cleaner and avoids ambiguous intermediate states.

**Guideline:** Prefer one domain event per endpoint invocation; if multiple side effects are needed, consider a single composite event or discuss the tradeoff before merging.

---

## 4. Surface Errors to the User, Not Just the Console

When an async operation (like copying form data) fails on the frontend, silently catching the error and only logging to the console gives the user no indication that something went wrong. They may assume the copy succeeded and continue working with incomplete data.

**Guideline:** Always show user-facing feedback (toast, alert, inline error) when a mutation fails, in addition to any background error logging.

---

## 5. Log Unexpected Errors to Sentry for Observability

When catching errors in frontend code, a bare `console.error` is invisible in production. Calling `captureException` (Sentry) ensures the team gets alerted to real failures.

**Guideline:** In catch blocks for unexpected errors, call your error-reporting service (e.g., Sentry `captureException`) rather than relying solely on `console.error`.

---

## 6. Keep Test Mocks Minimal -- Only Mock What You Assert On

In a test file, the entire `@sentry/nextjs` module was mocked when only `captureException` was needed for assertions. Broad mocks can hide real failures if the module's behavior changes, and they make tests harder to understand because the reader does not know which parts of the mock actually matter.

```typescript
// Too broad
vi.mock("@sentry/nextjs");

// Better: only mock what you need
vi.mock("@sentry/nextjs", () => ({
  captureException: vi.fn(),
}));
```

**Guideline:** Mock only the specific exports you need for assertions; let everything else use the real implementation or fail visibly.

---

## 7. Assert on Specific Error Types in Failure Tests

Tests that assert only `expect(...).rejects.toThrow()` without specifying the error class or message pass even when the wrong error is thrown (e.g., a database connection error instead of a validation error). This makes the test less reliable as a regression guard.

**Guideline:** When testing error paths, assert on the specific error type or message (e.g., `rejects.toThrow(NotAuthorizedError)`) to ensure the right failure is caught.

---

## 8. Verify Outcomes, Not Just Non-Failure, in Integration Tests

A test that only checks the copy operation does not throw misses bugs where the operation "succeeds" but produces incorrect data. Adding assertions that the copied data matches the source (answers, missing-data reasons, etc.) catches data-corruption bugs.

**Guideline:** Integration tests should assert on the resulting state (data correctness), not just the absence of errors.

---

## 9. Prefer Self-Contained Components Over Prop Drilling

The episode page accumulated many props being passed through multiple component layers (`onCopySuccess`, `isEventRepeatable`, `creatingEventEntryForEventId`, etc.). This makes the component tree fragile and hard to refactor. Self-contained components that fetch their own data or use context are more maintainable.

**Guideline:** When a component requires many pass-through props, consider refactoring to use context, hooks, or self-contained subcomponents to reduce coupling.

---

## 10. Return Minimal API Responses Unless the Client Needs More

The GraphQL mutation for copying form data returns the full `PatientEventEntry` and all `FormDataEntry` objects. A reviewer questioned whether just returning `success: true` would suffice. Returning more data than the client needs increases payload size and couples the API contract to internal representations.

**Guideline:** Return only what the calling client needs from a mutation; if the client can refetch or already has the data, prefer a minimal success response.
