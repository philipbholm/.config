# Learnings from PR #1379

This PR implemented a new data import flow supporting CSV file upload to S3 with streaming into the project, including cron-based job processing, frontend import dialogs for patients and episodes, and backend pagination.

---

## 1. Keep One Use Case Per Folder

When multiple use cases share a folder, the folder bloats quickly and becomes harder to navigate. Each use case should live in its own folder with its own test file, making the codebase easier to scan and reducing merge conflicts.

**Guideline:** Place each use case in a dedicated folder -- do not bundle multiple use cases into one directory, even if other parts of the codebase do it differently.

---

## 2. Extract Multi-Step Dialog Components into Separate Files

Large dialog components with multiple steps (e.g., file upload, column mapping, confirmation) become unreadable when all steps live in one file. Each step should be a separate component in its own file, encapsulating its own logic and dictionary entries. This also keeps translation dictionaries a manageable size.

**Guideline:** When a dialog has distinct steps, extract each step into its own component file with co-located translations, then compose them in a parent orchestrator component.

---

## 3. Colocate Pagination Logic with the Component That Uses It

Pagination state was initially defined in a separate file from the component that consumed it (PatientList). Keeping state and its usage in different files makes the data flow harder to follow and increases cognitive load.

**Guideline:** Move pagination state and logic into the component that renders the paginated data, rather than keeping it in a parent page component.

---

## 4. Combine Related Database Queries Instead of Making Separate Calls

A separate query was created to get the total count for pagination, but this belongs as part of the existing `getPatients` query. Combining them into a single database round-trip is simpler and more efficient.

**Guideline:** When adding pagination, include the total count as part of the primary data query rather than introducing a separate count query.

---

## 5. Use Semantic HTML Instead of Generic Span Elements

AI-generated code tends to overuse `<span>` elements where semantic HTML (`<p>`, `<h2>`, `<time>`, `<address>`, etc.) would be more appropriate. Semantic HTML improves accessibility, SEO, and code readability.

**Guideline:** Prefer semantic HTML elements over `<span>` and `<div>`, and consider adding a linting rule to flag excessive use of non-semantic container elements.

---

## 6. Commit to a File Naming Convention and Enforce It with Lint Rules

The team internally decided on kebab-case for new files, but the codebase was inconsistent (some Pascal, some kebab). This led to a back-and-forth discussion in the PR. Stylistic decisions should be encoded in lint rules so they are enforced automatically and do not consume review time.

**Guideline:** Encode file naming conventions (e.g., kebab-case) in lint rules so they are enforced automatically rather than debated in every PR.

---

## 7. Clean Up Dead Code and Unused Files

An import projection file was left in the codebase even though it was no longer used. Stale files confuse future readers and AI tools about what is active code.

**Guideline:** When refactoring replaces a file's functionality, delete the old file in the same PR rather than leaving it for later cleanup.

---

## 8. Write Integration Tests for Pagination

When adding pagination to an existing query, the change in behavior (skip, take, total count) should be covered by integration tests. Without them, regressions in offset calculation or count logic can go unnoticed.

**Guideline:** When adding pagination to a data query, add integration tests that verify correct page slicing and total count.

---

## 9. Test at the Resolver Level for gRPC Endpoints

gRPC is more loosely typed than GraphQL, making it easier to accidentally drop fields or misconfigure schemas. Testing at the resolver level (not just the use case level) catches mapping errors early -- for example, a zod schema mismatch or a forgotten field in the protobuf-to-domain mapping.

**Guideline:** When introducing gRPC endpoints with zod validation, write resolver-level integration tests to verify the full request/response mapping, not just the underlying use case.

---

## 10. Write Unit Tests for Data Parsing Helpers Even When Time Is Short

Helper functions that parse CSV rows into domain types (handling type coercion, BOM stripping, etc.) are easy to break and hard to debug in integration tests. Spending 10 minutes generating unit tests documents how the helpers work and prevents regressions when the parsing logic evolves.

**Guideline:** Always write unit tests for data parsing and transformation helpers -- they are cheap to write and expensive to debug when broken.
