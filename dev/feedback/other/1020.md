# Learnings from PR #1020

Added patient diagnoses feature with ICD-10 code seeding, patient CRUD operations, and diagnosis management UI.

---

## 1. Use Event Sourcing for All Data Manipulation

Direct database writes bypass the audit trail and make it impossible to reconstruct state history. All create, update, and delete operations on domain entities should go through event sourcing rather than direct Prisma calls, because the system needs a reliable record of every state change for compliance and debugging.

**Guideline:** Never use direct database mutations for domain entities; always emit domain events and let projections handle persistence.

---

## 2. Use Enums for Constrained String Fields in the Schema

When a database field can only hold a known set of values (e.g., diagnosis status), using a plain `String` type loses that constraint at the data layer. Prisma enums enforce valid values at the database level and generate TypeScript types that catch invalid assignments at compile time.

**Guideline:** Define a Prisma `enum` for any field with a fixed set of valid values instead of using a bare `String` type.

---

## 3. Report Unexpected Errors to Monitoring

Catching errors in the UI and only showing a toast to the user means unexpected failures go unnoticed by the team. When an error is not a known, expected failure mode, it should be forwarded to the monitoring service (e.g., Datadog) so it can be investigated.

**Guideline:** In error handlers, distinguish between expected errors (show user feedback) and unexpected errors (also report to Datadog).

---

## 4. Extract Reusable Hooks for Common UI Patterns

Inline debounce logic (e.g., `setTimeout` + `clearTimeout` inside a component) is repeated boilerplate that obscures the component's intent. Extracting it into a `useDebouncedValue` hook makes the debounce behavior testable independently and keeps components focused on rendering.

**Guideline:** When a side-effect pattern like debouncing appears inline, extract it into a custom hook and pass the derived value to the query.

---

## 5. Use Semantic HTML Elements for Text Content

Using `<span>` or `<div>` for paragraph text hurts accessibility and SEO. Screen readers and browser default styles rely on semantic elements like `<p>` to convey document structure. This applies broadly to any text-heavy display components.

**Guideline:** Use `<p>` tags (and other semantic HTML elements) for text content rather than generic `<span>` or `<div>` wrappers.

---

## 6. Prefer Specific Types Over Loose Union Types

Declaring state as `ApolloError | undefined` when a more specific type or simply using the generic parameter of `useState<ApolloError>()` would suffice introduces unnecessary complexity. The generic parameter already handles the `undefined` initial state.

**Guideline:** Use `useState<T>()` (which implicitly allows `undefined`) rather than `useState<T | undefined>()`.

---

## 7. Name Mapper Functions for Composability

A function named `mapPrismaPatientToPatient` reads awkwardly when passed as a callback to `.map()` (resulting in `patients.map(mapPrismaPatientToPatient)`). Naming it without the `map` prefix (e.g., `prismaPatientToPatient`) reads more naturally and composes better as a callback.

**Guideline:** Name conversion/mapper functions as `sourceToTarget` (not `mapSourceToTarget`) so they read cleanly when passed to `.map()`.
