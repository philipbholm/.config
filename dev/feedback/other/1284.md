# Learnings from PR #1284

This PR added a "Patient data form" feature to registries, allowing storage of arbitrary data on patients independent of any visit/event.

---

## 1. Place Shared Test Fixtures in the Outermost Applicable Setup Hook

When test setup like `createSingleSiteRegistry()` is used by all tests in a describe block and does not need to vary between tests, place it in `beforeAll` rather than inside individual tests or in an ad-hoc location. This avoids redundant setup work, reduces test runtime, and makes the test structure easier to follow.

**Guideline:** Hoist shared, invariant test fixtures to `beforeAll`; reserve `beforeEach` and inline setup for state that must differ between tests.

---

## 2. Use Test Isolation to Prevent Cross-Test Interference

When multiple test cases modify shared state (e.g., writing data to a registry), each test should ideally operate on its own isolated instance. Creating a fresh registry per test (via `beforeEach`) prevents one test's side effects from causing flaky failures in another. The tradeoff is speed vs. reliability -- use `beforeAll` for read-only shared fixtures and `beforeEach` for mutable state.

```typescript
// Good: mutable state isolated per test
beforeEach(async () => {
    registry =
        await testApplicationSetup.application.createSingleSiteRegistry({
            context: mockContext({ userId }),
            input: { name: "Test Registry" },
        });
});
```

**Guideline:** Create mutable test fixtures in `beforeEach` to guarantee test isolation, and reserve `beforeAll` for expensive, read-only setup like building the test application.

---

## 3. Nest Setup Hooks Inside Describe Blocks for Scoped Fixtures

When a fixture is needed by a subset of tests within a larger describe block, place it inside a nested `describe` with its own `beforeAll` or `beforeEach` rather than at the top level. This keeps unrelated tests from depending on that fixture and makes the test's preconditions explicit.

**Guideline:** Scope test setup to the narrowest describe block that needs it, using nested `beforeAll`/`beforeEach` to keep test dependencies clear and minimal.
