# Learnings from PR #778

Replaced the Datadog dd-trace library with OpenTelemetry (OTEL), enabled source maps, and added commit SHA/repo metadata to the admin service.

---

## 1. Prefer Uploading Source Maps Over Enabling Them in Production

When enabling source maps for better stack traces in production, consider uploading them to your error tracking/APM service (e.g., Datadog, Sentry) rather than using `--enable-source-maps` in the Node.js runtime. For backend services the performance impact of `--enable-source-maps` is minimal, but for frontend builds it exposes source code to end users. Uploading source maps keeps them off production servers while still providing readable stack traces in your monitoring tools.

**Guideline:** For backend Node.js services, `--enable-source-maps` is acceptable; for frontend bundles, always upload source maps to the APM/error-tracking service instead of shipping them to clients.

---

## 2. Document the Rationale When Reverting a Technical Decision

This PR removed OTEL and re-added dd-trace (the reverse of what the title suggests when read alongside PR #799 which added dd-trace back). When a technical decision is reversed shortly after being made, the PR body should clearly explain why the original approach was insufficient. In this case, the body noted "we need a lot more than OTEL has: sourcemap, system metrics, connect infra" -- but that context was on the follow-up PR, not this one. Reviewers benefit from seeing the decision rationale inline.

**Guideline:** When reverting or replacing a recently-introduced approach, explain in the PR description what was tried, why it fell short, and what the replacement provides that the original did not.

---

## 3. Scope Observability Changes to Avoid Cross-Cutting Surprises

This PR combined three concerns: removing dd-trace, enabling source maps, and adding commit SHA/repo metadata. Each of these has different risk profiles and review needs. Bundling unrelated observability changes makes it harder for reviewers to assess each change independently and increases the blast radius if something goes wrong in production.

**Guideline:** Keep observability changes (tracing library swaps, source map config, metadata injection) in separate PRs when they affect different runtime behaviors, so each can be reviewed, deployed, and rolled back independently.
