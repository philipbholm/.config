# Learnings from PR #1050

This PR added new shared UI components (Input, PasswordInput, ButtonGroup, Select, etc.) and removed the old input component.

---

## 1. Document the Boundary Between Component Layers

When a codebase has multiple component directories (e.g., `components/` for composed application-level components and `ui/` for primitive building blocks), reviewers will ask what belongs where. Without explicit documentation, contributors will place components inconsistently, leading to confusion about which directory to import from and duplication of concerns.

**Guideline:** Add a brief README or comment in each component directory explaining its purpose, abstraction level, and when to use it versus the other directories.

---

## 2. Prefix Boolean Props with `is` or `has`

Props like `optional`, `loading`, and `asChild` were flagged for not following a boolean naming convention. Unprefixed boolean props are ambiguous -- `optional` could be read as a noun or adjective, while `isOptional` is unambiguously a boolean state. Consistent naming makes component APIs self-documenting and reduces guessing at call sites.

**Guideline:** Always prefix boolean props with `is` or `has` (e.g., `isOptional`, `isLoading`, `hasError`) to make their type and meaning immediately clear.

---

## 3. Use Stable Keys Instead of Array Indexes in List Components

The password requirements list used `key={index}`, which is fragile when items can be reordered, added, or removed. React uses keys to track identity across renders; index-based keys cause incorrect state preservation and unnecessary re-renders when the list changes.

**Guideline:** Use a stable, unique identifier from your data as the `key` prop; only fall back to index when the list is truly static and will never be reordered.
