# Learnings from PR #938

Added a build-and-publish CI workflow for the registries service, including Docker image build, ECR push, and environment publish steps.

---

## 1. Do Not Configure Deployment Targets That Do Not Exist Yet

The initial workflow included a `publish-prod` job even though there was no production environment available for the registries service at that time. Including deployment steps for non-existent environments creates false expectations, can cause confusing CI failures, and adds dead code that must be maintained. It is better to add deployment targets incrementally as environments are actually provisioned.

```yaml
# Removed: publish-prod job that targeted a non-existent environment
publish-prod:
    name: "Publish to Production"
    needs: build-and-push
    if: github.ref == 'refs/heads/main' || github.ref == 'refs/heads/master'
    uses: ./.github/workflows/publish.yml
    with:
        service: "registries-service"
        image: ${{ needs.build-and-push.outputs.image }}
        environment: "prod"
```

**Guideline:** Only add CI/CD deployment jobs for environments that are actually provisioned and ready to receive deployments.

---

## 2. Copy Workflow Patterns Carefully When Bootstrapping New Services

When creating a build-and-publish workflow for a new service by copying from an existing one, review every section for applicability rather than copying wholesale. The prod deployment block was likely carried over from another service's workflow without considering that this new service did not yet have a prod environment. Boilerplate copying without adaptation leads to configuration drift and broken pipelines.

**Guideline:** When copying CI workflow templates for new services, audit each job and conditional to confirm it applies to the new service's current infrastructure state.
