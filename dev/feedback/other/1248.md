# Learnings from PR #1248

This PR added a registry import feature that parses Excel files on the frontend and creates registries with variables, treatments, visits, forms, and patient data via a new GraphQL mutation.

---

## 1. Validate GraphQL Inputs with Zod on the Backend

A reviewer asked "zod?" when looking at the GraphQL resolver that accepts complex nested import data. Unvalidated resolver inputs allow malformed or malicious data to reach business logic, causing cryptic downstream errors.

**Guideline:** Always validate GraphQL mutation inputs with a schema validator (e.g., Zod) at the resolver layer before passing them to application logic.

---

## 2. Vet Dependencies for Security and Maintenance Status

The `xlsx@0.18.5` library was flagged for a prototype pollution vulnerability, and it was discovered that the maintainers no longer publish fixes to npm. Using unmaintained or vulnerable libraries creates security debt that compounds over time.

**Guideline:** Before adding a new dependency, check its vulnerability status, maintenance cadence, and npm release history -- prefer actively maintained alternatives.

---

## 3. Keep Test Setup Files Minimal and Explicit

A reviewer expressed concern about test setup files with heavy default behavior being overridden, noting they become "unmaintainable over time." Global test setup that silently polyfills or mocks broad behavior makes tests fragile and hides what is actually being tested.

**Guideline:** Only add global test setup (polyfills, matchers) when components genuinely require it, and document why each entry is necessary to prevent unchecked growth.

---

## 4. Distinguish Expected Errors from Unexpected Errors in Catch Blocks

A reviewer noted that bare `catch` blocks that silently swallow all errors make it impossible to detect unexpected failures. When handling errors, you should differentiate between errors the user caused (e.g., bad file format) and errors that indicate bugs (e.g., unexpected server failures), routing the latter to monitoring.

**Guideline:** In catch blocks, log unexpected errors to your monitoring service (e.g., Datadog) while showing user-friendly messages for expected failures.

---

## 5. Align Infrastructure Changes Across Services

When wrapping the event store's `createAndPublishEvent` in a transaction, a reviewer asked "why we need this? should be aligned with trials afaict." Diverging infrastructure patterns between services (e.g., registries vs. trials) creates maintenance burden and subtle behavioral differences.

**Guideline:** When modifying shared infrastructure patterns (event stores, transaction handling), check how the equivalent code works in sibling services and keep them aligned unless there is a documented reason to diverge.

---

## 6. Preserve Trailing Newlines in Files

A reviewer flagged a diff that removed a trailing newline from a `.graphql` file. Missing trailing newlines cause unnecessary diff noise in future commits and can trigger linter warnings.

**Guideline:** Ensure files end with a single trailing newline, and configure your editor or linter to enforce this automatically.
