# Learnings from PR #1508

Large-scale rename of domain concepts across ~277 files, including frontend components, GraphQL queries, Prisma schemas, and backend services.

---

## 1. Remove Redundant Type Annotations That Can Be Inferred

When TypeScript can infer a type from context (e.g., from a function return type or a generic parameter), explicitly specifying it adds noise without adding safety. Reviewers will flag redundant annotations as clutter, and they increase the surface area for future type mismatches during refactors.

**Guideline:** Omit explicit type annotations when the compiler can infer the type; run the linter to confirm inference works before removing them.

---

## 2. Isolate Formatting-Only Changes Into Separate PRs

When a tool (like Prisma's formatter) reformats an entire file with different indentation rules, mixing that diff with substantive code changes makes the PR nearly impossible to review. The reviewer cannot distinguish real logic changes from whitespace noise. This slows review and increases the risk of real bugs slipping through.

```
# Bad: one PR with rename + full-file reformat
services/registries/prisma/schema.prisma  (hundreds of indentation changes mixed with real renames)

# Good: two PRs
PR 1: format schema.prisma (pure whitespace, trivial to approve)
PR 2: rename concepts in schema.prisma (only meaningful changes visible)
```

**Guideline:** When a formatter rewrites an entire file, submit the formatting as a standalone PR and merge it before your feature branch to keep the feature diff clean.

---

## 3. Understand Why Tool-Imposed Formatting Differs Before Working Around It

Prisma's built-in formatter enforces 2-space indentation with no configuration option, similar to Go's `gofmt`. Before fighting the tool or introducing workarounds, understand whether the formatting difference is intentional and whether the team should simply adopt the tool's convention. If the team insists on a different convention, use a dedicated plugin (e.g., `prettier-plugin-prisma`) rather than manual overrides that will drift.

**Guideline:** When a tool enforces an opinionated format, either adopt that convention project-wide or set up a plugin to override it consistently -- never rely on manual formatting.
