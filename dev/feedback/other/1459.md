# Learnings from PR #1459

Adding inline form rename functionality for both single-site and multi-site registries, with uniqueness validation within visits.

---

## 1. Verify Authorization Works Across All Registry Types

The initial implementation wired `renameRegistryForm` only to the `singleSiteAuthorizationService`, but the multi-site registry UI also called this mutation. Multi-site admins have different permissions (`multi-site-registries.*`), so their rename requests would be silently rejected as unauthorized. This was caught during review and fixed by adding a separate `renameMultiSiteRegistryForm` use case with the correct authorization service.

**Guideline:** When adding a new mutation, check whether the feature applies to multiple entity types (e.g., single-site vs multi-site registries) and wire authorization for each one.

---

## 2. Use Natural, Idiomatic Translations

A reviewer suggested changing `"Skjemanavn allerede i bruk"` to be more natural Norwegian, and correcting the English to `"Form name already in use"` rather than `"Form name already exists"`. Translation strings should read naturally in each language rather than being mechanical translations. Poor translations erode user trust.

**Guideline:** Have native speakers review UI translation strings, and prefer natural phrasing over literal translations from the primary language.

---

## 3. Enforce Domain Uniqueness Constraints at the Use Case Level

The rename use case validates that no other form in the same visit has the same name (case-insensitive). This validation happens in the `run` method before publishing the event, throwing a `FailedPreconditionError` if a duplicate is found. The frontend then maps this specific error code to a user-friendly message. This pattern keeps the domain logic in the backend while giving the frontend enough information to display contextual errors.

```typescript
// Backend: check uniqueness before persisting
const duplicateForm = formsInVisit.find(
  (f) => f.name.toLowerCase() === input.name.toLowerCase() && f.id !== input.formId,
);
if (duplicateForm) {
  throw new FailedPreconditionError(`A form with the name "${input.name}" already exists`);
}

// Frontend: map error code to user message
if (isFailedPreconditionError(apolloError)) {
  setRenameError(translation.formNameAlreadyExists);
}
```

**Guideline:** Enforce uniqueness constraints in the use case layer (not just the database) and use typed error codes (like `FAILED_PRECONDITION`) so the frontend can display specific, actionable error messages.
