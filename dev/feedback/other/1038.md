# Learnings from PR #1038

Added unauthenticated GraphQL endpoints and a public domain-check query to the admin service, plus language header support in the context.

---

## 1. Use Separate Context Types for Authenticated vs. Unauthenticated Use Cases

When adding unauthenticated endpoints, making the `authentication` property nullable on the context object means every authenticated use case must now handle a potentially-missing auth object. Instead, define separate context types (`AuthenticatedContext` and `UnauthenticatedContext`) so that authenticated use cases can rely on auth being present without null checks.

```typescript
// Prefer this:
return async (args: { context: UnauthenticatedContext; input: Input }) => { ... }

// Over making authentication optional in a shared context type
```

**Guideline:** Use distinct context types for authenticated and unauthenticated flows so that type safety guarantees the presence of auth data where it is required.

---

## 2. Use Separate Routes for Public Endpoints Instead of Modifying Auth Logic

Rather than adding conditional authentication checks within a single route handler, create a dedicated public route (e.g., `/public/graphql`) with its own Apollo Server instance. This keeps the auth middleware clean and avoids accidental exposure of authenticated endpoints. Framework-level middleware (e.g., Fastify middleware/hooks) can enforce authentication based on route path.

```typescript
app.route({
    url: "/public/graphql",
    method: ["GET", "POST"],
    handler: fastifyApolloHandler(publicServer, { ... })
});
```

**Guideline:** Introduce a separate route and server instance for unauthenticated endpoints rather than making authentication optional on existing routes.

---

## 3. Leverage Framework Middleware for Route-Based Concerns

Fastify (and similar frameworks) provide middleware/hook mechanisms that can be applied per-route. Using these for cross-cutting concerns like authentication keeps handler code focused on business logic and makes the auth boundary visible in the routing configuration rather than buried in use case code.

**Guideline:** Apply authentication and other cross-cutting concerns through framework middleware scoped to route paths, not through conditional logic inside handlers.
