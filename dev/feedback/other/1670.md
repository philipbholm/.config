# Learnings from PR #1670

Refactored the diagnosis model from a registry-level concept to a patient-level concept, adding new clinical fields (clinicalStatus, severity, onsetDate, etc.), an edit mutation, and reusable form components while removing the old registry-level diagnosis management.

---

## 1. Use GraphQL-Generated Types Instead of Manually Defining Enums on the Frontend

When the backend schema defines enums (e.g., `ClinicalStatus`, `VerificationStatus`, `Severity`), the frontend should import the generated TypeScript types from the GraphQL codegen output rather than re-declaring them locally. Manually duplicating enum definitions creates a maintenance burden: if the backend adds or renames a value, the frontend copy silently drifts out of sync, leading to runtime mismatches or missing options in select components.

**Guideline:** Always import enum types from GraphQL-generated code; only define the display-value mapping (e.g., label arrays for select options) locally.

---

## 2. Pass Prisma Update Payloads Directly Without Redundant Undefined Checks

AI-generated code frequently wraps each field in `field !== undefined ? field : undefined` guards before passing data to Prisma. This is unnecessary because Prisma treats `undefined` fields as "do not update" by default. The extra conditionals add noise, obscure intent, and make projections harder to review for actual logic.

```typescript
// Bad: redundant guards
await prisma.diagnosisAssignment.update({
  where: { id },
  data: {
    clinicalStatus: event.payload.clinicalStatus !== undefined
      ? event.payload.clinicalStatus
      : undefined,
    severity: event.payload.severity !== undefined
      ? event.payload.severity
      : undefined,
  },
});

// Good: pass payload directly
await prisma.diagnosisAssignment.update({
  where: { id },
  data: event.payload,
});
```

**Guideline:** When updating Prisma records with optional fields, pass the payload object directly -- Prisma already ignores `undefined` properties.

## 3. Reset React Forms Using the `key` Prop Instead of `useEffect`

When a dialog or form needs to reset its state (e.g., after closing and reopening with different data), prefer setting a `key` prop on the form component from the parent rather than calling `reset()` inside a `useEffect`. The `key` approach is idiomatic React -- it unmounts and remounts the component, guaranteeing a clean slate without coordinating effect dependencies or risking stale closures.

**Guideline:** Reset form components by changing their `key` prop from the parent rather than wiring up `useEffect`-based reset logic inside the form.

---

## 4. Use Zod at Trust Boundaries, Not for Internal Domain Rules

Zod excels at validating data crossing trust boundaries: API request bodies, form input, environment variables, external API responses, and untyped JSON. For internal domain logic between tightly coupled TypeScript modules (e.g., a function like `isEndDateAllowedForStatus`), TypeScript's compile-time checks are sufficient. However, if a constraint acts more like a type invariant (e.g., polymorphic data that Prisma cannot model, email format validation), embedding it in a Zod schema ensures it is always enforced at parse time rather than relying on callers to remember to invoke a standalone validation function.

**Guideline:** Place validation in Zod schemas when data enters from an untrusted source or when runtime type safety is needed beyond what TypeScript and Prisma provide; use plain functions for pure business rules between your own modules.

---

## 5. Add Error Handling to All Mutation Dialogs Consistently

When building a new mutation dialog (create, update, delete), always include error handling for the mutation call -- typically a try/catch with user-facing error feedback (toast, inline message, etc.). If the existing create dialog has error handling but the new edit dialog omits it, users get a silent failure on update errors, which is worse than a visible error message.

**Guideline:** Every mutation dialog must have error handling that surfaces failures to the user, matching the error-handling pattern used by sibling dialogs in the same feature.

---

## 6. Ensure Select Inputs Support Deselection When the Field Is Optional

When a field is optional (e.g., `verificationStatus`), the corresponding `<select>` component must include a way to clear the selection -- either a blank/placeholder option or an explicit "None" choice. Without this, once a user picks a value, they cannot revert to "not set," effectively making an optional field required after first interaction.

**Guideline:** For optional fields rendered as select inputs, always include a deselectable empty/placeholder option so users can clear their selection.

---

## 7. Add Authorization Boundary Tests for Multi-Tenant Endpoints

When an endpoint operates on a resource scoped to a parent entity (e.g., a patient's diagnosis within a specific registry), add integration tests that verify a caller cannot access or mutate resources belonging to a different parent. Without cross-boundary tests, authorization regressions can silently expose data from other tenants.

**Guideline:** For every new endpoint that acts on a scoped resource, include at least one test asserting that access to a resource under a different parent/tenant is rejected.

---

## 8. Extract Shared Translation Keys When Multiple Components Use the Same Labels

When several components within a feature (e.g., a list view, a create dialog, and an edit dialog) all need the same translated labels (column headers, status names, field labels), extract those translations into a shared file rather than duplicating them in each component. Duplication leads to inconsistent wording when one copy is updated but others are missed.

**Guideline:** If three or more components share the same translation keys, extract them into a dedicated shared translations file for that feature domain.
