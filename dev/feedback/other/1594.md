# Learnings from PR #1594

Redesigning the registry sidebar: removing the name header, moving settings to the bottom, adding a home tab, flattening sub-menus, and persisting sidebar state across refreshes.

---

## 1. Use localStorage (or useLocalStorage) Instead of Cookies for Client-Side UI State

Storing sidebar open/collapsed state in a cookie is unnecessary when the value is only needed client-side. Cookies are sent with every HTTP request, adding overhead. Use `localStorage` via a hook like `useLocalStorage` from `@mantine/hooks` for clean, SSR-safe persistence of UI preferences.

**Guideline:** Use `useLocalStorage` (from @mantine/hooks or similar) for persisting UI state like sidebar collapse, and avoid cookies for client-only preferences.

---

## 2. Name Event Handlers Descriptively, Not Generically

A handler named `handleClick` tells the reader nothing about what will happen. Naming it `toggleSidebarOnBackgroundClick` makes the behavior obvious when reading `onClick={toggleSidebarOnBackgroundClick}` in isolation, without needing to look up the implementation.

**Guideline:** Name event handlers to describe their effect (e.g., `toggleSidebarOnBackgroundClick`) rather than using generic names like `handleClick`.

---

## 3. Ensure Components Forward All Props When Used With Radix Composition

When a component (like a custom `Link`) is used inside Radix UI's `asChild` / `Slot` pattern, it must forward all props (data attributes, event handlers, refs) to its rendered DOM element. If it only accepts a whitelist of props, Radix features like tooltips and focus management will silently break because their event handlers get dropped.

```typescript
// Before: only className, href, ref forwarded -- tooltips broken
// After: ...props spread onto RouterLink -- all Radix features work
export const Link = ({ children, href, onClick, ...props }: Props) => {
  return <RouterLink to={href} {...props}>{children}</RouterLink>;
};
```

**Guideline:** Components used with Radix `asChild` must spread all remaining props onto the underlying DOM element to preserve composition behavior.

---

## 4. Consider Misclick Risk When Adding Click-to-Collapse Regions

Adding "click sidebar background to collapse" can be convenient but risks accidental collapses if the clickable region is too close to interactive elements. Guard against this by checking that the click target is not an interactive element before toggling, and be prepared to revert if user feedback indicates accidental triggers.

**Guideline:** When adding click-to-dismiss/collapse behavior on large areas, explicitly exclude interactive child elements from the trigger and gather user feedback before committing to the pattern.

---

## 5. Write Tests for Shared UI Infrastructure Components

Components like `Link` that are foundational to the app (used everywhere, relied upon by Radix composition) should have dedicated tests. When a subtle bug in prop forwarding silently breaks tooltips across the entire app, tests would catch it early.

**Guideline:** Write focused tests for shared infrastructure components (Link, navigation primitives) that verify prop forwarding, ref handling, and integration with the composition patterns they participate in.
