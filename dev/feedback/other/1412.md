# Learnings from PR #1412

Added numeric aggregation form elements (AVG, COUNT, LATEST) for registry forms, spanning backend use cases, Prisma schema, GraphQL API, and frontend UI components.

---

## 1. Spell Out Abbreviations in Enum Values

A reviewer pointed out that `AVG` in the Prisma schema enum would be clearer as `AVERAGE`. Enum values are part of the domain vocabulary and appear in APIs, database records, and UI code. Abbreviated names save a few characters but cost readability every time someone encounters them.

**Guideline:** Use full, unabbreviated words for enum values (e.g., `AVERAGE` over `AVG`) because clarity in the domain model compounds across every layer of the stack.

---

## 2. Format Output Appropriately for the Operation Type

The review noted that showing decimals on a COUNT aggregation does not make sense. Different aggregation operations have different output semantics: COUNT always produces an integer, while AVG may produce a decimal. Applying a single formatting rule to all operations creates confusing output.

**Guideline:** Tailor number formatting to the aggregation operation -- use integer formatting for COUNT and decimal formatting for AVG.

---

## 3. Add Comments Explaining Non-Obvious Filtering Logic

A reviewer asked "Why are we filtering away events?" about a filter step in the aggregation calculation. The code was filtering visit entries to only include those relevant to the aggregation's source visit, but this was not obvious from the code alone. The ensuing thread required several back-and-forth messages to clarify the intent, revealing that the filter implicitly relied on a current domain constraint (variables cannot be shared across episodes).

**Guideline:** When filtering a collection based on domain-specific constraints, add a brief comment explaining the "why" -- especially if the constraint may change in the future.

---

## 4. Validate Circular References When Allowing Self-Referential Types

The aggregation form element allows AGGREGATION elements as potential sources, which could create circular dependencies (an aggregation referencing itself, directly or indirectly). Without validation, this can cause infinite loops or incorrect calculations at runtime.

**Guideline:** When a new entity type can reference entities of its own type, add validation at creation time to prevent circular or self-referencing chains.

---

## 5. Avoid Unnecessary beforeAll in Tests With No Shared State

A reviewer flagged a `beforeAll` block in the integration test as potentially unnecessary. Using `beforeAll` implies shared setup across multiple `it` blocks, but if each test can stand alone, the setup adds coupling between tests and makes failures harder to diagnose.

**Guideline:** Only use `beforeAll` when multiple test cases genuinely share expensive setup; otherwise, set up state directly in each test or use `beforeEach`.

---

## 6. Keep Package Lock Changes Scoped to Intentional Dependency Changes

The PR contained numerous `package-lock.json` modifications (peer dependency flag changes) unrelated to the feature. This happens when developers run `npm install` with a different npm version or configuration, causing lock file drift that obscures the actual dependency changes in review.

**Guideline:** Ensure the team uses the same npm version (e.g., via `.nvmrc` and `engines` in `package.json`), and revert unrelated lock file changes before opening a PR.

---

## 7. Clarify Unsupported Feature Paths When Accepting Them in the Schema

The use case explicitly allows `CALCULATION` as a valid source type for aggregations, but the calculation logic only reads persisted answers -- meaning aggregations over calculated fields silently return null/zero. Accepting a source type in the schema without implementing its computation path creates a confusing silent failure.

**Guideline:** If you accept a value in the schema but do not yet support it at runtime, either block it at creation time with a clear error or document it as a known limitation with a TODO and tracking issue.
