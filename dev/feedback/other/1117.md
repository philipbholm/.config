# Learnings from PR #1117

Refactored the Dialog component from HeadlessUI to Radix UI, fixed dialog/popover layering issues by wrapping popover content in a Portal, and migrated Select and MultiSelect away from HeadlessUI.

---

## 1. Use Static Tailwind Class Names Instead of Dynamic String Interpolation

Tailwind CSS scans source files statically to detect which classes to include in the build. When class names are constructed dynamically via string interpolation (e.g., passing a `color` variable into a template literal), Tailwind cannot detect them at build time, which means the styles may silently break in production even if they appear to work in development.

```tsx
// Bad: Tailwind can't detect the class at build time
className={`border-2 ${color} ${isSelected ? "ring-2" : ""}`}

// Good: Use clsx/cn with static, complete class names
className={clsx(
  "border-2",
  {
    "border-success-400 bg-success-50": isCompleted,
    "border-warning-400 bg-warning-50": isStarted,
    "border-baseline-300 bg-baseline-50": isDefault,
    "ring-2 ring-offset-2": isSelected,
  },
)}
```

**Guideline:** Always write complete, static Tailwind class names in source code so the compiler can detect them; use `clsx`/`cn` with object syntax for conditional classes instead of string interpolation with variables.

---

## 2. Wrap Popover and Overlay Content in a Portal for Correct Layering

When a popover or dropdown renders inside a dialog, it inherits the dialog's stacking context. Without a Portal, the popover content can appear behind the dialog overlay or get clipped by `overflow: hidden` containers. Wrapping the content in a Portal (`PopoverPrimitive.Portal`) renders it at the document root, ensuring it always floats above other layers.

**Guideline:** Always wrap popover, dropdown, and tooltip content in a Portal component when the content may appear inside dialogs or other stacking contexts.

---

## 3. Keep Styling in UI Primitive Components, Not in Consuming Components

When building a component library with UI primitives (e.g., a `Select` or `Dialog` component), the base styling should live in the UI primitive itself rather than being scattered across consuming components. This keeps the design system consistent and avoids duplication. When a quick fix requires adding styles at the consumption site (e.g., because two UI libraries don't play well together), flag it as technical debt.

**Guideline:** Place default styling in the shared UI component; only override or extend at the consumption site when there is a specific, documented reason.

---

## 4. Remove Unnecessary z-index Values to Avoid Stacking Context Bugs

Sprinkling `z-1`, `z-0`, and other z-index utilities across elements creates implicit stacking contexts that interact unpredictably with portals, dialogs, and sticky positioning. When refactoring layering (e.g., moving to Radix Portals), audit and remove z-index values that are no longer needed. Fewer stacking contexts means fewer surprises.

**Guideline:** Only add z-index utilities when there is a specific layering problem to solve; remove them when the underlying layering mechanism changes (e.g., switching to portals).

---

## 5. Prefer Radix UI Primitives Over HeadlessUI for Dialog and Overlay Patterns

When the codebase already uses Radix UI for most primitives, mixing in HeadlessUI for dialogs creates inconsistency and interoperability problems (e.g., focus trapping conflicts, layering issues between HeadlessUI popovers and Radix dialogs). Standardizing on one primitive library reduces the surface area for bugs.

**Guideline:** Standardize on a single headless UI primitive library across the project; avoid mixing HeadlessUI and Radix UI in the same component tree.
