# Learnings from PR #975

Added collaborator invite and accept/decline flow for registries, including UI layout changes and scoping getRegistry to return only registries for the logged-in user.

---

## 1. Avoid Inline Styles When a Utility-Class Framework Is Available

Inline styles (e.g., `style={{ width: '2px', backgroundColor: '#f59e0b' }}`) increase bundle size and make it impossible to maintain a consistent theme. When Tailwind or a similar utility-class framework is available, every inline style that maps to a utility class should be replaced with it (e.g., `w-0.5`, `bg-warning-500`). Copying code from another component does not excuse carrying over bad patterns.

**Guideline:** Never inline CSS properties that have a direct Tailwind equivalent; use the utility class instead to keep theming consistent and bundles small.

---

## 2. Remove Unused Code Instead of Leaving It for Later

A `ref` prop was passed through to a component but never used. The reviewer asked "why" and the author confirmed it was dead code copied from another component. Dead code obscures intent, confuses future readers, and inflates the surface area that needs to be reviewed.

**Guideline:** Remove unused props, imports, and variables immediately rather than leaving them with a mental note to clean up later.

---

## 3. Prefer `toEqual` Over Individual Field Assertions in Tests

Asserting individual fields on an object (e.g., `expect(obj.name).toBe(...)`) makes it easy to miss unexpected fields or regressions. Using `toEqual` on the full object ensures the entire shape is verified and makes test failures more informative.

**Guideline:** Use `toEqual` with the full expected object shape instead of asserting fields one at a time, so regressions in any field surface immediately.

---

## 4. Test With Multiple Entities and Cross-Entity Isolation

When testing a query that retrieves items scoped to an entity (e.g., collaborators for a registry), create at least two parent entities with their own children. This catches accidental cross-contamination where a query returns records from the wrong parent due to a missing or incorrect filter.

**Guideline:** Always seed at least two parent entities in integration tests to verify that queries correctly scope results and do not leak data across boundaries.

---

## 5. Do Not Skip Tests for New Use Cases

A new use case (`getCollaboratorInvitations`) was added without any corresponding integration test. The reviewer flagged this as a missing test. Every public use case that performs I/O should have at least one happy-path integration test to catch regressions early.

**Guideline:** Every new use case or endpoint must ship with at least one integration test covering the main success scenario.

---

## 6. Keep Function Names Accurate After Refactoring

A test helper function name no longer matched its behavior after the code evolved (it referenced a single registry/site but was used in a broader context). Misleading function names erode trust in the test suite and slow down debugging.

**Guideline:** When changing what a helper function does, rename it to match its current behavior so readers can trust the name without reading the implementation.

---

## 7. Pass All Relevant IDs to Authorization Checks

When checking permissions (e.g., `hasPermission`), pass all available contextual identifiers (like `siteId`) even if the current implementation does not strictly require them. This prevents subtle authorization bugs where a valid permission for one entity is mistakenly accepted for another.

**Guideline:** Always supply every available scope identifier (site, registry, org) to authorization checks, even if the check currently works without them, to prevent cross-entity privilege escalation.
