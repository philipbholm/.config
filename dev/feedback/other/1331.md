# Learnings from PR #1331

Added runtime guards to prevent `NonValidatingAuthenticationProvider` from being instantiated outside development environments, after discovering that `z.coerce.boolean()` silently coerces `"false"` to `true`.

---

## 1. Prefer Whitelisting Over Blocklisting for Security-Sensitive Configuration

When deciding which environments are allowed to use a dangerous feature (like a non-validating auth provider), whitelist the allowed environments rather than blocklisting known production environments. Blocklisting is fragile because new environment names (e.g., `"prod"` vs `"production"`) can slip through. Whitelisting is closed by default -- anything not explicitly listed is denied.

**Guideline:** For security-sensitive runtime guards, explicitly enumerate the allowed values rather than trying to enumerate and block all disallowed ones.

---

## 2. Treat Undefined Environment Variables as Unsafe by Default

If a required environment variable like `NODE_ENV` is not set, the guard should fail closed (throw an error), not silently allow the operation. In cloud environments, `NODE_ENV` may be undefined if not explicitly configured, and failing open could accidentally enable dev-only code paths in production.

**Guideline:** When a missing environment variable could lead to a security-relevant code path, treat `undefined` as disallowed and throw immediately.

---

## 3. Be Wary of Zod's `z.coerce.boolean()` for String Environment Variables

The Zod schema method `z.coerce.boolean()` coerces any truthy string to `true`, meaning the string `"false"` becomes `true`. This caused the `USE_STUBBED_AUTH_PROVIDER` flag to enable the stubbed provider even when explicitly set to `false`. Use a custom string-to-boolean parser (e.g., `z.stringbool()`) for environment variable booleans instead.

```typescript
// Bad: z.coerce.boolean() -- "false" becomes true
USE_STUBBED_AUTH_PROVIDER: z.coerce.boolean().default(false),

// Good: custom parser that correctly handles "false"
USE_STUBBED_AUTH_PROVIDER: z.stringbool().default(false),
```

**Guideline:** Never use `z.coerce.boolean()` for environment variables; use a string-to-boolean parser that correctly interprets `"false"`, `"0"`, etc.

---

## 4. Add Constructor-Time Validation for Dangerous Implementations

Rather than relying on the caller to check environment conditions before instantiating a dangerous class, put the validation inside the constructor itself. This way, even if a caller forgets the check, the guard still fires.

```typescript
export class NonValidatingEndUserAuthenticationProvider {
    constructor() {
        assertNonProductionEnvironment(); // Fail immediately if misused
    }
}
```

**Guideline:** Place environment/safety assertions in constructors of dangerous implementations so they are impossible to bypass at the call site.

---

## 5. Set `NODE_ENV` Explicitly in All Docker Compose Services

When guard logic depends on `NODE_ENV`, every service in `docker-compose.yml` must have it explicitly set. If even one service is missing it, the guard will throw in local development. The PR had to add `NODE_ENV=development` to multiple compose service definitions that previously omitted it.

**Guideline:** When adding runtime guards that check `NODE_ENV`, audit all `docker-compose.yml` service definitions and ensure the variable is explicitly set.

---

## 6. Complement Compile-Time Configuration Checks with Runtime Shutdown Visibility

A code-level guard against misconfiguration is valuable but not sufficient on its own. Consider also building observability and alerting to detect and shut down services that are running with insecure configurations in non-local environments. Defense in depth means a single guard is never the only safeguard.

**Guideline:** Treat runtime environment guards as one layer of defense, and plan for monitoring or alerting to catch misconfigurations that slip through.
