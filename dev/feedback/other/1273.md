# Learnings from PR #1273

This PR replaced a Playwright E2E test for user state changes (disable/enable) with a unit/integration test using Testing Library, and addressed unhandled rejection issues from Apollo GraphQL errors in the test environment.

---

## 1. Use MSW Instead of Mocking the Apollo Client Directly

A reviewer explicitly stated that the Apollo client should not be mocked out manually. The project has an established pattern using MSW (Mock Service Worker) for GraphQL mocking in integration tests (e.g., `patient-form-data-entry-container.integration.test.tsx`). Mocking the client directly couples tests to implementation details of the data layer, making tests brittle when the client configuration changes. MSW intercepts at the network level, which is closer to how the real app behaves.

**Guideline:** Use MSW to mock GraphQL requests in integration tests rather than creating custom Apollo client mocks, because MSW keeps tests decoupled from client implementation and closer to production behavior.

---

## 2. Do Not Globally Suppress Errors in Test Setup

The PR added a `process.on('unhandledRejection')` handler to silently ignore Apollo errors. A reviewer pointed out this pattern is dangerous because it can mask real test failures -- if a GraphQL call fails unexpectedly, the test would still pass. The accepted compromise was to scope the error suppression narrowly within the specific test that needs it, not in the global test setup.

**Guideline:** Never add global error suppression (`process.on('unhandledRejection')`) in test setup files, because it hides real failures across the entire test suite; instead, handle expected errors locally within individual tests.

---

## 3. Prefer ErrorBoundary Components Over Process-Level Error Swallowing

A reviewer suggested wrapping the component under test in a React `ErrorBoundary` to handle expected errors cleanly. This keeps error handling within React's component model rather than reaching for Node.js process-level workarounds. While it did not fully resolve the issue in this case, the principle is sound: test error handling should mirror production error handling patterns.

**Guideline:** When testing components that are expected to encounter errors, wrap them in an `ErrorBoundary` in the test rather than suppressing errors at the process level, because it mirrors how the production app handles those errors.

---

## 4. Avoid Mocking Unless Absolutely Necessary

A reviewer directly challenged the need for mocking: "I dont think we should mock stuff unless we absolutely need to." Over-mocking makes tests pass in artificial conditions that do not reflect real usage, giving false confidence. Every mock is a divergence from reality, so each one should be justified.

**Guideline:** Default to not mocking; only introduce mocks when there is a concrete reason (external service, non-deterministic behavior, prohibitive cost), and document why the mock exists.

---

## 5. Ensure Error Scenario Tests Verify the Error Is Displayed

A review comment noted that the "retry after error" test did not verify the error message was actually displayed to the user before the retry. Testing the happy path of error recovery without first asserting the error state was rendered means you cannot be sure the error handling UI actually works.

**Guideline:** In error-recovery test scenarios, always assert that the error state is visible to the user before testing the recovery action, because skipping that assertion leaves the error UI untested.

---

## 6. Keep Test Dependencies in devDependencies Only

The `@testing-library/jest-dom` package appeared in both `dependencies` and `devDependencies`. Testing libraries should never be in production dependencies because they inflate the production bundle and can introduce unnecessary security surface area.

**Guideline:** Always place testing libraries (`@testing-library/*`, `vitest`, `jest`, `msw`) in `devDependencies` only, and verify they do not leak into `dependencies` during package.json modifications.
