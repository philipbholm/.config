# Learnings from PR #1355

Adds patient-reported outcome measures (PROMs) to the registry system, including a public-facing form for unauthenticated users to submit PROM responses.

---

## 1. Place Unauthenticated Routes Inside the Standard Route Structure

A PROM response page was placed outside the normal route hierarchy to allow unauthenticated access. The reviewer noted that the standard router structure already supports unauthenticated routes and that special-casing the route placement creates an exception that future developers must know about. The counter-argument was that the Apollo provider errors when no access token exists, but the reviewer clarified that a missing token (as opposed to an invalid one) should not cause failures.

**Guideline:** Use the existing route structure's unauthenticated route support rather than placing routes outside the hierarchy; if the auth provider throws on missing tokens, fix the provider rather than working around it.
