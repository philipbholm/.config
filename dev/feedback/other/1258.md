# Learnings from PR #1258

This PR added the ability to reuse forms across multiple visits/treatments, including a confirmation dialog when editing a shared form.

---

## 1. Avoid Triggering Side Effects via useEffect When a Direct Handler Suffices

A reviewer flagged a `useEffect` that called `onConfirm` when state became valid. The author agreed it was unnecessary. Using `useEffect` to trigger callbacks in response to derived state creates unintuitive control flow -- other developers cannot easily trace when or why the callback fires. If the action should happen on a user interaction (e.g., button press), call it directly from the event handler.

**Guideline:** Call callbacks directly from event handlers rather than routing them through `useEffect` on derived state, because indirect triggers obscure causality and make the component harder to reason about.

---

## 2. Keep Comments in Tests When the Setup Is Non-Trivial, But Prefer Page Objects

A reviewer asked whether comments in test files should be reduced. The author argued they were warranted because the test steps were not trivially understandable, but acknowledged that page-object abstractions would be the better long-term solution. The tension here: comments can rot, but removing them from complex test orchestration without replacing them with better structure leaves the code opaque.

**Guideline:** When test setup is complex enough to need explanatory comments, treat that as a signal to extract a page object or test helper -- but do not prematurely abstract until the patterns stabilize.

---

## 3. Clean Up Dead Code -- Even Emoji-Only Review Comments Signal It

A reviewer left a cleanup emoji on an unused GraphQL file (reorderFormsInParentEvent.graphql). Leftover files from refactoring silently increase the codebase surface area and confuse future developers about what is still in use.

**Guideline:** When moving or refactoring functionality, explicitly search for and remove orphaned files (GraphQL operations, old components, unused imports) in the same PR.

---

## 4. Use Confirmation Dialogs with Name-Matching for Destructive Shared-Resource Operations

The PR introduced a pattern where deleting a shared form requires the user to type the form name. A reviewer asked why the name-match check existed. This is a standard safety pattern for destructive operations on shared resources -- it forces the user to pause and confirm intent, reducing accidental data loss. The pattern is especially important when a single action can cascade to multiple contexts.

**Guideline:** For destructive operations that affect multiple entities (shared forms, cascading deletes), require explicit user confirmation by typing the resource name, because generic "Are you sure?" dialogs are too easily dismissed.
