# Learnings from PR #1085

This PR added support for additional form element types (categories, separators, static text) in the registry service and frontend, including category structures and new GraphQL types.

---

## 1. Remove Redundant Wrapper Elements in React Components

A `StaticText` component wrapped its content in a `<div>` that served no purpose beyond what the parent already provides. Extra DOM nodes add noise, hurt accessibility tree clarity, and make styling harder to reason about.

**Guideline:** Do not wrap component output in an extra DOM element unless it provides specific styling, layout, or semantic purpose that the parent cannot provide.

---

## 2. Avoid Manually Re-Declaring Types That Should Be Inferred From the Domain

A form element type was manually declared as a union (`"TEXT" | "CATEGORY" | ...`) instead of being inferred from the event payload or schema. Manual type declarations drift from the source of truth and create maintenance burden when new types are added.

**Guideline:** Derive types from the domain model or schema rather than redeclaring them inline; this ensures type changes propagate automatically.

---

## 3. Extract Complex Parsing Logic Out of Use Case `run` Functions

Inline parsing code (mapping form elements to variable IDs, category structure IDs, etc.) inside the `run` function makes it hard to follow the high-level business flow. Extract these transformations into named helper functions so the `run` function reads like a summary of what happens.

```typescript
// Instead of inline:
const variableIds = formElements.map(el => {
    if ("variableId" in el) return el.variableId;
    return undefined;
}).filter((id): id is string => id !== undefined);

// Extract to:
const variableIds = extractVariableIds(formElements);
```

**Guideline:** Keep use case `run` functions high-level by extracting data transformation and parsing into named helper functions.

---

## 4. Prefer `findFirstOrThrow` Over Manual Null Checks

When querying for a record that must exist, using `findFirst` followed by a manual null check and error throw is unnecessarily verbose. Prisma provides `findFirstOrThrow` which does the same thing with less code and a clearer intent.

**Guideline:** Use Prisma's `findFirstOrThrow` (or `findUniqueOrThrow`) when the record is expected to exist, rather than manually checking and throwing.

---

## 5. Structure the Domain to Avoid Runtime Type Switches

A switch statement was used to determine the form element type when mapping database rows to domain objects. If the schema included a `latestVersion` relation or a discriminator field directly on the element, the code could avoid the switch entirely and let the type system handle it.

**Guideline:** When you find yourself writing a switch/case on a type discriminator during data mapping, consider restructuring the domain model (e.g., adding a direct relation or field) so the mapping is type-safe by construction.

---

## 6. Design Category Values for Future Export and Multi-Format Needs

A `RegistryCategory` with a single `value` field was flagged as too simple for real-world use. Registries often need multiple representations: one for display (e.g., "Kvinne"), one for imported data (e.g., "K"), one for export to other systems (e.g., "1"), and possibly translations.

**Guideline:** When designing data models for categorical values, plan for multiple representations (display, import, export, translation) from the start rather than retrofitting them later.

---

## 7. Remove Unexplained Schema Fields and Dead Code

A field `referenceKey` in the Prisma schema was questioned and turned out to be a copy-paste artifact with no purpose. Unexplained fields in the schema confuse other developers and may cause incorrect assumptions about the data model.

**Guideline:** Never leave unexplained or vestigial fields in database schemas; if you cannot explain a field's purpose, remove it.

---

## 8. Extract Nested Test Assertions Into Variables

Deeply nested `expect.objectContaining` assertions inside test expectations make failure messages hard to read. When a nested field fails, the diff output is nearly impossible to parse. Extract the expected nested objects into named variables and use `toEqual` for clearer failures.

```typescript
// Instead of:
expect(result).toMatchObject({
    categoryStructure: expect.objectContaining({
        categories: expect.arrayContaining([...])
    })
});

// Prefer:
const expectedCategories = [...];
expect(result.categoryStructure.categories).toEqual(expectedCategories);
```

**Guideline:** Extract deeply nested expected objects in tests into named variables and use `toEqual` for precise, readable failure messages.

---

## 9. Use Past-Tense Event Names Consistently But Name Handlers by Action

Event types use past tense (`CATEGORY_STRUCTURE_CREATED`) because they represent facts that have occurred. Handler method names should match this convention. A reviewer suggested `createCategoryStructure` (imperative) but the author clarified `handleCategoryStructureCreated` matches the event name. Both conventions have merit -- choose one team-wide and be consistent.

**Guideline:** Establish a team convention for event handler naming (either `handle{EventName}` or imperative `{action}`) and apply it consistently across all projections.
