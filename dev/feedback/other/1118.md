# Learnings from PR #1118

Added codelist (personal data) support to registries, including a new OAuth client for service-to-service authentication, AWS Secrets Manager integration, and frontend forms for managing codelist records.

---

## 1. Avoid Leaking Domain Concepts Across Service Boundaries in the GraphQL Schema

When integrating a new domain (e.g., codelist) into an existing service (e.g., registries), consider whether the new domain's types should be exposed directly in the parent service's GraphQL schema or wrapped inside existing domain concepts (e.g., "patient"). Leaking a separate domain directly into another service's schema creates tight coupling and makes future refactoring harder. Prefer wrapping the new data under the existing aggregate that consumers already interact with.

**Guideline:** When exposing data from an external domain in your GraphQL schema, wrap it under the existing aggregate root (e.g., Patient) rather than exposing the foreign domain's types directly.

---

## 2. Infer Types from Schemas Rather Than Defining Them Manually

When there is already a schema (Zod, GraphQL, database) that describes the shape of the data, derive TypeScript types from it rather than manually duplicating the type definition. Manual duplication drifts out of sync and doubles the maintenance burden.

**Guideline:** Use schema inference (e.g., `z.infer<typeof schema>`) to derive types from a single source of truth instead of maintaining parallel type definitions.

---

## 3. Write Unit Tests for Pure Formatting and Utility Functions

Formatting functions (date formatters, string transformers, value displayers) are pure functions with well-defined inputs and outputs. They are cheap to test and fragile to leave untested because they affect user-visible output. Skipping tests for them makes it easy for edge cases (null values, locale differences, unexpected formats) to slip through.

**Guideline:** Always add unit tests for pure formatting and utility functions, especially those that transform data for display to users.

---

## 4. Use `jest.fn()` with `toHaveBeenCalledWith` for Integration Test Assertions

When testing that your application code calls a third-party adapter or external service with the correct arguments, prefer `jest.fn()` spies with `expect(mock).toHaveBeenCalledWith(expectedArgs)` over manually capturing arguments. This approach is more expressive, produces better error messages on failure, and communicates the test's intent more clearly.

**Guideline:** Use `jest.fn()` with `toHaveBeenCalledWith` to assert on adapter/service call arguments rather than capturing and comparing args manually.

---

## 5. Validate External Configuration with Zod at the Boundary

When reading secrets from AWS Secrets Manager, environment variables, or any external source, validate the shape of the data immediately after retrieval using Zod (or equivalent). Unvalidated external data leads to cryptic runtime errors deep in application code rather than clear validation failures at the boundary.

**Guideline:** Parse and validate all external configuration (secrets, env vars, API responses) with a schema validator at the point of ingestion.

---

## 6. Keep Integration Test Setup Variables Named Simply

When using a shared test setup pattern (e.g., a `beforeAll` block that returns a context object), name the variable something generic like `testSetup` or `ctx`. Naming it after the specific test scenario it was created for (e.g., `createPatientCodelistRecordResult`) becomes confusing when multiple tests assert on the same setup, and the name stops making sense as more assertions are added.

**Guideline:** Name shared test setup variables generically (e.g., `testSetup`, `ctx`) rather than after a specific assertion, and use `beforeAll` when multiple tests share the same setup.

---

## 7. Consider Regulatory Requirements When Designing Access Control Granularity

When dealing with health data, regulatory requirements (e.g., patient journal laws) may mandate that personally identifiable information be restricted to only those who need it. Before simplifying access control by merging permission scopes (e.g., collapsing "codelist access" into "patient access"), verify that the simplified model still satisfies legal requirements for data minimization.

**Guideline:** When designing access control for sensitive data, check applicable regulations before simplifying permission models -- data minimization requirements may mandate finer-grained access than seems technically necessary.

---

## 8. Use Clear, Non-Ambiguous Parameter Names When Wrapping SDK Calls

When wrapping an AWS SDK call (or any external API), ensure your function's parameter names do not conflict with or confuse the SDK's own naming. For example, naming a parameter `secretName` and then passing it as `{ SecretId: secretName }` can mislead readers into thinking the value is a name when the SDK actually expects an ID or ARN.

**Guideline:** Align your wrapper function parameter names with the external SDK's expected field names (or clearly document the mapping) to avoid confusion about what value is actually expected.
