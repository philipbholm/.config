# Learnings from PR #1466

This PR implemented PII masking for patient detail pages with per-field reveal buttons and audit logging, to comply with Norwegian patient medical records regulations.

---

## 1. Do Not Reveal Sensitive Data When the Audit Log Fails

When a reveal action must be logged for compliance, the UI must block the reveal if the logging mutation fails. In this PR, the handler caught the error but still toggled visibility, allowing PII to be seen without any audit trail. For compliance-critical features, the operation (showing data) and the side-effect (logging) must succeed atomically from the user's perspective.

```typescript
// Bad: reveals PII even if logging fails
try {
  await logPiiAccessMutation(...);
} catch (error) {
  console.error("Failed to log PII access:", error);
}
piiVisibility.setVisible(!piiVisibility.isVisible); // always runs

// Good: only reveal if logging succeeds
try {
  await logPiiAccessMutation(...);
  piiVisibility.setVisible(true);
} catch (error) {
  // show error to user, do not reveal
}
```

**Guideline:** When an action requires a successful audit log for compliance, make the user-visible state change conditional on the audit log succeeding.

---

## 2. Move Security-Critical Logging to the Backend

Client-side audit logging is trivially bypassable -- a sophisticated user can modify the JavaScript, disable the mutation, or inspect the network response to see PII without any log entry. If the requirement is "no PII access without logging," the enforcement must happen server-side: the API should not return PII data unless access has been logged, or the query itself should log access before returning the data.

**Guideline:** Never rely on frontend code for security enforcement; implement access logging and gatekeeping on the server where the user cannot bypass it.

---

## 3. Use Correct ARIA Roles for Loading and Error States

Using `role="status"` for a loading indicator is semantically incorrect. The `status` role is for advisory information that changes (like a status bar), not for content that is actively loading. Use `role="progressbar"` or a spinner component for loading states, and keep `role="alert"` for actual error messages.

**Guideline:** Choose ARIA roles based on their semantic meaning (progressbar for loading, alert for errors, status for live-region updates), not just to satisfy accessibility linting.

---

## 4. Add Frontend Tests for Security-Sensitive UI Components

A PII masking component has significant user interaction logic (initial masked state, reveal/hide toggling, mutation triggering, error handling) that should be covered by unit tests. Shipping a security-sensitive feature with only backend integration tests leaves the most user-facing logic untested.

Key test scenarios for a component like `PiiField`:
- Initial state shows masked value
- Click reveals value and triggers API call
- Click again hides value (no API call on hide)
- Error handling when API call fails (value stays masked)
- Accessibility attributes update correctly

**Guideline:** Security-sensitive frontend components (PII masking, auth flows, permission gates) must have unit tests covering their state transitions and error paths.

---

## 5. Prefer `toEqual` Over `toMatchObject` for Database Verification Tests

Using `toMatchObject()` in database verification tests allows extra unexpected fields to pass silently. When verifying that a record was stored correctly, use `toEqual()` (with `expect.any()` for dynamic fields) to assert the complete shape, so new columns or unexpected data don't slip through unnoticed.

```typescript
// More strict - catches unexpected fields
expect(logs[0]).toEqual({
  id: expect.any(String),
  userId: userId,
  patientId: patient.id,
  registryId: registry.id,
  fieldName: fieldName,
  action: "reveal",
  createdAt: expect.anything(),
});
```

**Guideline:** Use `toEqual` with `expect.any()` matchers for database verification tests to assert the full record shape.

---

## 6. Follow Codebase Naming Conventions in Test Files

Test describe blocks should follow the codebase convention (PascalCase for the use case name, e.g., `"LogPiiAccess"`) rather than lowercase (`"log pii access"`). Consistent naming makes it easier to search for tests and correlate them with the implementation.

**Guideline:** Match the test describe block naming convention used in the rest of the codebase.
