# Learnings from PR #1428

Fixing CSV import to support semicolon delimiters, remove BOM markers, and detect duplicate patient IDs during import.

---

## 1. Do Not Both Log and Throw the Same Error

When a function logs an error and then throws it, the global error handler will log it again, producing duplicate entries in monitoring (e.g., Datadog). This makes incident triage harder because the same error appears twice with different stack contexts, creating confusion about whether it happened once or twice.

**Guideline:** Either log the error and handle it locally, or throw it for a higher-level handler to log -- never both.

---

## 2. Validate the Entire Input Before Committing Partial Results

When processing data in batches (like importing a large CSV), validation that is scoped to a single batch will miss violations that span batches. For example, a duplicate check within one batch of 1000 rows will not catch a duplicate that appears in row 500 and row 1500. If earlier batches are already committed when a later batch fails, the user is left with a partial import and a confusing error.

**Guideline:** For operations that must be all-or-nothing, either pre-scan the entire input for constraint violations before writing, or wrap the full operation in a transaction that rolls back on any failure.

---

## 3. Plan for Auto-Detection Fallback in User-Facing Parsing

When auto-detecting file format properties (like CSV delimiters), there will be edge cases where detection fails (e.g., single-column files). Providing a sensible default is not enough -- the user should be able to override the detected value. This prevents silent data corruption where a wrong delimiter causes all columns to merge into one.

**Guideline:** When auto-detecting a format property, always allow the user to manually override the detected value, and do not hard-code a default that could silently produce wrong results.

---

## 4. Add Unit Tests for Extracted Helper Functions

When moving logic into shared helper files (like `import/helpers.ts`), add unit tests for those helpers even if the calling use cases have integration tests. Helpers are likely to be moved, reused, or modified independently, and having their own test suite makes refactoring safe.

**Guideline:** Any utility or helper function extracted into its own module should have dedicated unit tests, independent of the integration tests that exercise the calling code.

---

## 5. Create Reusable Test Data Builders for Structured Formats

When multiple tests need to construct CSV strings (or similar structured data), consider creating a shared builder or storing fixture files in a `test-data` folder. Inline string construction of CSV data in tests is fragile, hard to read, and leads to inconsistent formatting across test files.

**Guideline:** For structured test inputs like CSV or JSON, use either a shared builder utility or fixture files that can be imported by multiple test suites.
