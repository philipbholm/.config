# Learnings from PR #1628

Stopped logging expected client errors (`NotFoundError`, `NotAuthorizedError`, `AlreadyExistsError`) to Sentry/Datadog in the codelist gRPC service to reduce monitoring noise.

---

## 1. Be Explicit When Suppressing Error Logging by Category

When filtering which errors get sent to error monitoring, using a broad type guard like `isApplicationError()` is convenient but risks silently suppressing errors you do want to see if new application error types are added later. A reviewer raised this concern: "if we add a new application error, that we wish to log, it doesn't get skipped accidentally." The counter-argument was that all application errors represent expected flows and should never be logged. The key insight is that this decision should be made deliberately -- either explicitly enumerate which errors to skip (allowlist approach, safer) or document the convention that all `ApplicationError` subclasses are expected (convention approach, simpler).

**Guideline:** When suppressing error reporting for a category of errors, document the convention clearly so future error types are handled intentionally rather than silently.

---

## 2. Distinguish Client Errors from Server Errors at the Monitoring Layer

The change correctly identifies that 4xx-equivalent errors (not found, not authorized, already exists) are client mistakes, not server bugs, and should not trigger alerts. However, the implementation only covers `ApplicationError` subclasses -- other client-side failures like `ZodError` (invalid arguments) or missing auth tokens would still be reported to Sentry. For a comprehensive solution, consider filtering based on the final gRPC status code (only capture `INTERNAL` status errors) rather than checking error type, which would cover all current and future client error paths.

**Guideline:** When deciding what to report to error monitoring, filter by the resulting status code category (5xx vs 4xx equivalent) rather than by individual error class, to catch all client error paths uniformly.

---

## 3. Keep Error Monitoring Signal-to-Noise Ratio High

Logging expected client errors to Sentry creates alert fatigue and buries real server issues. The 4xx errors in this service (codelist not found, unauthorized access) are routine and carry no actionable information for the backend team. Removing them from error capture makes the monitoring dashboard useful again -- every alert represents a genuine problem that needs investigation.

**Guideline:** Only send unexpected server errors (5xx equivalent) to error monitoring services; expected client errors should be handled in logs or metrics, not alerts.
