# Learnings from PR #1309

Added a template-based registry creation flow with UI for selecting templates, and moved a domain-specific seed definition to the backend.

---

## 1. Avoid Premature Abstractions for Single Use Cases

When implementing the template creation logic, the code was hard-coded for a single known template (myelomatose registry). A reviewer asked if it was specific to that one registry, and the response was to intentionally avoid hasty abstractions for a single use case. The plan was to generalize later when more templates existed, or possibly keep separate implementations per template.

**Guideline:** When there is only one concrete use case, write concrete code for it rather than building an abstraction -- generalize once you have two or three real cases to inform the design.

---

## 2. Plan for Extensibility Without Over-Engineering

A reviewer noted that the template system would need to support multiple episodes later. The PR author acknowledged this but chose to merge the working implementation and extend it when additional templates arrive. This is a deliberate tradeoff: ship the working feature now, document the known limitation, and refactor when the requirement materializes.

**Guideline:** When you know a feature will need to generalize, note it in the review discussion or code comments, but do not block shipping on speculative abstractions.

---

## 3. Extract Shared Dictionaries When Splitting Pages

When a single page was split into multiple pages (choose creation method, create from scratch, create from template, select template), the localization dictionary was extracted into a shared `dictionary.ts` file. This avoids duplicating translation strings across related pages.

**Guideline:** When refactoring a single page into multiple related pages, extract shared localization/dictionary objects into a common file to keep translations in sync.
