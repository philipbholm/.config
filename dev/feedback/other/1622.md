# Learnings from PR #1622

Fixed two import bugs: patient imports with no data columns failing, and numeric values with comma decimal separators (e.g., "1,5") not displaying because the system expected period separators.

---

## 1. Prefer Zod for Validation Over Manual Parsing Logic

When writing validation or parsing logic (e.g., checking if a string is a valid number format, normalizing input), prefer using Zod schemas rather than hand-rolled conditionals. Zod provides a declarative, composable approach that is easier to read, test, and extend. Manual validation functions tend to accumulate edge cases and become brittle over time.

**Guideline:** Use Zod schemas for data validation and transformation; reserve manual parsing only for cases where Zod cannot express the constraint.

---

## 2. Normalize Data at the Backend, Not Just the Frontend

The PR initially only handled comma-to-period normalization in the frontend `NumberFormInputElement`. A reviewer's question ("should we have test cases for the parsing as well?") revealed that the backend import path for patients was missing the same normalization. Data normalization must happen at every ingestion point -- frontend form submission, CSV import, API endpoints -- because each is an independent path into the system. If normalization only exists in one layer, data entering through other paths will be inconsistent.

**Guideline:** Apply data normalization at the backend/service layer so it is enforced regardless of how data enters the system (UI, import, API).

---

## 3. Normalize Only Type-Appropriate Fields

The decimal normalization correctly checks the element type before applying: only `NUMBER` fields get comma-to-period replacement, while `TEXT` fields like "Hello, world" are left untouched. This is important because a naive global comma replacement would corrupt text data. The `normalizeAnswerValue` helper takes the column data type as a parameter to make this distinction explicit.

```typescript
export function normalizeAnswerValue(
    value: string,
    columnName: string,
    columnDataTypes: Record<string, VariableDataType>,
): string {
    return columnDataTypes[columnName] === "NUMBER"
        ? normalizeDecimalSeparator(value)
        : value;
}
```

**Guideline:** When normalizing input data, always scope the transformation to the specific data types that need it, and pass type information explicitly to the normalization function.

---

## 4. Handle Incomplete Transient Input Gracefully on Blur

When building numeric input components that allow intermediate states (like typing "-" before the digits), the blur handler must clean up these incomplete tokens rather than persisting them. The PR handles this by resetting to the original value when the normalized result is empty (meaning the user typed something like "-", ".", or "-."). This prevents invalid non-numeric strings from being saved.

**Guideline:** On input blur, validate and normalize the final value; if it is incomplete or invalid, revert to the previous valid value rather than persisting garbage.

---

## 5. Guard Against Empty Column Sets in Import Logic

The PR fixed a crash when importing patients with only a patient ID column and no data columns. The `loadFormElementsMap` function threw an error when it found zero form elements, but this is a valid scenario when the CSV has only identifier columns. Adding an `expectColumns` parameter to conditionally skip this check prevents the false failure.

**Guideline:** When building import/batch operations, handle the degenerate case of zero data columns gracefully, as identifier-only imports are a valid use case.
