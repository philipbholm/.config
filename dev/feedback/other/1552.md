# Learnings from PR #1552

This PR added codelist as an optional feature for registries, with enable/disable functionality in both the creation flow and settings page.

---

## 1. Default new features to off (opt-in, not opt-out)

When adding a new feature toggle like `codelistEnabled`, defaulting it to `true` risks exposing incomplete or sensitive functionality to all users before it is ready. A reviewer flagged that `codelistEnabled` should default to `false` so people do not use it by accident, especially since this feature deals with personally identifiable information (PII).

**Guideline:** Always default new feature flags to `false` so features are opt-in, particularly when they involve sensitive data or are not yet fully validated.

---

## 2. Prefer safe defaults for PII-related settings

The create-registry form initially had PII storage enabled by default. Review feedback emphasized that storing directly identifiable personal information should require an explicit user action to enable, not be pre-checked. This is both a UX safety concern and a compliance concern.

**Guideline:** Any setting that enables collection or storage of personally identifiable information must require explicit opt-in rather than opt-out.

---

## 3. Replace TODO comments with tracked issues

A reviewer pointed to a TODO comment in the codebase and asked to create a GitHub issue and reference it instead. Inline TODOs are easy to forget and invisible in project management tools. Linking to a tracked issue ensures the work is visible, prioritized, and not lost.

**Guideline:** When leaving a TODO in code, always create a corresponding GitHub issue and reference its number in the comment so the work is tracked.

---

## 4. Flatten nested conditionals for readability

A use case function with nested if-else branches was flagged as hard to reason about. The reviewer recommended restructuring with early returns or guard clauses to reduce nesting depth. Deeply nested conditionals increase cognitive load and make it harder to verify correctness.

**Guideline:** Use early returns and guard clauses to keep conditional logic flat; avoid nesting if-else blocks more than one level deep.

---

## 5. Evaluate whether AI-generated tests add real value

The PR included multiple unit tests that were mostly AI-generated. A reviewer questioned whether all of them were necessary given that similar behavior was already covered by end-to-end tests. Redundant tests increase maintenance burden without improving confidence. The author agreed and considered removing them.

**Guideline:** Before committing AI-generated tests, review each one for actual value -- remove tests that duplicate existing coverage or test trivial behavior that e2e tests already verify.

---

## 6. Keep AI-generated code comments under control

A reviewer noticed verbose auto-generated comments in test files and suggested updating the project's CLAUDE.md to limit such comments. AI assistants can produce noisy commentary that clutters the codebase if not configured with constraints.

**Guideline:** Configure AI coding assistants (via CLAUDE.md or similar) to minimize unnecessary comments, and review AI-generated output for noise before committing.

---

## 7. Verify GQL resolvers return all required fields after refactoring

When a resolver was changed to return a mapped object (`registryToGqlRegistry`), a field (`patientDataFormId`) was silently dropped because the mapper did not include it. Consumers relying on that field would receive `null`, breaking downstream behavior. This was caught by automated review and then fixed.

**Guideline:** After refactoring a GraphQL resolver to use a mapper function, verify that all fields previously returned are still present in the new mapping, or explicitly handle their removal in the schema.
