# Learnings from PR #1290

Fixed environment variable parsing after upgrading to Zod 4, where `z.coerce.boolean()` changed behavior (now treats any non-empty string as truthy) and `z.coerce.number()` changed to accept `unknown` input.

---

## 1. Understand Coercion Semantics When Upgrading Validation Libraries

Zod 4 changed `z.coerce.boolean()` so that any present, non-empty string (including `"false"`) coerces to `true`, because the underlying JavaScript `Boolean("false")` is truthy. This silently broke environment variable parsing where `ENV_VAR=false` should mean `false`. The fix was to use `z.stringbool()`, which explicitly maps string values `"true"`/`"false"` to their boolean equivalents.

```typescript
// Broken in Zod 4: "false" coerces to true
POSTGRES_IAM_AUTH: z.coerce.boolean().default(false),

// Fixed: explicitly parses "true"/"false" strings
POSTGRES_IAM_AUTH: z.stringbool().optional().default(false),
```

**Guideline:** When upgrading schema validation libraries, audit every coercion call against the new version's semantics -- especially boolean coercion from string environment variables.

---

## 2. Treat Environment Variables as Strings at the Schema Boundary

All environment variables arrive as strings from `process.env`. Rather than relying on generic coercion (which may behave differently across library versions), explicitly parse from string using `.transform()` or dedicated string-typed validators. This makes the parsing intent clear and version-resilient.

```typescript
// Explicit: clearly transforms a string to a number
GRAPHQL_PORT: z.string().transform((val) => Number.parseInt(val)),

// Implicit: coercion behavior may change across versions
GRAPHQL_PORT: z.coerce.number(),
```

**Guideline:** For env var schemas, prefer explicit string-to-type transforms over generic coercion to avoid subtle breakage across library upgrades.

---

## 3. Add Tests for Environment Schema Parsing

The PR added a comprehensive test file (`env.test.ts`) that validates the behavior of the schema transforms -- including edge cases like empty strings, missing values, and default behavior. This caught issues that would otherwise only surface at runtime in deployed environments.

**Guideline:** Write unit tests for environment variable schemas, especially for boolean defaults and number parsing edge cases, to catch coercion bugs before deployment.

---

## 4. Understand the Interaction Between `.optional()` and `.default()`

A reviewer questioned why `.optional()` was added to a previously required field. The answer: `.default(false)` means the field does not need to be provided (it falls back to the default), so adding `.optional()` makes the schema accurately reflect that the env var may be absent. In Zod 4, `z.stringbool().optional().default(false)` correctly handles: absent -> `false`, `"true"` -> `true`, `"false"` -> `false`.

**Guideline:** When a schema field has a `.default()` value, mark it `.optional()` to signal that the input may be absent -- this makes the schema self-documenting.
