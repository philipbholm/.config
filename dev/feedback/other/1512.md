# Learnings from PR #1512

Added a full-stack "registry sites" feature including Prisma schema, event-sourced domain model, GraphQL API, use cases (CRUD), integration tests, and React frontend pages.

---

## 1. Justify Every Import and Dependency in New Files

When a reviewer asks "why is this needed here?" about a specific import or piece of code, it signals that the code's purpose is not self-evident. Unnecessary imports or dependencies create coupling, increase bundle size, and confuse future readers. Every line in a new file should earn its place.

**Guideline:** Before submitting, audit each import in new files and remove anything that is not actively used or clearly required by the module's responsibility.

---

## 2. Choose Domain Names That Reflect Actual Usage, Not Internal Jargon

When naming fields like `abbreviation` in a domain model, consider whether a more intuitive name like `prefix` better describes how the value is actually used in the system. The field was used as a prefix for patient numbering (e.g., `TS-001`), making `prefix` a more accurate description than `abbreviation`. Poor naming forces every future reader to mentally translate between what the code says and what it means.

**Guideline:** Name domain fields based on how the value is used in the system, not what it technically is -- prefer `prefix` over `abbreviation` when the value prefixes identifiers.

---

## 3. Explain the Purpose of Runtime Assertions and Invariant Checks

When a reviewer asks "what's the point of this assertion?", the assertion's intent is not clear from context. Runtime assertions and invariant checks should either include a descriptive error message or a comment explaining what invariant they protect. Without this, they look like dead code or accidental leftovers, and future developers may remove them.

**Guideline:** Add a brief comment or descriptive error message to every runtime assertion explaining what invariant it guards and what would go wrong if it failed.
