# Learnings from PR #776

Added a gRPC health check endpoint to the codelist service, implemented graceful shutdown, and renamed "dependencies" to "ports" across the codebase.

---

## 1. Verify Infrastructure Configuration When Changing Health Check Protocols

When adding or changing health check endpoints, always check the existing infrastructure configuration (e.g., Terraform, load balancer settings) to ensure compatibility. In this case, the infrastructure was already configured for gRPC health checks on port 50051 via `/grpc.health.v1.Health/Check`. Adding a REST-based health check on a different port would have required coordinated infra changes. Failing to check this can lead to deployments where the load balancer cannot reach the new health endpoint, causing containers to be marked unhealthy and killed.

**Guideline:** Before changing how a service exposes health checks, review the infrastructure config (load balancer, target group, ECS task definition) to confirm the new approach is compatible or plan the infra changes alongside.

---

## 2. Return Shutdown Handles from Service Initializers

When starting servers (gRPC, HTTP, etc.), return a shutdown function from the initializer so the caller can orchestrate graceful shutdown. This avoids the anti-pattern of trapping signals deep inside library code and allows the top-level entry point to coordinate shutdown order across multiple servers.

```typescript
// Good: return shutdown handle
export const runGrpcHandler = async ({ ... }) => {
    await server.listen(`0.0.0.0:${port}`);
    return { shutdown: server.shutdown };
};

// Top-level orchestration
const { shutdown: shutdownHandlers } = await runHandlers({ ... });
registerShutdownHandler(shutdownHandlers);
```

**Guideline:** Every function that starts a long-running server or listener should return a shutdown/cleanup function rather than managing its own lifecycle internally.

---

## 3. Use Domain-Aligned Naming for Dependency Injection Parameters

Renaming `overrideDependencies` to `overridePorts` and `dependencies` to `ports` aligns the codebase with hexagonal architecture terminology. This makes the architectural intent explicit -- "ports" clearly signals the boundary between application logic and external adapters. Consistent naming reduces cognitive load when onboarding and makes it easier to reason about what can be swapped in tests.

**Guideline:** Use architecture-aligned naming (ports, adapters, use cases) consistently rather than generic terms like "dependencies" or "services" to make design intent self-documenting.
