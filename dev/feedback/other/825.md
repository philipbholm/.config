# Learnings from PR #825

This PR added support for deleting codelists and codelist records, including audit event cleanup for deleted records.

---

## 1. Export Use Case Return Types Instead of Inferring Them Inline

When writing test files, avoid using complex inline type inference like `Awaited<ReturnType<typeof testApplicationSetup.application.records.create>>`. Instead, export the return type directly from the use case module (e.g., `CreateRecordUseCase`) and reference it as `Awaited<ReturnType<CreateRecordUseCase["run"]>>`. The inline inference chains are hard to read and break if the intermediate object shape changes.

**Guideline:** Export use case types explicitly and reference them by name in tests rather than chaining `Awaited<ReturnType<typeof ...>>` through deeply nested objects.

---

## 2. Add Cross-Namespace Isolation Tests for Destructive Operations

When implementing delete or cleanup operations that are scoped to a namespace, add a test that creates a second namespace and verifies the delete operation does not affect it. This catches bugs where a missing `WHERE` clause or incorrect filter silently destroys data in other namespaces.

**Guideline:** For any destructive operation scoped by namespace (or similar tenant key), include at least one test asserting that records in a different namespace remain untouched.

---

## 3. Extract Repetitive Test Setup Into Helper Functions

When test setup involves repeated patterns like `encrypt(JSON.stringify(codelist))`, extract them into small helper functions (e.g., `encryptCodelist(codelist)`). This reduces noise in the test body and makes the intent of each setup step clearer without sacrificing readability.

**Guideline:** When the same encrypt/serialize/setup pattern appears more than twice in a test file, extract it into a named helper function at the bottom of the file.

---

## 4. Prefer `createMany` for Bulk Test Data Unless Ordering Matters

When seeding multiple records in test setup, `createMany` is more concise than sequential `create` calls. However, be aware that `createMany` does not guarantee insertion order, which makes it unsuitable when tests assert against `databaseEntry[0]` etc. by position.

**Guideline:** Use `createMany` for bulk test data setup when ordering is irrelevant; use sequential `create` calls only when assertion logic depends on insertion order.

---

## 5. Split Large Integration Tests Before They Become Unmaintainable

When a single integration test file grows to cover many edge cases with shared setup state (e.g., 500+ lines with nested `describe`/`beforeAll` blocks), it becomes hard to maintain. Consider splitting into smaller test files with isolated setup per scenario, or moving data-layer assertions to repository-level tests.

**Guideline:** When an integration test file exceeds ~300 lines or has more than two levels of nested `describe` with shared mutable state, evaluate splitting it into focused test files with independent setup.
