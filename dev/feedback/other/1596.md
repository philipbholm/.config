# Learnings from PR #1596

This PR updated registry cards in the frontend and backend, adding patient/collaborator counts and avatar displays to the registry listing.

---

## 1. Do Not Leave Unimplemented Code in the Codebase

Adding placeholder or stub code for features that are not yet built creates confusion for other developers who encounter it. They cannot tell whether it is broken, intentionally incomplete, or abandoned. If a feature is not ready, omit it entirely rather than shipping a skeleton.

**Guideline:** Never commit unimplemented stubs or placeholder fields to production code; only add code when the feature is actually being built.

---

## 2. Keep Backend Naming Domain-Neutral

Naming a GraphQL type after a frontend UI concept (e.g., `RegistryCard`) couples the backend schema to a specific frontend layout. If the frontend changes its card design to a list or table, the backend type name becomes misleading. Use domain-oriented names like `RegistryWithMetadata` instead.

**Guideline:** Name backend types and fields after domain concepts, not frontend UI components, so the API remains meaningful regardless of how clients render the data.

---

## 3. GraphQL Types Should Map 1:1 With Domain Types

When a separate internal type exists for a registry but the GraphQL type requires additional required fields (like `patientCount` and `collaboratorCount`), there is a mismatch that causes confusion about which is the source of truth. Keeping GraphQL types aligned with domain types avoids inconsistencies.

**Guideline:** Align GraphQL schema types directly with domain models so that resolvers do not need to fabricate or transform data into a different shape.

---

## 4. Prefer Composition Over Bloated Repository Methods

Adding many variant-specific methods (e.g., `getRegistryWithMetadata`, `getRegistryWithCounts`, `getRegistryWithUsers`) to a single repository or projection class creates a "god object" that grows uncontrollably. Instead, create focused query functions (like `getPatientCounts`) and compose them in the use case layer.

**Guideline:** Keep projection/repository methods focused on a single data concern; compose multiple queries at the use case level to avoid repository bloat.

---

## 5. Follow Existing Service Patterns for Cross-Cutting Concerns

When implementing a concept that already exists in another service (e.g., `StudyUser` in the studies service), replicate that pattern rather than inventing a new one. This keeps the codebase consistent and reduces the learning curve for developers moving between services.

**Guideline:** Before implementing a cross-cutting concept, check how sibling services handle it and follow the same pattern.

---

## 6. Keep Shared Test Setup Minimal

Adding custom configuration to shared test setup files (like a global `test-application.ts`) forces every test to inherit that configuration even when they do not need it. Each test should clearly define its own specific behavior for the dependencies it requires.

**Guideline:** Move custom test configurations into the specific test files that need them rather than adding them to shared test setup utilities.

---

## 7. Assign Presentation Logic to the Frontend

Determining things like "who is the owner" or "which users are collaborators" for display purposes is a presentation concern. Pushing this logic to the backend increases coupling and makes it harder to change the UI without backend changes. The frontend can derive these from the raw data it already receives.

**Guideline:** Keep presentation-only transformations (sorting, labeling, display formatting) in the frontend; the backend should return raw domain data.

---

## 8. Use Intuitive Icons and Verify Visual Communication

Using a folder icon for patient count and a person icon for collaborator count is counterintuitive since the semantic mapping is reversed. Consider using person icons for patients and avatar/initials circles for collaborators to match user expectations.

**Guideline:** Always validate that icon choices communicate the intended meaning by testing them with someone unfamiliar with the feature.

---

## 9. Add E2E and Integration Tests for New UI Features

When introducing a new visual component like registry cards, include both an end-to-end main success scenario test and integration tests for rendering. This catches regressions when the data contract or component dependencies change.

**Guideline:** Ship new UI features with at least one E2E happy-path test and integration tests covering the main rendering states.

---

## 10. Test Selectively in Assertions

In integration tests, only assert on fields that are relevant to the behavior being tested. Specifying every property in an assertion makes the test fragile and harder to read, especially when unrelated fields change.

**Guideline:** Use `expect.objectContaining()` or similar patterns to assert only on the fields that matter for the specific test case.
