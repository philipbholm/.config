# Learnings from PR #955

Added a wrapper shell script for docker compose that automatically seeds local development data after starting containers.

---

## 1. Place Helper Scripts in the Established Scripts Directory

When adding new utility scripts, place them in the existing conventional location (e.g., `scripts/`) rather than at the project root. Keeping scripts co-located makes them discoverable and avoids cluttering the top-level directory. When a reviewer suggests relocation, it usually signals that the repo already has an established convention.

**Guideline:** Before placing a new script, check where similar scripts already live in the repository and follow the existing convention.

---

## 2. Avoid Exposing the Docker Socket to Containers

Mounting the host Docker socket (`/var/run/docker.sock`) into a container gives that container full control over the Docker daemon, which is a security risk even in local development setups. Prefer wrapper scripts or init scripts that run on the host side instead of Docker-in-Docker approaches.

**Guideline:** Use host-side orchestration scripts rather than container-mounted Docker sockets for development automation tasks like database seeding.
