# Learnings from PR #1181

Added calculation form elements to the registries feature, including formula parsing, parameter mapping, and a new form element type with both backend projection logic and frontend UI.

---

## 1. Avoid type casting -- find type-safe alternatives

A reviewer flagged a type cast in the GraphQL mapper layer with "can we find a way that does not involve casting?" Type assertions (`as`) bypass the compiler's ability to catch type mismatches at the boundary where data enters your domain. If the shape changes upstream, the cast silently produces a lie.

**Guideline:** Prefer narrowing, discriminated unions, or schema validation over `as` casts, especially at data boundaries like GraphQL mappers.

---

## 2. Extract per-type creation functions when a factory grows too large

A reviewer noted: "I'm starting to think we should extract each type into a separate function, i.e. `createCalculationFormElement`, `createStaticTextFormElement` etc." When a projection or factory function handles many form element types in a single function, it becomes hard to follow and modify safely. Each type's logic should be isolated so that adding or changing one type does not risk breaking others.

**Guideline:** When a single function branches on many type discriminators, extract each branch into its own named function for readability and isolation.

---

## 3. Do not duplicate utility code across frontend and backend -- use a shared package

A reviewer identified that an `expression.ts` file was duplicated between frontend and backend, asking whether it should be a shared package. Duplicated parsing or evaluation logic will inevitably diverge, causing subtle inconsistencies between what the frontend validates and what the backend accepts.

**Guideline:** When the same logic (like formula parsing) is needed in both frontend and backend, extract it into a shared package and discuss the approach with the team.

---

## 4. Derive state instead of syncing it with useEffect

A reviewer pushed back on using a `missingParameter` state variable that was synchronized from other state via `useEffect`, saying: "using state unnecessarily like this is the cause of like 90% of bugs in the frontend." If a value can be computed from existing state or props, compute it inline or with `useMemo` rather than storing a derived copy that can fall out of sync.

**Guideline:** If a value can be derived from existing state, compute it on render rather than storing it in separate state synchronized via useEffect.

---

## 5. Use semantic HTML elements and component-provided labeling APIs

Multiple review comments called for using semantic elements (`fieldset`/`legend` instead of `span`) and using the component's `label` prop rather than manually rendering labels alongside inputs. Semantic HTML improves accessibility, screen reader support, and form association. Using a component's built-in `label` prop ensures consistent styling and proper `for`/`id` association.

**Guideline:** Always use semantic HTML elements (fieldset, legend, label) and leverage component `label` props instead of manually wrapping inputs with generic elements.

---

## 6. Narrow try-catch blocks and explain empty catches

A reviewer flagged a broad try-catch that silently swallowed errors and returned undefined. The author acknowledged the formula evaluation can throw when variables are missing, but the broad catch made this invisible. Narrowing the try-catch to just the throwing call, and adding a comment explaining why the catch is intentionally empty, makes the code's error-handling strategy explicit.

**Guideline:** Narrow try-catch to the smallest necessary scope, and always document the reason when using an empty catch block.

---

## 7. Add tests for complex derived-value logic

A reviewer noted "if not at least add a unit test or something so we know when we inevitably break it" about a complex calculation rendering component. When logic involves formula evaluation, variable substitution, and conditional rendering, tests are essential to prevent regressions -- especially in code paths that are hard to manually verify.

**Guideline:** Complex data transformation or formula evaluation code must have unit tests, particularly when the logic is dense and hard to review visually.

---

## 8. Be cautious with GraphQL InputMaybe types at the resolver-to-usecase boundary

A discussion revealed that GraphQL optional types are parsed as `InputMaybe<T>` (which is `T | undefined | null`), creating friction at the resolver-to-usecase boundary. The team decided to keep the current approach for now but recognized the real distinction between `null` and `undefined` may matter in the future.

**Guideline:** Be aware that GraphQL code generation produces three-valued optionals (`T | undefined | null`); document the convention for how your usecases handle the null vs. undefined distinction.

---

## 9. Use Prisma naming conventions for relation fields

A reviewer suggested using `calculationParameter CalculationParameter[]` (camelCase for the relation field name) in the Prisma schema, noting the need for a lint rule to enforce this. Consistent naming in schemas prevents confusion and ensures generated client code follows predictable patterns.

**Guideline:** Follow Prisma's convention of camelCase relation field names and consider linting to enforce schema naming consistency.
