# Learnings from PR #1063

This PR added the ability to create patient visit entries and submit form data answers in the registries service, spanning both backend domain events/projections and frontend form data entry UI.

---

## 1. Keep Application Composition Functions Focused

When wiring up use cases in a central application builder (e.g., `buildApplication`), avoid dumping all new use cases directly into the top-level function. Instead, group related use cases into their own wrapper functions (like `createPatientUseCases`). A flat, ever-growing builder function becomes hard to navigate and makes it unclear which use cases are related.

**Guideline:** Group related use cases into dedicated composition functions and call those from the top-level builder, rather than adding each use case individually.

---

## 2. Leverage Existing Metadata Schemas Instead of Duplicating Fields

When defining domain event schemas, check whether fields like `registryId` are already available on shared metadata schemas (e.g., `EntityEventMetadataSchema`). Duplicating data that the infrastructure already provides creates inconsistency risks -- the payload version could drift from the metadata version, and consumers have to decide which source of truth to use.

**Guideline:** Before adding a field to an event payload, check if it already exists in the event metadata schema and use that instead of duplicating it.

---

## 3. Extract Mapping Logic Out of GraphQL Resolvers

GraphQL resolvers that manually construct response objects inline become bloated and hard to read. Extracting the mapping to a dedicated mapper function keeps the resolver focused on orchestration (calling use cases, handling errors) while the mapper handles shape transformation. This also makes the mapping testable and reusable.

```typescript
// Instead of inline mapping in the resolver:
return {
  patientVisitEntry: {
    id: result.patientVisitEntry.id,
    patientId: result.patientVisitEntry.patientId,
    // ...many more fields
  },
};

// Extract to a mapper:
return {
  patientVisitEntry: toGqlPatientVisitEntry(result.patientVisitEntry),
};
```

**Guideline:** Extract response-shaping logic from resolvers into dedicated mapper functions when the mapping involves more than trivial field forwarding.

---

## 4. Skip Interface-Before-Implementation When Not Mocking

Defining a TypeScript interface for a projection class and then immediately implementing it adds indirection without benefit if you never mock that dependency in tests. You can always derive a type from the class later with `typeof` if needed. The extra interface means two things to maintain and forces you to jump through an indirection layer when navigating code.

**Guideline:** Only introduce a separate interface type for a class when you have a concrete need (e.g., multiple implementations, test mocking); otherwise, use the class directly and derive types with `typeof` if needed.

---

## 5. Prefer Route-Based Navigation Over In-Component Step State

Using `useState` to manage multi-step views within a single page component (e.g., `type: "patient-overview" | "visit-forms-list" | "form-data-entry"`) leads to complex conditional rendering, prop drilling of step-transition callbacks, and loss of browser navigation (back/forward). Using the router with layout + nested pages gives you URL-addressable states, cleaner component boundaries, and free browser history integration.

```typescript
// Instead of step state in one page:
const [currentStep, setCurrentStep] = useState<ViewStep>({
  type: "patient-overview",
});

// Use nested routes:
<Route path=":patientId" element={<PatientDetailLayout />}>
  <Route index element={<PatientOverviewPage />} />
  <Route path="visits/:visitId" element={<VisitFormsPage />} />
  <Route path="visits/:visitId/forms/:formId" element={<FormDataEntryPage />} />
</Route>
```

**Guideline:** When a UI has distinct "steps" or "views" with different data needs, model them as nested routes with a shared layout rather than toggling state within a single component.

---

## 6. Use Existing Icon Libraries Instead of Inline SVGs

Inline SVG markup in components is hard to read, maintain, and keep consistent. When the project already has an icon library (e.g., Lucide), use it. If the icon does not exist in the library, extract the SVG into a named icon component so its purpose is clear from the component name rather than requiring someone to decode raw SVG paths.

**Guideline:** Always check the existing icon library first; if the icon is unavailable, create a named component for it rather than inlining SVG markup.

---

## 7. Consider Collocating Form Submission with Form Display

When a form submission handler lives in a parent component while the form rendering lives in a child, the parent accumulates orchestration complexity (useEffect for state sync, mutation state management, etc.). Moving the submission logic into the component that owns the form display often simplifies both components and reduces prop threading.

**Guideline:** Collocate mutation/submission logic with the form component that renders the fields, unless there is a specific reason for the parent to orchestrate submission.

---

## 8. Consolidate Duplicate Imports from the Same Module

Importing different symbols from the same module in separate import statements (e.g., two import lines from the same events file) clutters the imports section and is flagged by linters. Merge them into a single import statement.

```typescript
// Instead of:
import { ConnectionInvitationCreated } from "./connection-invitation/events";
import { RegistryConnectionAccepted } from "./connection-invitation/events";

// Consolidate:
import {
  ConnectionInvitationCreated,
  RegistryConnectionAccepted,
} from "./connection-invitation/events";
```

**Guideline:** Always merge imports from the same module into a single import statement to keep the imports section clean and avoid linter warnings.
