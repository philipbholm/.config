# Learnings from PR #1312

Added display variant support (dropdown, radio vertical, radio horizontal, checkbox vertical, checkbox horizontal) for category and multi-value category form elements.

---

## 1. Prefer Making Props Optional Over Using No-Op Callbacks

A reviewer flagged an empty `onChange={() => undefined}` being passed to a RadioBox component in preview/disabled mode. The ideal fix would be to make the `onChange` prop optional in the component's API so callers do not need to pass a meaningless handler. When a component is in a read-only or preview state, requiring a change handler is a design smell.

```typescript
// Current: no-op callback to satisfy required prop
<RadioBox
  value=""
  onChange={() => undefined}  // reviewer flagged this
  disabled
/>
```

**Guideline:** When a component supports a disabled/preview mode, make interactive callbacks (onChange, onSubmit) optional in the component's type signature rather than forcing callers to pass no-op functions.

---

## 2. Use Discriminated Rendering for Variant-Based Components

The PR correctly uses early-return patterns to render different component trees based on the `variant` prop (DROPDOWN renders Select, RADIO_* renders RadioBox). This is cleaner than conditional props or complex ternaries within a single JSX tree.

```typescript
if (variant === "DROPDOWN") {
  return <Select ... />;
}

return <RadioBox direction={variant === "RADIO_HORIZONTAL" ? "horizontal" : "vertical"} ... />;
```

**Guideline:** When a component has display variants that map to fundamentally different UI elements, use early returns to render each variant separately rather than trying to conditionally configure a single component.

---

## 3. Use Type Extraction Over Direct GraphQL Type Imports

The PR shifted from importing generated GraphQL types directly (e.g., `RegistryCategoryFormInputElement`) to using `Extract<FormElement, { __typename: "..." }>`. This pattern ensures the type stays in sync with the GraphQL union/fragment and avoids importing overly broad generated types that may include fields not requested in the query.

```typescript
type CategoryFormElement = Extract<
  FormElement,
  { __typename: "RegistryCategoryFormInputElement" }
>;
```

**Guideline:** When working with GraphQL union types, prefer `Extract<UnionType, { __typename: "..." }>` over importing the full generated type to ensure your component type matches what the query actually returns.

---

## 4. Use GraphQL Field Aliases to Avoid Naming Collisions

Both `RegistryCategoryFormInputElement` and `RegistryMultiValueCategoryFormInputElement` have a `variant` field, but they have different enum types. The PR used GraphQL aliases (`categoryVariant: variant` and `multiValueCategoryVariant: variant`) to give them distinct names in the query result, preventing ambiguity in the TypeScript types.

```graphql
... on RegistryCategoryFormInputElement {
  categoryVariant: variant
}
... on RegistryMultiValueCategoryFormInputElement {
  multiValueCategoryVariant: variant
}
```

**Guideline:** Use GraphQL field aliases when multiple types in a union share the same field name but with different semantics or enum types, to produce unambiguous TypeScript types.
