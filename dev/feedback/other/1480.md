# Learnings from PR #1480

Fixed a bug where form element updates were not saved because the SharedFormSaveDialog returned null for non-shared forms without invoking the save callback, and refactored the save logic into a reusable hook.

---

## 1. Avoid Custom Testing Abstractions -- Use Existing Primitives

The PR introduced a new test utility function for rendering form elements. The reviewer pushed back: "Why do we need this? Isn't `testRender` sufficient? I'm very wary of introducing testing abstractions, because they so quickly become hard to maintain and also make the tests hard to reason about." The project already had `testRender` and `registriesMocks()` as established test primitives.

**Guideline:** Before creating new test helper functions, check if existing test utilities (`testRender`, mock builders like `registriesMocks()`) already cover the need -- prefer composing existing primitives over introducing new abstractions.

---

## 2. Use Existing Mock Builders Instead of Inline Custom Setup

A reviewer pointed to `registries-mocks` as a better alternative to building custom MSW handler setup logic inside test files. Custom mock setup logic with if/else branching in test files "quickly deteriorates into a long function nobody really understands anymore as the test cases expand."

```typescript
// Preferred: declarative mock builder
registriesMocks()
  .withFormUsage([{ visitId: "visit-1", visitName: "Visit 1" }])
  .withUpdateFormInputElement({ formElementId: ELEMENT_ID, onMutate: spy })
  .apply();
```

**Guideline:** When mocking GraphQL or API responses in tests, use the project's mock builder pattern (e.g., `registriesMocks().withX().apply()`) rather than writing one-off MSW handlers inline.

---

## 3. Unit Test Extracted Hooks Independently

When refactoring logic from a component into a custom hook (like `useSaveWithSharedFormConfirmation`), the reviewer asked for the hook to have its own unit tests in addition to the component-level integration tests. Hook tests are faster, more focused, and make it clear which behavior belongs to the hook versus the component.

**Guideline:** When extracting shared logic into a custom React hook, write dedicated `renderHook` unit tests for the hook's behavior separate from the component tests that consume it.
