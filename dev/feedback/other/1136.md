# Learnings from PR #1136

Added GraphQL DoS protection middleware across three services to limit the number of operations and directives per request, rejecting excessive requests with a 400 error before they reach Apollo.

---

## 1. Extract shared middleware into a package instead of duplicating across services

The same `validateGraphqlMiddleware` and `stripCommentsAndStrings` functions were copied identically into `services/admin`, `services/registries`, and `services/studies`. When validation logic is shared across multiple services, it should live in a shared package so bug fixes and improvements propagate to all consumers automatically. Duplicated code means duplicated bugs and divergent behavior over time.

**Guideline:** If the same utility is needed by more than one service, extract it into a shared package rather than copy-pasting it.

---

## 2. Regex-based validation must account for all valid input forms

The initial regex `/(query|mutation|subscription)\s+\w+/g` only matched named operations (e.g., `query GetUser { ... }`) and completely missed anonymous operations (`query { ... }`). A reviewer correctly identified this as a security gap -- attackers could bypass the DoS limit simply by omitting operation names. The fix was to make the name capture group optional: `/(query|mutation|subscription)(\s+\w+)?/g`.

**Guideline:** When writing regex-based validation for security purposes, enumerate all syntactically valid forms of the input and test each one explicitly.

---

## 3. Strip comments and string literals before parsing untrusted text with regex

The final implementation added a `stripCommentsAndStrings` function that removes GraphQL comments (`# ...`), block strings (`"""..."""`), and regular strings (`"..."`) before counting operations and directives. Without this, an attacker could embed operation keywords inside string arguments or comments to produce false counts, or hide real operations inside comment blocks to evade detection.

```typescript
function stripCommentsAndStrings(query: string): string {
    let result = query;
    result = result.replace(/"""[\s\S]*?"""/g, '""');
    result = result.replace(/"[^"\\]*(?:\\.[^"\\]*)*"/g, '""');
    result = result.replace(/#[^\n]*/g, "");
    return result;
}
```

**Guideline:** When counting or validating patterns in a structured language, first strip non-semantic content (comments, string literals) to avoid false positives and bypasses.

---

## 4. Keep security fixes scoped to the specific vulnerability being addressed

A reviewer suggested adding rate limiting to guard against "volumetric DDoS" in addition to the operation/directive limits. The PR author correctly pushed back, noting the PR was specifically addressing operation-level DoS vulnerabilities from a pentest report. Scope creep in security fixes delays remediation and complicates review.

**Guideline:** Address the specific vulnerability identified in a security finding; file separate issues for related but distinct protections like rate limiting.

---

## 5. Write tests for security-critical middleware

A reviewer asked "Is this something we should have tests for?" -- and yes, comprehensive unit tests were added covering named operations, anonymous operations, operations hidden in strings/comments, directives in various positions, and edge cases like empty queries and GET requests. Security middleware without tests is a liability because regressions silently re-open vulnerabilities.

**Guideline:** Security validation logic must have thorough test coverage including bypass attempts and edge cases.

---

## 6. Preserve error codes through middleware layers

The PR updated error formatting in the studies service to preserve middleware-generated error codes (like `TOO_MANY_OPERATIONS`) rather than letting Apollo's default error formatter strip them. When middleware rejects a request with a structured error, downstream error formatting must pass those codes through so clients can programmatically identify the rejection reason.

**Guideline:** When adding pre-handler middleware that returns structured errors, verify that the service's error formatting pipeline does not strip the error codes or extensions.
