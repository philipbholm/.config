# Learnings from PR #1587

Adding codelist-based patient search with PII filters in both the backend query layer and a frontend popover UI with active search criteria display.

---

## 1. Do Not Create Barrel Files (index.ts Re-Exports)

Barrel files (`index.ts` that just re-export from other files) add indirection without value. They make it harder to trace imports, can cause circular dependency issues, and are frequently generated unnecessarily by AI coding tools. Use direct imports to the actual module instead.

**Guideline:** Import directly from the source file (e.g., `./CodelistSearchPopover`) rather than creating or using barrel `index.ts` files.

---

## 2. Extract Branching Logic Into Named Helper Functions

When a use case's `run` method contains a large conditional block (e.g., "if PII filters exist, do X"), extract it into a named helper function like `getPiiFilteredPatients(...)`. This keeps the main flow readable and makes the branching intent self-documenting.

```typescript
if (input.piiFilters) {
  return getPiiFilteredPatients(...)
}
```

**Guideline:** When a conditional branch in a use case exceeds 5-10 lines, extract it into a descriptively named helper function.

---

## 3. Use Existing Design System Components Instead of Custom Styling

When the design system already provides components like tags/badges (e.g., shadcn Badge), use them instead of crafting custom styled elements with raw Tailwind classes. This ensures visual consistency and reduces maintenance burden when the design system evolves.

**Guideline:** Before building custom styled elements, check if the design system (shadcn) already provides a matching component.

---

## 4. Avoid Unnecessary Indirection Between Dictionary Keys and Domain Keys

If your form fields and dictionary keys can share the same identifiers, use them directly rather than creating a separate mapping layer. Extra mappings add boilerplate and a maintenance surface without improving clarity.

**Guideline:** Use domain keys directly as dictionary keys when possible to avoid redundant key-to-label mapping objects.

---

## 5. Prefer Integration Tests Over Many E2E Tests for Edge Cases

E2E (Playwright) tests are slow and expensive. Use a single E2E test for the "main success scenario" and push edge case coverage into faster integration tests. This keeps the E2E suite fast while maintaining thorough coverage.

**Guideline:** Write one E2E test per feature for the happy path, and cover edge cases and error states with integration tests.

---

## 6. Use Resolver Mappers for GraphQL Input Transformation

When GraphQL inputs need null-to-undefined coercion or other transformations, extract a dedicated mapper function (e.g., `mapPiiFiltersInput`) rather than inlining the mapping in the resolver. This keeps resolvers thin and the transformation testable.

**Guideline:** Extract GraphQL input transformations into named mapper functions co-located with the resolver.

---

## 7. Be Mindful of AI Instruction Context Bloat

Adding too many rules to shared AI instruction files (CLAUDE.md, AGENTS.md) can degrade AI performance through context bloat. Keep shared instructions minimal and put personal preferences in local files (`.CLAUDE.local.md`). Symlink between AI tools (Claude Code, Cursor) to share personal instructions.

**Guideline:** Keep shared AI instruction files lean and use personal local instruction files for individual preferences, symlinking between tools as needed.
