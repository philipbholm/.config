# Learnings from PR #1463

This PR added unique constraints to database models (categories, visits, form elements, treatments) with both database-level indexes and application-level validation.

---

## 1. Register Custom Error Types in the Error Handler

When introducing a new error class (e.g., `ValidationError`), you must also add it to the error-type checking function (like `isApplicationError()`). Otherwise, the GraphQL layer will swallow the friendly message and return a generic "Internal server error" to the user, making the validation completely invisible.

**Guideline:** Whenever you create or use a new error subclass, verify it is recognized by the error serialization layer so its message reaches the client.

---

## 2. Add Application-Level Validation for Every Mutation That Can Violate a Constraint

Adding a database unique constraint is necessary but not sufficient. Every use case that can trigger the constraint -- create AND update/rename -- must have application-level validation. Without it, the user sees a raw database error (e.g., Prisma P2002) instead of a helpful message.

In this PR, `rename-visit` initially lacked duplicate-name validation despite the new `(treatmentId, name)` unique index. The same pattern applied to `create-category-structure`.

**Guideline:** For every new database constraint, enumerate all code paths that write to that table and add explicit validation with a user-friendly error message in each one.

---

## 3. Understand PostgreSQL NULL Semantics in Unique Constraints

PostgreSQL does not consider `NULL = NULL` for unique constraint purposes. If a unique constraint includes a nullable column (e.g., `@@unique([treatmentId, name])` where `treatmentId` is nullable), multiple rows with `NULL` in that column and the same value in the other column will NOT violate the constraint. This can silently allow duplicates.

The fix in this PR was to make `treatmentId` non-nullable since visits always have a treatment in practice.

**Guideline:** When adding a unique constraint involving a nullable column, either make the column non-nullable or add application-level validation to cover the NULL case.

---

## 4. Plan Data Migration Before Adding Constraints to Existing Tables

Adding a unique index to a table with existing duplicate data will cause the migration to fail, blocking deployment. Similarly, making a nullable column non-nullable fails if NULL rows exist. The migration must include a data cleanup step (e.g., `DELETE FROM "visits" WHERE "treatment_id" IS NULL`) before applying the schema change.

**Guideline:** When adding constraints to existing tables, always include a pre-constraint data cleanup step in the migration and document why it is safe.

---

## 5. Exclude Self When Validating Uniqueness on Updates

When checking for duplicates during an update operation, you must filter out the entity being updated from the comparison set. Otherwise, updating a record without changing its name/label will incorrectly report a duplicate against itself.

```typescript
assertUniqueLabelInFormVersion({
    label: input.newLabel,
    existingFormElements: existingFormElements.filter(
        (e) => e.formElementId !== input.formElementId,
    ),
});
```

**Guideline:** Always exclude the current entity from uniqueness checks during update operations to avoid false-positive self-collision.

---

## 6. Write Tests for Both Positive and Self-Collision Cases

When adding uniqueness validation, test both: (a) that a true duplicate is rejected, and (b) that updating an entity with its own current value succeeds. The self-collision test guards against overly aggressive validation that would break normal save operations.

**Guideline:** For every uniqueness validation, include an integration test for the duplicate case and an integration test confirming the self-update case passes.

---

## 7. Hoist Shared Queries When Multiple Checks Need the Same Data

If both uniqueness validation and circular-dependency checks need the same `existingFormElements` list, fetch it once and pass it to both. The original code fetched `existingFormElements` only inside the `if (input.visibilityCondition)` block; refactoring it out enabled reuse by the new label-uniqueness check without an extra database query.

**Guideline:** When adding a new validation that needs data already fetched in a conditional block, hoist the query above both conditionals to avoid redundant fetches.
