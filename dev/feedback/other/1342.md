# Learnings from PR #1342

Migrates category form element answers from category IDs to option IDs, centralizing answer storage through a new options table and removing the `CategoryStructureProjection` dependency.

---

## 1. Follow Existing Naming Conventions Even When They Are Suboptimal

When adding a new GraphQL input type like `CategoryMappingElementRegistryInput`, the reviewer asked for `RegistryCategoryMappingElementInput` (prefix-first). The author pointed out that the existing types in the file already use the suffix pattern. The resolution was to follow the existing convention for consistency, and fix naming holistically in a separate effort.

```graphql
# Existing convention in the file (suffix pattern):
input AddSeparatorRegistryFormElementInput { ... }

# New type followed the existing pattern:
input CategoryMappingElementRegistryInput { ... }
```

**Guideline:** Match the naming convention already established in the surrounding code, even if imperfect -- inconsistency is worse than a suboptimal but uniform pattern.

---

## 2. Write Actionable TODO Comments with Specific Context

A reviewer flagged a vague `// TODO: What happens here?` comment as insufficient. TODO comments should describe the specific concern, the expected behavior, and what needs investigation. Vague TODOs become permanent noise that nobody acts on because they lack enough context to be actionable.

```ts
// BAD:
// TODO: What happens here?

// GOOD:
// TODO: Review how categoryMapping interacts with the new option-based
// system. Ensure mappingItems and categoryId/value logic correctly
// handle cases where options may have changed structure or identifiers.
```

**Guideline:** Every TODO comment must state what the problem is, why it matters, and what specifically needs to be done or investigated.
