# Learnings from PR #1475

Added notification triggers for form elements, enabling automated email notifications when specific category form answers are submitted in a study.

---

## 1. Maintain Visual Indication for Disabled Interactive Elements

When restyling a component (in this case a Checkbox), the disabled state's `opacity-50` class was accidentally removed along with other styling changes. The reviewer caught that there was no longer any visual indication that the checkbox was disabled, beyond the cursor change on hover. Disabled elements must remain visually distinguishable from enabled ones for both usability and accessibility.

**Guideline:** When refactoring component styles, always verify that all interactive states (disabled, focused, hover, active) remain visually distinct and accessible.

---

## 2. Remove Leftover Debug or Temporary Code Before Merging

A reviewer flagged leftover code in `StudyChangeLog.tsx` with a simple "rm" comment. Even small remnants of temporary code, console logs, or commented-out blocks create noise and confusion for future readers. These are easy to miss in large PRs.

**Guideline:** Before requesting review, search the diff for any temporary code, debug statements, or unintentional changes that are not part of the feature.

---

## 3. Protect Referenced Configuration from Deletion

The approving reviewer noted: "Maybe look at whether category options used in triggers should be blocked from removal." When entity A references entity B by ID (e.g., a notification trigger references a category option value), deleting entity B can silently break entity A. Either prevent deletion of referenced entities or cascade the cleanup.

**Guideline:** When a new feature creates references to existing configurable entities, add validation that prevents deletion of those entities while they are still referenced, or handle the orphaned reference gracefully.

---

## 4. Verify Intentionality of Behavioral Changes

A reviewer asked "Is this intentional?" about a change in `FormProjection.ts`. When a diff contains changes that look incidental or unrelated to the PR's purpose, reviewers cannot tell if they are deliberate or accidental. Unexplained behavioral changes slow down review and risk introducing unintended side effects.

**Guideline:** If a PR includes changes outside the main feature scope, add a brief inline comment or PR description note explaining why they are necessary.
