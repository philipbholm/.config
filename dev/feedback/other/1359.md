# Learnings from PR #1359

This PR attempted to implement category structures for reusing category options across variables, modeled after the existing trials implementation, but was closed without merge after a reviewer flagged an architectural direction change.

---

## 1. Verify Architectural Direction Before Implementing Features

The PR copied an existing pattern (structures) that the team was actively migrating away from in favor of a simpler "Options" approach. Structures were being reserved for large standard structures like ICD-10. Building on a deprecated pattern wastes effort and creates code that will need to be rewritten.

**Guideline:** Before implementing a feature by copying an existing pattern, confirm with the team that the pattern is still the preferred approach and has not been superseded.

---

## 2. Distinguish Between Reusable Patterns and Deprecated Patterns

A pattern existing in the codebase does not mean it is the right pattern to follow. In this case, the trials implementation was a valid reference, but the team had already decided to move away from structures for simple category options. The presence of code is not an endorsement of that approach.

**Guideline:** When a PR description says "basically a copy of X implementation," treat that as a signal to ask whether X is still the canonical approach.

---

## 3. Communicate Design Decisions Early in Large PRs

This PR had 965 additions across 14 files before the fundamental approach was questioned. An early design check (even a Slack message or draft PR) would have saved the effort of a full implementation that was immediately closed.

**Guideline:** For features that introduce new domain concepts or copy architectural patterns, open a draft PR or discuss the approach before building the full implementation.
