# Learnings from PR #1341

Adds conditional visibility for registry form variables, including a visibility condition evaluator on the frontend and backend event/projection support for toggling variable visibility based on category selections.

---

## 1. Write Unit Tests for Pure Logic Extracted into Utility Files

When you extract evaluation logic into a standalone utility file (like `visibility-condition-evaluator.ts`), that file becomes a natural unit-testing target. Pure functions that evaluate conditions against data are easy to test in isolation and are especially important to cover because bugs in condition evaluation silently produce wrong UI states. Integration tests cover the full stack, but unit tests catch edge cases in the logic itself much faster.

**Guideline:** When you create a new utility file with pure evaluation logic, add unit tests for it in the same PR -- do not defer testing to a follow-up.

---

## 2. Avoid Implicit Ordering Requirements in Database Queries

When querying versioned records with `findFirstOrThrow` + `orderBy: { version: "desc" }`, every caller must remember to sort by version to get the latest record. If any caller forgets, it silently gets the wrong version. This is a latent bug source that scales with the number of query sites.

```ts
await this.getClient().registryFormElementVersion.findFirstOrThrow({
    where: { formElementId: newCondition.categoryElementId },
    orderBy: { version: "desc" },
});
```

**Guideline:** Encapsulate "get latest version" logic in a single repository method so callers cannot forget the sort order.

---

## 3. Prefix Entity IDs with a Recognizable Type Tag

Using bare `randomUUID()` for entity IDs makes it impossible to identify the entity type from the ID alone during debugging, logging, and cross-service tracing. Prefixing IDs (e.g., `reg_`, `frm_`) makes logs self-documenting and prevents accidentally passing an ID of the wrong type. Consider compact ID formats like nanoid or cuid for shorter, URL-safe identifiers.

```ts
const formElementId = randomUUID(); // bare UUID -- hard to distinguish types
```

**Guideline:** Prefix all entity IDs with a 3-4 character type identifier, and consider using nanoid/cuid instead of raw UUIDs for readability.

---

## 4. Ensure Multi-Query Projection Updates Run Inside a Transaction

When a projection handler performs multiple dependent database writes (read a version, compute a condition, upsert a record), all operations must share the same transaction. If they do not, a crash between writes leaves the projection in an inconsistent state. In this codebase, `createAndPublishEvent` wraps everything in a transaction -- but this is an implicit guarantee that new contributors may not know about.

**Guideline:** Document or enforce at the framework level that projection handlers always run inside a transaction, so new code cannot accidentally bypass this guarantee.
