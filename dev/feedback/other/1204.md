# Learnings from PR #1204

This PR added registry design UX improvements including visit reordering (drag-and-drop), visit renaming, visit deletion, and form deletion, spanning both frontend and backend (event-sourced domain events, projections, GraphQL resolvers, and integration tests).

---

## 1. Use the Project's Structured Error Reporting Instead of console.error

When catching errors in frontend code, use the project's `captureException` utility (which routes to Datadog or similar observability tools) rather than `console.error`. Logging to the browser console is invisible in production and makes errors undetectable.

**Guideline:** Always use the project's structured error-capture mechanism (e.g., `captureException`) so errors are surfaced in the monitoring system, not just the browser console.

---

## 2. Prefer Batch/Transactional Database Operations Over Sequential Updates

When updating multiple rows (e.g., reordering visits), consider whether `updateMany` or a single transactional operation is more appropriate than looping with individual `update` calls. Sequential updates risk partial failures leaving data in an inconsistent state.

```typescript
// The PR used sequential updates with a two-pass approach to avoid unique constraint conflicts:
for (const visitOrder of event.payload.visitOrders) {
    await client.visit.update({
        where: { id: visitOrder.visitId },
        data: { order: -(maxOrder + 1 + visitOrder.order) },
    });
}
for (const visitOrder of event.payload.visitOrders) {
    await client.visit.update({
        where: { id: visitOrder.visitId },
        data: { order: visitOrder.order },
    });
}
```

The counterpoint raised was that all Prisma changes within event dispatch are already transactional by framework convention. Even so, batch operations are clearer in intent and more efficient.

**Guideline:** When updating multiple related rows, prefer batch operations or explicitly document why sequential updates are safe (e.g., because they run inside a framework-managed transaction).

---

## 3. Document Complex Workarounds with a Rationale

The two-pass order update (first setting negative orders, then correcting them) is a workaround for unique-constraint conflicts during reordering. This pattern is non-obvious and needs a clear comment explaining the constraint being avoided, so future readers do not accidentally simplify it.

**Guideline:** When a workaround exists for a database constraint or concurrency issue, add a code comment explaining the specific problem it solves and why the simpler approach does not work.

---

## 4. Validate Experimental Config Changes Before Committing

The PR included speculative Docker/Vite config changes with comments like "Does this fix the issue with the checker not updating inside docker??" These should be validated before merging, as speculative polling configs (e.g., `usePolling: true` with a 1-second interval) can degrade dev performance and clutter configuration.

```typescript
server: {
    // Does this fix the issue with the checker not updating inside docker??
    watch: {
        usePolling: true,
        interval: 1000,
    },
},
```

**Guideline:** Do not commit speculative configuration changes with question-mark comments; validate them first, or track the investigation in an issue and apply the fix in a dedicated PR.
