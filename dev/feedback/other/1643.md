# Learnings from PR #1643

Addressed PR review feedback for the forms overview page: replaced manual TypeScript interfaces with generated GraphQL types, added error handling for delete mutations, optimized DB queries with early returns, and added component tests.

---

## 1. Use Generated Types Instead of Manual Interface Definitions

When GraphQL code generation produces types, use them directly rather than re-declaring equivalent interfaces manually. Manual interfaces drift from the schema over time and create maintenance burden. The PR replaced hand-written `Form` and `FormVersion` interfaces with a type extracted from the generated query:

```typescript
type Form = GetRegistryFormsQuery["getRegistryForms"]["forms"][number];
```

**Guideline:** Always derive frontend types from generated GraphQL types (using indexed access types) rather than maintaining parallel manual interface definitions.

---

## 2. Prefer Direct Luxon Calls Over Wrapper Utilities for Date Formatting

Rather than creating or using a `formatDate` utility function that wraps Luxon, use `DateTime.fromISO(...).toLocaleString(DateTime.DATE_MED)` directly. Wrapper functions add an unnecessary layer of indirection, and when they lack locale parameters, they silently break localization (e.g., showing English dates for Norwegian users). The recommendation from review was to remove the `formatDate` utility entirely.

**Guideline:** Use Luxon's `DateTime` API directly for date formatting; avoid utility wrappers that obscure locale handling.

---

## 3. Add Early Returns for Empty Array Queries

When a database query depends on a list of IDs (e.g., `formIds`), check for an empty array before issuing the query. Some ORMs or databases handle `WHERE id IN ()` poorly or inefficiently. The PR added early returns in two projection methods:

```typescript
if (formIds.length === 0) {
    return [];
}
```

**Guideline:** Guard database queries that filter by an array of IDs with an early return when the array is empty, to avoid unnecessary round-trips and potential ORM edge cases.

---

## 4. Use Domain-Specific Error Types Instead of Generic Errors

The PR changed `throw new Error(...)` to `throw new FailedPreconditionError(...)` in `dbFormToForm`. Using domain-specific error types enables the error handling middleware to categorize them correctly (e.g., mapping to the right gRPC status code, deciding whether to log to Sentry). A generic `Error` would be treated as an unexpected server error, triggering alerts for what is actually a data integrity issue.

**Guideline:** Throw domain-specific error types (e.g., `FailedPreconditionError`, `NotFoundError`) rather than generic `Error` so middleware and monitoring can classify and handle them appropriately.

---

## 5. Question Whether Test Mocks Are Necessary

A review comment asked "do we need to mock this?" about a mock in the FormsTable tests. Over-mocking is a common test anti-pattern that makes tests brittle and hard to maintain. Each mock is a maintenance liability and a potential source of false positives. Before adding a mock, verify that the dependency actually needs to be replaced for the test to work.

**Guideline:** Only mock dependencies when the test cannot function without the mock; prefer real implementations or lightweight stubs to keep tests realistic and maintainable.
