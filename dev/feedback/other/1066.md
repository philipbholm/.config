# Learnings from PR #1066

This PR implemented two-way role creation when linking registries, extended single-site-registry endpoints to read forms and form elements, and added frontend components for creating/adding forms to visits.

---

## 1. Use `captureException` Instead of `console.error` for Error Logging

Using `console.error` in production code pollutes the console and provides no observability. The codebase has a `captureException` utility that routes errors to Datadog, giving structured error tracking with context (e.g., registryId, visitId, formId). Always prefer the project's error reporting tool over raw console statements.

**Guideline:** Never use `console.error` for error handling in production code; use `captureException` with relevant context so errors are tracked in your observability platform.

---

## 2. Internationalize All User-Facing Strings, Including Toast Messages

Hardcoded English strings in toast notifications ("Form added to visit", "Failed to add form to visit") and button labels ("Add form") were flagged. When a component already has a translation dictionary, all user-visible text must go through it, not just labels and headings.

**Guideline:** Every user-facing string -- including toast messages, button text, and error messages -- must use the translation dictionary, never inline English literals.

---

## 3. Batch Permission Checks Instead of Looping Per Entity

When checking permissions across multiple registries, avoid iterating and making one authorization call per registry. This creates N database transactions where a single batched query could suffice. The code was looping through `allSingleSiteRegistriesUserIsIn` with individual `hasPermission` calls.

**Guideline:** Create query functions that accept multiple entities at once rather than calling single-entity functions in a loop, to reduce transaction overhead.

---

## 4. Use Dependency Object Pattern When Constructor Has More Than 4 Parameters

A class constructor with 5+ positional parameters (PrismaClient, EventStore, PrismaTransactionManager, etc.) is hard to read and easy to misorder. Refactoring to accept a single `dependencies` object makes the call site self-documenting and order-independent.

```typescript
// Before (fragile ordering)
constructor(
    private readonly client: PrismaClient,
    private readonly eventStore: EventStore,
    private readonly transactionManager: PrismaTransactionManager,
    private readonly singleSiteAuth: SingleSiteAuthorizationService,
    private readonly multiSiteAuth: MultiSiteAuthorizationService,
) {}

// After (self-documenting)
constructor(dependencies: {
    client: PrismaClient;
    eventStore: EventStore;
    transactionManager: PrismaTransactionManager;
    singleSiteAuthorizationService: SingleSiteAuthorizationService;
    multiSiteAuthorizationService: MultiSiteAuthorizationService;
}) {}
```

**Guideline:** When a constructor or function exceeds 4 parameters, switch to a single options/dependencies object to prevent ordering mistakes and improve readability.

---

## 5. Name Variables Carefully to Avoid Misleading Semantics

A method named `getVisitCount` was used to determine the ordering of a form, but the reviewer questioned whether "visit count" is the correct concept for "form order." Misleading names create confusion about what a value actually represents.

**Guideline:** Ensure variable and method names precisely describe what they return; if a count is used to derive an order, name it to reflect its actual purpose (e.g., `getNextFormOrder`).
