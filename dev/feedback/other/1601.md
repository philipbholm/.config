# Learnings from PR #1601

This PR added a Forms Overview page to the registry design section, with a table listing forms, search filtering, delete confirmation dialog, and form usage display.

---

## 1. Handle Mutation Errors Visibly

When a destructive action like form deletion fails, the user needs clear feedback. Without error handling in the UI (like a toast or inline error message), the dialog simply closes and the user has no idea the operation failed. Always pair mutation calls with `onError` feedback.

**Guideline:** Every mutation that affects user data must have visible error handling -- show a toast, alert, or inline message when the operation fails.

---

## 2. Use the Project's Standard Date Library

If the project already uses a date library (e.g., luxon), do not introduce raw `Date` manipulation or alternative formatting approaches. Inconsistent date handling leads to locale bugs and makes it harder to change formatting globally. In this codebase, use `DateTime.fromISO()` from luxon.

**Guideline:** Always use the project's established date library (luxon) for parsing and formatting dates rather than native Date methods or ad-hoc formatting.

---

## 3. Use Generated Types From the GraphQL Schema

When working with GraphQL query results, import types from the generated type definitions (e.g., `~/generated/graphql`) rather than defining manual TypeScript interfaces. Manual types drift from the schema and miss fields that get added or renamed.

**Guideline:** Always use codegen-generated GraphQL types for query results and mutation inputs to stay in sync with the schema automatically.

---

## 4. Write Tests for Destructive Actions

Features with destructive operations (like deleting a form) need dedicated test coverage. Test both the happy path (correct name typed, mutation called with right variables) and the error path (mutation fails, error displayed). The confirmation dialog flow is particularly important to test.

**Guideline:** Every destructive action in the UI must have tests covering the confirmation flow, successful execution, and error handling.

---

## 5. Use Specific Error Types Instead of Generic Errors

When a precondition fails (like a form missing its registry ID), throw a domain-specific error (e.g., `FailedPreconditionError`) rather than a generic `Error`. Specific error types enable better error handling upstream and more informative error messages for debugging.

**Guideline:** Replace generic `throw new Error(...)` with domain-specific error classes that convey the category of failure.

---

## 6. Show Useful Context in Overview Tables

When listing entities in a table, include metadata that helps users make decisions without drilling into each item. For example, showing how many events use a form (usage count) helps users understand the impact of deleting or modifying it.

**Guideline:** Include actionable context columns (like usage counts, status, or last-modified dates) in overview tables so users can make informed decisions at a glance.
