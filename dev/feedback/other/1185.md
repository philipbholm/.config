# Learnings from PR #1185

Fixed codelist integration issues in the registries service: corrected credential field names (snake_case from AWS Secrets Manager), added JSON.parse for the secret value, and ensured the codelist API was copied in the Dockerfile.

---

## 1. Verify multi-stage Dockerfile COPY instructions at every stage that needs the files

A reviewer noted that adding `COPY services/codelist/api` at line 4 of the Dockerfile might not be sufficient -- it likely also needed to be added at the second build stage (line 28). When a multi-stage Dockerfile has separate `COPY` instructions for builder and runtime stages, missing files in any stage causes silent failures or build errors that only surface at deploy time.

**Guideline:** When adding a new dependency path to a multi-stage Dockerfile, audit every stage to determine which ones need the files and add COPY instructions to all of them.

---

## 2. Match your schema to the actual data source format

The credential fields were originally named `clientId`/`clientSecret` (camelCase) in the Zod schema, but AWS Secrets Manager returned `client_id`/`client_secret` (snake_case). This mismatch caused the Zod parse to fail silently or produce undefined values. When consuming external secrets or APIs, the schema must match the exact field names of the data source.

```typescript
// Before (wrong -- assumed camelCase)
const CodelistCredentialsSchema = z.object({
    clientId: z.string(),
    clientSecret: z.string(),
    domain: z.string(),
});

// After (correct -- matches AWS Secrets Manager format)
const CodelistCredentialsSchema = z.object({
    client_id: z.string(),
    client_secret: z.string(),
    domain: z.string(),
});
```

**Guideline:** Always verify the exact field names returned by external services and match your validation schema to them; do not assume a naming convention.

---

## 3. Parse raw secret strings before schema validation

The code originally passed the raw secret string directly to `CodelistCredentialsSchema.parse()`, but the secret value from AWS Secrets Manager is a JSON string, not an already-parsed object. Adding `JSON.parse(secretValue)` before Zod validation was the fix. Forgetting this step means Zod tries to validate a string against an object schema, which either fails or silently produces wrong results.

**Guideline:** When loading secrets from external stores like AWS Secrets Manager, remember that the value is typically a JSON string that must be parsed before schema validation.
