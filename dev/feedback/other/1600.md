# Learnings from PR #1600

This PR reset shadcn UI component styling to native defaults by removing custom font overrides, resetting shadow scales, and regenerating 18 components.

---

## 1. Keep Related UI Components Visually Consistent

When resetting one component family (e.g., radio buttons) to a new styling baseline, diverging from a sibling component family (e.g., checkboxes) creates a jarring user experience. Update related components together or plan an immediate follow-up to align them.

**Guideline:** When restyling a UI primitive, identify and update all visually related primitives in the same PR or create tracked follow-up tasks.

---

## 2. Preserve Behavioral Features When Restyling

Regenerating components from a library's defaults can silently remove behavioral customizations like sticky dialog headers. These are not purely cosmetic -- they affect usability for forms that scroll. Audit each regenerated component for behavioral regressions, not just visual ones.

**Guideline:** Before regenerating library components, document existing behavioral customizations and verify they are preserved or consciously removed.

---

## 3. Do Not Override Generated Library Code Unnecessarily

When a component library (like shadcn) generates files with `"use client"` directives, resist the urge to remove them just for cleanliness. These directives are harmless in non-RSC contexts and removing them creates maintenance overhead when regenerating components later.

**Guideline:** Accept harmless generated boilerplate (like `"use client"`) rather than customizing generated files in ways that create merge conflicts on regeneration.

---

## 4. Coordinate Cross-Team UI Changes With Clear Release Criteria

Large styling changes that affect multiple teams require explicit coordination. Identify which team will be the bottleneck for release, define clear acceptance criteria, and decide whether to collaborate on one PR or split into sequential changes.

**Guideline:** For cross-cutting UI changes, establish explicit release criteria with affected teams before merging and define who owns which follow-up work.
