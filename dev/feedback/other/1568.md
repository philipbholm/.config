# Learnings from PR #1568

Adding the ability to mark form answers as "data absent" with a free-text reason, spanning both frontend UI and backend service layer.

---

## 1. Prefer User-Centric Selectors Over Test IDs in Tests

Using `data-testid` attributes couples tests to implementation details rather than what users actually see and interact with. When an element's only visible content is an icon, use `aria-label` instead. This follows the testing-library query priority: accessible roles and labels first, then semantic queries, and test IDs only as a last resort.

**Guideline:** Follow the [testing-library query priority](https://testing-library.com/docs/queries/about/#priority) -- prefer `getByRole`, `getByLabelText`, and `getByText` over `getByTestId`.

---

## 2. Make Conditional Permission Checks Explicit, Not Ternary

When different entity types require different permissions (e.g., `patients/write` vs `dataset/write`), a ternary in the authorization block is a code smell that obscures the branching logic. Instead, explicitly check the entity type first (e.g., `if (formDataEntry.patientId)`) and handle each case in a separate, readable block. This makes the permission model discoverable and avoids bugs when new entity types are added.

**Guideline:** When authorization depends on entity type, use explicit early-return guards with comments explaining the special case, rather than inline ternaries.

---

## 3. Guard Against Overwriting Audit Fields on Updates

When a projection upserts a record, fields like `createdByUser` can silently be overwritten with the current user on every update. This destroys the audit trail. The `createdByUser` should only be set on insert, or the field should be reconsidered entirely if versioning is in play.

**Guideline:** Separate insert and update paths in projections to ensure audit fields like `createdByUser` and `createdAt` are immutable after initial creation.

---

## 4. Remove Redundant Tests That Duplicate Existing Coverage

If a test exists only to verify "a mix of values" but other tests already cover the individual cases, the extra test adds maintenance cost without meaningful coverage gain. Every test should have a clear, articulable reason for existing.

**Guideline:** Before adding a test, confirm it covers a scenario not already exercised by existing tests -- if you cannot name the unique risk it mitigates, remove it.

---

## 5. Write Test Descriptions in Plain English, Not AI-Speak

When AI tools generate test descriptions, the phrasing can be unnatural or overly technical. Test names should read like plain English sentences describing observable behavior so that any team member can understand them at a glance.

**Guideline:** Review and rewrite AI-generated test descriptions to use natural, human-readable language that describes user-facing behavior.

---

## 6. Align Terminology with Industry Standards and Existing Products

Naming features consistently with competitor products (e.g., Castor/Trials using "missing data") and industry standards (e.g., FHIR data-absent-reason codes) reduces confusion for domain experts and simplifies future interoperability. Compare values across standards before finalizing enum names.

**Guideline:** When introducing domain terminology, cross-reference existing standards (FHIR, competitor products) and prefer terminology that aligns with both user expectations and future data exchange needs.

---

## 7. Review Translations for Natural Language Quality

Machine-generated or developer-written translations can sound awkward to native speakers. Always have a native speaker (or at least a second opinion) review user-facing translations, especially for UI labels that users see frequently.

**Guideline:** Run all user-facing translations past a native speaker or language tool before merging to avoid unnatural phrasing.
