# Learnings from PR #1354

Implements a complete authentication flow (sign in, sign up, email verification, TOTP/MFA setup, user onboarding, and subscription management) spanning 101 files across frontend and backend.

---

## 1. Name API Methods to Reflect Their Behavior, Not Just Their Domain

A query named `getAuthCallbackError` was renamed to `resolveAuthErrorDetails` because the operation resolves technical error codes into user-friendly information -- it is not just a simple getter. Similarly, `getOptionalLoggedInUser` was renamed to `findCurrentLoggedInUser` because `find` operations conventionally signal that the result may be absent. Naming that reflects the operation's semantics prevents callers from making wrong assumptions.

**Guideline:** Use verb prefixes that match the operation's semantics: `get` for guaranteed returns, `find` for optional lookups, `resolve` for transformations, `check` for boolean queries.

---

## 2. Name Queries by What They Return, Not What They Check

A query called `checkAuthCompletion` was renamed to `getPendingAuthFlowSteps` because it returns a list of pending steps, not a boolean. The name `check` implies a yes/no answer, which misleads callers into expecting a simpler contract than what actually exists.

**Guideline:** If a query returns structured data (not a boolean), name it after what it returns (e.g., `getPendingSteps`) rather than what it conceptually does (e.g., `checkCompletion`).

---

## 3. Keep Mutation Scope Narrow and Accurately Named

A mutation called `upsertMyUserDetails` was flagged as too generic. The name `userDetails` invites scope creep -- developers will add unrelated fields because the name is broad. Since this mutation specifically handles profile fields needed during auth onboarding, `updateMyProfile` (or `updateMyBasicInfo`) better communicates its bounded scope.

**Guideline:** Name mutations after the specific bounded context they serve, and resist creating catch-all mutations with generic names like "details" or "data."

---

## 4. Handle External Service Failures in Multi-Step Workflows

When `startNewTrial` creates a Chargebee subscription and then writes to the database, a database failure after the Chargebee call creates an orphaned external subscription. The review flagged this as a potential source of duplicate subscriptions. The resolution was to add an idempotency check: verify the customer does not already have a subscription before creating one.

**Guideline:** When a workflow calls an external service then writes locally, add an idempotency guard (check-before-create) and document which step is not retryable and what compensating action exists.

---

## 5. Do Not Introduce Redundant Abstractions When Existing Tools Suffice

A custom `Loadable` type was introduced to model loading/success/error states, duplicating what React Query and Apollo Client already provide. The reviewer pointed this out and the author agreed to remove it. Custom state wrappers add a learning curve and diverge from the patterns the rest of the team already knows.

**Guideline:** Before creating a custom state-management abstraction, check whether your existing data-fetching library (React Query, Apollo) already provides typed loading states.

---

## 6. Log to Your Observability Platform, Not Just the Console

Multiple `console.error` calls in auth utility functions were flagged because they are invisible in production. Errors should go to the team's observability platform (e.g., Datadog) so they surface in dashboards and alerts. The author argued these errors are not important in prod, but that reasoning is self-defeating -- if the error is not important enough to log properly, it is not important enough to log at all.

**Guideline:** Replace `console.error` in production code paths with structured logging to your observability platform; remove any error logging you do not intend to act on.

---

## 7. Validate Required Data Presence Instead of Silently Falling Back

When a page like `confirm-sign-up` reads an email from URL search params and the email is missing, the code silently renders an empty state. A reviewer asked: should this not be an error, or at least redirect the user to a recoverable page? Silent fallbacks hide broken navigation flows and make debugging harder.

**Guideline:** When a page requires data from navigation state or URL params, redirect to a recovery point (e.g., the start of the flow) if the required data is absent, rather than rendering an incomplete state.

---

## 8. Use Form Keys Judiciously to Trigger Re-renders

A form component used a key derived from multiple data fields (`[userId, name, ...]`) to force remounting when query data resolved. A reviewer pointed out that only the `userId` transition (from undefined to a value) actually matters. Overly broad keys cause unnecessary re-renders and can discard user input mid-typing.

**Guideline:** When using React keys to reset a form on data load, use the narrowest possible key (typically just the entity ID) rather than a composite of all fields.

---

## 9. Mark Experimental APIs as Unstable

A subscription-related mutation (`startNewTrial`) was tied to a pricing model that may change. The reviewer suggested adding documentation that this API is experimental/unstable so other teams do not build dependencies on it. Without such markers, internal APIs get consumed widely and become impossible to change.

**Guideline:** Annotate experimental or model-dependent API endpoints with an `@experimental` or `@unstable` tag so consumers know the contract may change.

---

## 10. Use Strict Equality Operators Consistently

A comparison using `===` was flagged where `==` was being used. In TypeScript, always using `===` prevents subtle type coercion bugs, especially when comparing values that could be `null`, `undefined`, or empty strings.

**Guideline:** Always use `===` (strict equality) in TypeScript; configure your linter to enforce this automatically.
