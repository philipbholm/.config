# Learnings from PR #1724

Improved the diagnosis data model to support grouping diagnosis assignments over time (diagnosis history), with frontend visualization of the timeline.

---

## 1. Settle on One Name for Each Domain Concept Before Writing Code

When a domain concept is introduced (e.g., "DiagnosisHistory"), every layer of the stack must use that exact name. In this PR, the same concept appeared as `DiagnosisAssignment`, `DiagnosisStatusPeriod`, and `DiagnosisHistory` across the schema, events, and frontend. This forces reviewers to mentally map between synonyms, increases the chance of bugs from name mismatches, and makes grep/find-based navigation unreliable. When renaming, file names must also be updated -- not just the types and variables inside them.

**Guideline:** Before starting implementation, agree on the canonical name for each new domain concept and use it consistently across schema, events, application code, GraphQL types, and file names.

---

## 2. Standardize Directory Naming Conventions for CRUD Operations

The codebase had a mix of directory naming patterns: `assign-diagnosis/`, `delete-medication/`, `create/`, `update/`, `delete/`. This inconsistency makes it harder to navigate and for AI tools to follow conventions. The team agreed the shorter form (e.g., `medications/create/`, `medications/update/`) is preferred, matching the pattern in the medications service.

**Guideline:** Document the agreed CRUD directory structure in CLAUDE.md (e.g., `{entity}/create/`, `{entity}/update/`, `{entity}/delete/`) so both humans and AI follow the same convention.

---

## 3. Put Time-Varying Attributes on History Records, Not Parent Entities

Fields like `verificationStatus` that change over time should live on the history/timeline record rather than the parent entity. For example, a JIA diagnosis starts unconfirmed and becomes confirmed after six months of symptoms. If `verificationStatus` lives on `Diagnosis` rather than `DiagnosisHistory`, there is no way to track how it changed over time, losing clinically important temporal information.

**Guideline:** When modeling an attribute, ask whether its value can change over the entity's lifetime -- if yes, place it on the history/timeline record rather than the parent.

---

## 4. Always Handle Mutation Errors Visibly in the UI

Delete and update operations that call GraphQL mutations must surface errors to the user. In this PR, the delete dialog had no error handling, meaning a failed deletion would silently do nothing -- the dialog closes, the data looks unchanged, and the user has no idea something went wrong. At minimum, use the mutation's `onError` callback to show a toast or inline error. Also ensure unexpected errors reach alerting channels so the team can detect failures without relying on user reports.

**Guideline:** Every mutation call must have an error handling path that gives the user visible feedback and ensures unexpected errors are captured by monitoring.

---

## 5. Always Show a Confirmation Dialog Before Destructive Operations

Delete operations should require explicit user confirmation before executing. Without a confirmation step, an accidental click permanently removes data with no recourse. This applies to all delete operations, not just the ones that seem "dangerous."

**Guideline:** Wrap every delete/remove action in a confirmation dialog that clearly states what will be destroyed.

---

## 6. Avoid Nested Ternaries in JSX -- Use Explicit Conditional Blocks

Nested ternary expressions in React JSX become hard to read quickly. Instead of:

```tsx
{condition1 ? <A /> : condition2 ? <B /> : <C />}
```

Prefer explicit conditional rendering:

```tsx
{condition1 && <A />}
{!condition1 && condition2 && <B />}
{!condition1 && !condition2 && <C />}
```

This makes the rendering logic scannable and makes it obvious which elements belong together, reducing the chance of accidentally splitting related UI.

**Guideline:** Replace nested ternaries in JSX with separate conditional blocks using `&&` for clarity.

---

## 7. Use the Projection Layer Instead of Direct Prisma Calls

In an event-sourced architecture with projections, application code should go through the projection layer rather than calling Prisma directly. Direct Prisma calls bypass the established data access patterns, making it harder to reason about where data is read/written and potentially introducing inconsistencies if projection logic changes.

**Guideline:** Route all data access through the projection layer to maintain a single, consistent data access path.

---

## 8. Never Modify Already-Applied Migration Files

Editing a migration SQL file that has already been applied to any environment can put the database in a broken, inconsistent state. Prisma tracks applied migrations by filename and checksum -- changing the content of an existing migration causes a checksum mismatch, which can block future migrations or require manual intervention.

**Guideline:** Always create a new migration for schema changes; never edit a migration file that has already been applied.

---

## 9. Structure Integration Tests Around Three Assertions

Integration tests for command handlers in the event-sourced backend should verify three distinct outcomes: (1) the event is stored correctly, (2) the projection is updated correctly, and (3) the returned data matches expectations. Testing only the return value misses persistence bugs; testing only the database misses API contract regressions.

**Guideline:** Each integration test for a command handler should assert on event storage, projection state, and response data.

---

## 10. Avoid Type Casting -- Find Proper Type Inference Instead

Using `as` type casts (e.g., casting Apollo errors) suppresses the type system rather than working with it. If the types do not align naturally, that is a signal that the data flow needs adjustment. Type casts hide potential runtime errors that TypeScript is designed to catch.

**Guideline:** When tempted to use `as` for type casting, investigate whether the API provides a type-safe alternative (e.g., type guards, `instanceof` checks, or generic type parameters).
