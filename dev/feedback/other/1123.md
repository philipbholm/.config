# Learnings from PR #1123

Added repeatable visits to registries, allowing visits to occur multiple times for patient pathways where visit frequency is unknown upfront, including database migration, backend logic, and frontend display.

---

## 1. Prefer Type-Safe Error Checks Over Loosely-Typed Error Properties in Mutation Results

When checking whether a GraphQL mutation succeeded, checking the specific shape of `result.data` (e.g., `!result.data?.createX.entity.id`) provides type safety and is explicit about what constitutes success. Using `result.error` can miss cases where the mutation "succeeds" at the transport level but returns unexpected null data. However, this should be balanced against readability -- if the Apollo client provides a well-typed error object, it may be more readable to check that first. Either way, be consistent.

**Guideline:** When verifying GraphQL mutation success, prefer checking the specific data shape that constitutes success (`result.data?.entity.id`) over relying solely on `result.error`, especially when the error typing is less precise than the data typing.

---

## 2. Drop Unique Constraints When Introducing Repeatable Entities

When a previously one-to-one relationship becomes one-to-many (e.g., a patient can now have multiple visit entries for the same visit), remember to drop the corresponding unique constraint in the database. The PR correctly dropped the `patient_visit_entries_patient_id_visit_id_key` unique index and changed `findUnique` to `findFirst` in the projection layer. Failing to update both the schema and the query layer will cause runtime errors.

```sql
-- Migration to support repeatable visits
DROP INDEX "public"."patient_visit_entries_patient_id_visit_id_key";
```

```typescript
// Before: findUnique with composite key
await client.patientVisitEntry.findUnique({
  where: { patientId_visitId: { patientId, visitId } },
});

// After: findFirst since uniqueness no longer guaranteed
await client.patientVisitEntry.findFirst({
  where: { patientId, visitId },
});
```

**Guideline:** When changing a one-to-one relationship to one-to-many, update the database constraint, the Prisma schema, and all query methods (findUnique to findFirst) in a single migration.

---

## 3. Guard Business Rules in the Use Case Layer, Not in Database Constraints Alone

When the uniqueness rule becomes conditional (e.g., non-repeatable visits should still have at most one entry, but repeatable visits can have many), move the enforcement from a database unique constraint to explicit business logic in the use case. This makes the rule visible, testable, and allows for conditional behavior.

```typescript
if (!visit.repeatable) {
  const exists = await projection.findPatientVisitEntry({
    patientId, visitId,
  });
  if (exists) {
    throw new AlreadyExistsError("Patient visit entry already exists");
  }
}
```

**Guideline:** When a uniqueness rule becomes conditional on domain logic, enforce it in the use case layer with an explicit check rather than relying solely on database constraints.
