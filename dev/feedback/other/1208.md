# Learnings from PR #1208

This PR implemented a dataset view for single-site registries, adding a new `DatasetProjection` to query patient visit entries with their form answers, updating the GraphQL schema to return richer row data (patient identifier, visit name, per-variable answers), and improving the frontend table with loading/error/empty states and i18n.

---

## 1. Invest in Declarative Test Setup for Complex Integration Tests

When integration tests require lengthy imperative setup (creating a registry, patients, diagnoses, treatments, visits, forms, form elements, publishing, assigning treatments, creating visit entries, saving answers...), the test becomes very hard to follow. A more declarative approach -- such as a builder or fixture utility -- makes the "arrange" section readable and reduces boilerplate across tests.

```typescript
// The test setup was flagged as hard to follow:
// ~80 lines of sequential setup calls before any assertion
const registry = await app.createSingleSiteRegistry({ ... });
const patient = await app.createPatient({ ... });
const diagnosis = await app.addDiagnosisToRegistry({ ... });
const treatment = await app.createTreatment({ ... });
const visit = await app.createVisit({ ... });
// ... and so on
```

The feedback was: "We should start considering a more declarative test setup. It's really hard to follow the test setup." The response acknowledged this is a real problem, especially for complex scenarios.

**Guideline:** When integration test setup exceeds ~10 sequential calls, extract a builder or fixture utility that expresses the scenario declaratively, so the test reads like a specification rather than a script.

---

## 2. Create Dedicated Projections for Distinct Read Concerns

The original implementation fetched dataset answers through the `VariableProjection`, but dataset retrieval is a fundamentally different read concern -- it needs to join across patients, visits, form entries, and answers. Routing this through an unrelated projection couples distinct responsibilities and makes the query harder to optimize.

The feedback asked: "Does it make sense that getting the dataset answers comes from the variable projection?" The resolution was to create a dedicated `DatasetProjection`.

**Guideline:** When a query spans multiple aggregate boundaries or has a different shape than existing projections serve, create a dedicated projection rather than overloading an existing one.

## 3. Resolve Category IDs to Human-Readable Values at the Data Layer

When displaying dataset answers that reference category IDs, resolve them to their human-readable labels before returning the data to the frontend. This avoids leaking internal IDs into the UI and keeps the presentation layer simple.

```typescript
const normalizeAnswers = (args: {
    answers: string[];
    elementType: string | null;
    categories: CategoryForAnswer[];
}): string[] => {
    if (!CATEGORY_ELEMENT_TYPES.has(args.elementType ?? "")) {
        return args.answers;
    }
    const categoryValuesById = args.categories.reduce<Record<string, string>>(
        (acc, cat) => { acc[cat.id] = cat.value; return acc; }, {}
    );
    return args.answers.map((answer) => categoryValuesById[answer] ?? answer);
};
```

**Guideline:** Resolve internal reference IDs (like category IDs) to display values in the projection or service layer, so the frontend receives data ready for presentation without needing lookup tables.
