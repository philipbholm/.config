# Learnings from PR #1207

This PR fixed a bug where the same form entry was displayed for all visits by switching from visit-ID-based routing to visit-entry-ID-based routing, ensuring each patient visit entry maps to the correct form data.

## 1. Prefer Aliased Path Imports Over Deep Relative Imports

When importing modules from distant parts of the directory tree, use the project's path alias (e.g., `~/features/route-map/registries-route-map`) instead of long relative paths. Deep relative imports like `../../../../../../../features/...` are fragile, hard to read, and break if directory structure changes.

```typescript
// Avoid:
import { useNavigate } from "../../../../../../../../../features/communication-bridge/useRouter";
import { ROUTE_MAP } from "../../../../../../../../../features/route-map/routeMap";

// Prefer:
import { REGISTRIES_ROUTE_MAP } from "~/features/route-map/registries-route-map";
```

The review noted to "go the other way for this import" -- meaning use the aliased path. The response confirmed this should be the general preference.

**Guideline:** Always use the project's path alias (e.g., `~/ `) for imports that would otherwise require more than two levels of `../`; this keeps imports readable and resilient to structural changes.
