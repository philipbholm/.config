# Learnings from PR #1701

This PR separated patient forms from custom forms by adding a `formType` discriminator (`CUSTOM` | `PATIENT`) to `RegistryForm`, filtering list endpoints to only return custom forms, and creating the patient form atomically during registry creation instead of via a separate event.

---

## 1. Use Generated Prisma Types Instead of String Assertions

When a database column has an enum type defined in Prisma, import and use the generated Prisma enum type directly rather than typing the field as `string` and casting it to a domain union. String-typed parameters with type assertions weaken compile-time safety and can mask unexpected values that would otherwise be caught by the type checker.

```typescript
// Bad: string type with assertion
const dbFormToForm = (form: { formType: string }): RegistryForm => ({
  formType: form.formType as RegistryFormType,
});

// Good: use the Prisma-generated type directly
import { RegistryFormType } from "../../generated/prisma/client";
const dbFormToForm = (form: { formType: RegistryFormType }): RegistryForm => ({
  formType: form.formType,
});
```

**Guideline:** When mapping database models to domain models, use Prisma-generated types for enum fields to preserve type safety end-to-end and avoid manual assertions.

---

## 2. Understand Transactional Boundaries Before Requesting Transaction Wrappers

In event-sourced systems with listener-based projections, operations within a single event listener are often already executed inside a database transaction. Requesting an explicit `$transaction` wrapper in code that already runs transactionally adds unnecessary complexity and suggests the reviewer does not understand the infrastructure guarantees. Before asking "can we wrap this in a transaction?", verify whether the execution context (listener, middleware, handler) already provides transactional semantics.

**Guideline:** Before requesting transactional wrappers in review, check whether the surrounding infrastructure (event listeners, middleware) already guarantees atomicity.

---

## 3. Create Dependent Entities Atomically in the Same Event

When an entity always needs a companion entity (e.g., a registry always needs a patient data form), create both in a single event handler rather than emitting separate events. Separate events introduce a window where the system is in an inconsistent state -- the registry exists without its patient form, or the form exists with the wrong type. This PR eliminated the `PATIENT_DATA_FORM_SET` event entirely by creating the patient form directly in the `REGISTRY_CREATED` handler, removing a race condition where patient forms could briefly appear in custom form listings.

**Guideline:** When two entities must always be created together, handle both in a single event to avoid transient inconsistency windows.

---

## 4. Rename APIs to Reflect Filtered Semantics

When a query's behavior changes from returning all items to returning a filtered subset, rename the query to make the filtering explicit. This PR renamed `getRegistryForms` to `getCustomRegistryForms` across the full stack (GraphQL schema, resolvers, frontend queries, hooks, refetch queries). This prevents callers from assuming they receive all forms when they actually receive only custom ones, which would lead to bugs where patient forms silently disappear from UI components that depend on the old semantics.

**Guideline:** When a query gains implicit filtering, rename it to advertise the filter so callers understand exactly what subset they receive.

---

## 5. Guard Against Deletion of Special-Purpose Entities

When an entity type has a special-purpose variant (like a patient form vs. a custom form), add explicit guards to prevent accidental deletion of the special variant. This PR added a `FailedPreconditionError` check in the delete handler to block deletion of `PATIENT` forms. Without this guard, a user could delete the patient data form through the generic form deletion endpoint, leaving the registry in a broken state with a dangling `patientDataFormId` reference.

**Guideline:** Add precondition checks in mutation handlers to prevent deletion or modification of system-managed entities that other entities depend on.
