# Learnings from PR #1427

Adding an initial analysis dashboard feature for registries, including backend CRUD operations, analysis execution engine, GraphQL schema, and frontend dashboard UI.

---

## 1. Use Consistent Naming Across the Full Stack

When a concept is called "dashboard" in the UI and product language, the code (routes, GraphQL queries, file names, domain models) should also use "dashboard" -- not "analysis" or legacy names like "treatments/visits." Inconsistent naming creates confusion about which abstraction a piece of code belongs to and makes searching the codebase unreliable.

**Guideline:** Before merging, grep for naming inconsistencies between frontend routes, GraphQL operations, and backend domain models to ensure they all use the same vocabulary.

---

## 2. Avoid Barrel Files (index.ts Re-exports) in Application Code

Barrel files (`index.ts` that re-export from multiple modules) cause test runners and bundlers to load all exposed modules even when only one is needed, slowing down tests and increasing memory usage. They also make it unclear where a symbol is actually defined. This pattern is acceptable for published packages but harmful in application code.

**Guideline:** Import directly from the specific module file rather than creating or using barrel files in application service code; document this preference in your AI agent instructions so generated code follows the pattern.

---

## 3. One GraphQL Query Per File

Putting multiple GraphQL operations in a single `.graphql` file creates ambiguity about where to add new queries and makes it harder to find query definitions via file search. Splitting to one query per file provides clear organization, trivial discoverability (file name matches query name), and potentially better codegen output.

**Guideline:** Store each GraphQL query/mutation in its own file, named after the operation, colocated with the component that uses it.

---

## 4. Follow the Established Event Sourcing Pattern for Write Operations

When the codebase uses event sourcing for state mutations, new features must follow the same pattern. Directly writing to projections without emitting events breaks the audit trail, makes replaying events incomplete, and creates two divergent persistence paths in the same service.

**Guideline:** Any new create/update/delete use case must emit domain events and project from those events, consistent with the existing architecture.

---

## 5. Add Unit Tests for Complex Pure Logic, Even When Integration Tests Exist

Functions over ~100 lines that perform calculations (like statistical analysis, aggregation, or data transformation) need unit tests in addition to integration tests. Integration tests are slow, hard to read at scale, and tend to obscure which specific calculation path is being validated. Unit tests on pure functions are fast, cheap to write (especially with AI), and give targeted confidence.

**Guideline:** Extract complex calculation logic into pure functions and unit test them directly; reserve integration tests for verifying the wiring and database interactions.

---

## 6. Handle and Display Errors Instead of Silently Swallowing Them

When a frontend operation fails (like executing an analysis), the user must see a meaningful error, and the error should be logged to monitoring. Silently catching errors or showing nothing leaves users confused and makes production issues invisible to the team.

**Guideline:** Every catch block in UI code should both display a user-facing error message and log the error to the monitoring system.

---

## 7. Keep Translation Logic in the Frontend

Backend services should return raw values and identifiers; the frontend is responsible for translating and formatting them for display. Mixing translation into backend responses couples the API to a specific locale and makes it harder to support multiple languages or change display strings.

**Guideline:** Return machine-readable keys/identifiers from the API and perform all i18n/l10n formatting on the client side.

---

## 8. Avoid Recursive Types When Only One Level Is Needed

Defining a recursive type (a type that references itself) when the actual data only goes one level deep introduces unnecessary complexity and can cause infinite rendering or parsing issues. Be explicit about the depth the code actually supports.

**Guideline:** If your data structure only nests one level, model it as an explicit parent-child type rather than a recursive self-reference.

---

## 9. Remove Duplicate and Leftover Files Before Merging

Renamed or refactored files should not leave behind the old version. Duplicate files (like having both `AnalysisResultTable.tsx` and `analysis-results-table.tsx`) bloat the codebase, confuse imports, and risk the wrong version being used.

**Guideline:** After renaming or moving a file, verify the old file is deleted and no imports reference it.

---

## 10. Squash Migrations Before Merging Feature Branches

Multiple migration files created during iterative development on a feature branch should be consolidated into a single migration before merging to the main branch. Multiple intermediate migrations add noise to the migration history and can cause confusion about the intended final schema state.

**Guideline:** Squash all feature-branch database migrations into one clean migration that represents the final schema change.
