# Learnings from PR #1010

This PR initialized registry variables as a domain concept, linking them to form elements and introducing variable projections, events, and GraphQL types.

---

## 1. Make Opaque Code Self-Documenting

A reviewer asked "what is this for?" about a function that returns a map of variables keyed by variable ID. The author explained it avoids repeated `.find()` calls when looking up variables. Functions that transform data structures (e.g., converting an array to a lookup map) should have clear names or brief doc comments so the optimization intent is obvious without needing to ask.

**Guideline:** When a function exists primarily for a performance or ergonomic reason (like converting a list to a map for O(1) lookups), name it to convey the "why" -- e.g., `indexVariablesById` rather than `getVariables`.

---

## 2. Derive Properties From Their Canonical Source

A reviewer questioned why the element type was being derived from the variable rather than from the element itself. The PR mapped form element types through the variable's data type, but this only worked for variable-backed elements and would break for non-variable elements (static text, separators, images). When a property conceptually belongs to one entity, it should live on that entity, even if another entity happens to have a correlated value.

```typescript
// Problematic: element type derived from variable
export const formElementToGqlElement = (args: {
    element: RegistryFormElement;
    variable: Variable;
}): GraphqlRegistryFormElement => {
    return {
        elementType: "TEXT", // hardcoded because only variables were implemented
        ...
    };
};
```

**Guideline:** Place a property on the entity that owns it semantically; avoid deriving it from a sibling entity, even if the values happen to correlate today, because future entity types may not have that correlation.

---

## 3. Design Initial Implementations With Extension Points in Mind

The discussion revealed that the initial implementation only handled variable-backed form elements, and the author acknowledged "we will need to find another solution for this to support non-variable related elements." When building the first variant of a polymorphic system (like form elements that can be either variable-backed or static), design the abstraction to accommodate the known future variants, even if you only implement one now. This prevents a rewrite when the next variant arrives.

**Guideline:** When you know a union type will have multiple variants, structure the shared interface and mapping logic to handle the full set from the start, even if only one variant is implemented.
