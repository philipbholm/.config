# Learnings from PR #1471

Added the ability to delete patients from the registry, including a confirmation dialog, backend use case with authorization, codelist cleanup, and cascade deletion of related data.

---

## 1. Test Integration Paths with External Services

The integration test initially did not cover the scenario where a patient has an associated codelist record. When your domain entity has relationships to external services (like a codelist service), write explicit test cases that create the full relationship chain and verify that cleanup of the external resource happens correctly during deletion.

**Guideline:** When testing a delete operation, add a dedicated test case for each external service dependency that must be cleaned up alongside the primary entity.

---

## 2. Test Failure Modes of External Service Calls

When an operation depends on an external service call (like deleting codelist records before deleting the patient), you must verify what happens when that external call fails. In this PR, a test was added to confirm that when codelist deletion fails, the patient is NOT deleted -- preserving transactional consistency.

**Guideline:** For every external service call in a critical path, write a test that simulates service failure and verifies the operation rolls back or fails gracefully without leaving data in an inconsistent state.

---

## 3. Add Error Handling to Frontend Mutation Handlers

The initial frontend delete handler had no error handling -- if deletion failed due to network issues or authorization errors, the dialog would close and the user would be left uncertain. Error states should always be surfaced in the UI so users know whether a destructive action succeeded or failed.

**Guideline:** Every frontend mutation handler must include error handling that displays feedback to the user on failure, especially for destructive operations.

---

## 4. Verify Cascade Deletion Covers All Related Entities

The reviewer asked "Should we check diagnoses also?" -- the initial test only verified cascade deletion of medications but not diagnosis assignments. When building delete functionality, enumerate every entity type that has a foreign key relationship to the entity being deleted and verify each is cleaned up.

**Guideline:** When implementing entity deletion, create a checklist of all related entity types and write an explicit test assertion for each one to confirm cascade deletion works.

---

## 5. Clarify Data Retention Semantics in Event-Sourced Systems

A reviewer raised concern about whether event logs still contained PII after patient deletion. The response clarified that only identifiers (not PII) are stored in the event log. When building deletion features in event-sourced systems, document clearly what data persists in the event store versus what is removed from projections, because "permanently delete all data" may mislead if events are retained.

**Guideline:** When implementing deletion in an event-sourced system, explicitly document what data remains in the event store and ensure the deletion promise to users matches the actual data lifecycle.

---

## 6. Export Shared Test Stubs for Reuse

The PR introduced `stubbedCodelistService` as an exported constant from the test utilities, allowing all test files to spread it and override only the methods they need. This replaced duplicated inline stub definitions across multiple test files:

```typescript
const mockCodelistService: CodelistService = {
    ...stubbedCodelistService,
    deleteRecords: mockDeleteRecords,
};
```

**Guideline:** When multiple test files need to mock the same service interface, create a single exported stub with default implementations and let individual tests override only the methods they care about.
