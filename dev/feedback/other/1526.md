# Learnings from PR #1526

Architecture document for a new cross-service approval system, defining domain model, gRPC API, SNS/SQS event flow, database schema, and auto-approval workspace settings.

---

## 1. Choose Service Names That Reflect the Actual Domain, Not Generic Concepts

The system was initially called "tasking system" but reviewers pointed out that "task" is too generic and could be confused with task orchestration, cron jobs, or background workers. Since the system exclusively handles approval flows, naming it "Approval System" (with an `ApprovalService` gRPC service) communicates intent immediately. Generic names create confusion during onboarding, in logs, and in monitoring dashboards.

**Guideline:** Name services and systems after the specific domain they serve (e.g., `ApprovalService`) rather than generic abstractions (e.g., `TaskService`) -- the name should tell a new team member exactly what the service does.

---

## 2. Include Display Metadata When Passing Data Across Service Boundaries

When a service creates an approval request with key-value metadata, the consuming frontend needs to know how to render each field (its label, its data type). Without display hints like `meta_type` and `meta_label`, the frontend is forced to hard-code rendering logic per request type, which defeats the purpose of a generic metadata model. Plan for the rendering layer when designing the data model.

**Guideline:** When designing cross-service metadata schemas, include display hints (label, data type) alongside the raw value so consuming UIs can render fields generically without hard-coded mappings.

---

## 3. Separate Distinct Flows Rather Than Overloading a Single Request

The `RequestAddCollaborator` endpoint originally included fields for inviting new users (name, email) alongside fields for adding existing collaborators. Reviewers argued these are fundamentally different flows: inviting a non-existent user should be a separate interaction from adding an existing workspace member to a project. Combining them creates confusing optional-field combinations and makes the API harder to reason about.

**Guideline:** When an API endpoint handles two distinct user journeys (e.g., "invite new user" vs. "add existing user"), split them into separate endpoints or sequential flows rather than overloading one request with conditional fields.

---

## 4. Document Failure Modes and Recovery Paths for Async Architectures

The architecture described the happy path (request -> approve -> SNS -> SQS -> complete) but reviewers flagged critical gaps: what happens when SNS publish fails after the DB marks a request as APPROVED? What if the consuming service crashes mid-processing? Each failure scenario needs a documented recovery path -- retry policies, dead letter queues, or reconciliation jobs.

**Guideline:** For every async event-driven flow, document at least three failure scenarios (publish failure, consumer crash, partial processing) and their recovery mechanisms before implementation begins.

---

## 5. Add Idempotency Keys to Distributed Operations

Network retries are inevitable in service-to-service communication. Without idempotency keys, a timeout during `RequestAddCollaborator` followed by a retry can create duplicate approval requests. SNS can also deliver the same event multiple times. Including a `clientRequestId` or `idempotencyKey` in both gRPC inputs and event payloads allows consumers to deduplicate safely.

**Guideline:** Add an idempotency key field to every gRPC request and async event payload that triggers a state mutation, so consumers can safely handle retries and duplicate deliveries.

---

## 6. Build Reconciliation Mechanisms for Eventually Consistent Systems

Given the eventual consistency between the approval service and downstream consumers, there will inevitably be cases where events are lost or processing fails silently. A reconciliation job that periodically checks for APPROVED requests without corresponding downstream state (e.g., a study that should exist but does not) provides a safety net that catches the edge cases automated retries miss.

**Guideline:** For eventually consistent architectures, implement a periodic reconciliation job that detects and repairs state divergence between the source-of-truth service and downstream consumers.

---

## 7. Document SNS Subscription Filtering Strategy

When multiple services subscribe to the same SNS topic, each service should only receive events relevant to its domain (e.g., registries service only gets `CREATE_REGISTRY` approvals). Without documenting the subscription filter policy, implementers may accidentally subscribe to all events or miss setting up filters entirely, causing unnecessary processing or missed events.

**Guideline:** When designing pub/sub architectures, explicitly document which event types each subscriber filters for, and include the filter policy in the architecture document alongside the topic definition.
