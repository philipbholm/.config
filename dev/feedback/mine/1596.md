# Learnings from PR #1596

Update registry cards with patient counts, collaborator counts, and collaborator avatars in both the backend GraphQL API and frontend display.

---

## 1. Remove Unimplemented Code Instead of Leaving Stubs

Adding fields or resolvers that return hardcoded placeholder values (like `status: "DRAFT"` or zero counts) creates confusion for other developers and consumers of the API. It becomes unclear whether the code is broken, intentionally incomplete, or just wrong. Placeholder stubs get forgotten and ship as silent bugs.

**Guideline:** If a feature isn't ready, don't add its interface — leave it out entirely until the implementation exists.

---

## 2. Name Backend Types for the Domain, Not the UI

Naming a GraphQL type `RegistryCard` leaks the frontend's presentation concept into the backend. The backend shouldn't know how data is rendered. If the frontend redesigns from cards to a table view, the backend type name becomes misleading. Domain-neutral names like `RegistryWithMetadata` or `RegistrySummary` remain accurate regardless of how the frontend consumes them.

**Guideline:** Name backend types after what the data represents in the domain, not how the UI displays it.

---

## 3. Keep Frontend Concerns Out of Backend Logic

Sorting and formatting operations that exist purely for display purposes (like sorting collaborators by name for avatar display) belong in the frontend. Pushing presentation logic into the backend couples the API to a specific UI design and makes the backend harder to maintain when the UI changes.

**Guideline:** If the only consumer of a piece of logic is the UI layer's rendering, implement it in the frontend.

---

## 4. Follow Existing Patterns When Adding Parallel Features

When the codebase already has a working pattern for a similar feature (e.g., `StudyUser` for studies), replicate that pattern for the analogous feature in a new domain (e.g., user/collaborator handling in registries). Inventing a different approach creates inconsistency, makes the codebase harder to navigate, and misses battle-tested design decisions already made in the existing implementation.

**Guideline:** Before designing a new feature, check if a sibling domain already solved the same problem and follow its pattern.

---

## 5. Keep Domain Types and GraphQL Types Aligned

Having separate types for the same entity across layers — where the GraphQL type requires fields (like `patientCount`, `collaboratorCount`) that don't exist on the domain type — creates a mapping gap. This forces mappers to fabricate data or introduces bugs when fields are missing. The GraphQL schema should reflect what the domain actually provides.

**Guideline:** GraphQL types should map 1:1 with domain types — don't require fields in the schema that the domain doesn't naturally produce.

---

## 6. Prefer Composition Over Expanding Repository Methods

When you need aggregated data (like registries with patient counts), resist adding a single monolithic method to the repository/projection that fetches everything. Instead, create focused methods (`getPatientCounts`, `getCollaboratorCounts`) and compose them in the use case. Monolithic repository methods grow unboundedly as new data requirements appear.

**Guideline:** Add focused, single-purpose queries to projections and compose them in use cases rather than building one method that returns everything.

---

## 7. Keep Shared Test Setup Minimal and Generic

Adding test-specific configuration (like custom mock implementations for a single test scenario) into a shared test setup file couples all tests to that specific setup and obscures what each individual test actually needs. Tests should clearly declare their own dependencies and behavior rather than inheriting complex shared state.

**Guideline:** Move custom test configuration into the specific test that needs it — shared test helpers should only provide generic, broadly-applicable defaults.

---

## 8. Be Selective About What Tests Assert

When writing integration tests, only assert on the properties relevant to the behavior being tested. Asserting on every field — especially when other fields are not similarly asserted — creates noise and makes it unclear what the test is actually verifying. Each assertion should have a reason to exist.

**Guideline:** Assert on what the test is testing; if a property isn't central to the test's purpose, don't assert on it just because it exists.

---

## 9. Add E2E and Integration Tests for New UI Features

When adding new visual components or updating existing ones (like registry cards), include both end-to-end tests for the main success scenario and integration/rendering tests for the component. Visual changes without test coverage are fragile and regressions go undetected.

**Guideline:** Pair UI feature work with e2e main-path tests and component-level integration tests for rendering.
