# Learnings from PR #1568

Add the ability to mark form answers as absent/missing, with a free-text reason and predefined reason codes.

---

## 1. Prefer User-Centric Selectors Over Test IDs in Tests

Using `data-testid` attributes couples tests to implementation details rather than what users actually see and interact with. When a component only shows an icon, use an `aria-label` selector instead. Tests should follow the Testing Library query priority: accessible roles and labels first, then semantic HTML, and `data-testid` only as a last resort.

**Guideline:** Follow the [Testing Library query priority](https://testing-library.com/docs/queries/about/#priority) — prefer `getByRole`, `getByLabelText`, and `getByText` over `getByTestId`.

---

## 2. Make Authorization Logic Explicit, Not Ternary

When a single operation requires different permissions depending on context (e.g., patient-based vs event-based forms), a ternary in the authorization check is a code smell. It obscures the branching logic and makes it hard to understand why two different permissions exist. Instead, make the conditional explicit — check for the distinguishing property first, handle the special case with a comment explaining why, then fall through to the normal flow.

```typescript
// Explicit branch is clearer than a ternary permission check
authorize: async ({ input, context }) => {
    const formDataEntry =
        await dependencies.formDataEntryProjection.getFormDataEntry({
            formDataEntryId: input.formDataEntryId,
        });

    // Handle patient forms (patients/write permission)
    if (formDataEntry.patientId) {
        return ...
    }

    // Normal flow: dataset/write permission
    ...
}
```

**Guideline:** When authorization differs by context, use explicit conditional branches with comments explaining each case rather than inline ternaries.

---

## 3. Write Test Descriptions in Plain English

AI-generated test descriptions often use awkward, overly technical phrasing that reads like "claude-speak." Test descriptions should be written in natural, plain English that any team member can quickly understand. A description like "when the form data entry is associated with a patient" should simply be "when answering a patient form."

**Guideline:** Write test descriptions as short, natural sentences that a human would use to describe the scenario.

---

## 4. Every Test Must Justify Its Existence

If a test doesn't clearly test a distinct scenario that isn't already covered by other tests, it adds maintenance burden without value. When asked "what's the point of this test?" you should have a clear, specific answer. A test that just exercises "a bunch of mixed values" without asserting something unique is redundant noise.

**Guideline:** Before adding a test, verify it covers a scenario not already tested — if you can't articulate what unique case it validates, remove it.

---

## 5. Align Terminology with Industry Standards and Existing Products

When naming features and enum values, check what terminology competitors and industry standards (FHIR, Castor, etc.) use. Starting with "data absent" (FHIR terminology) then discovering the team's trials product uses "missing" creates rework. Similarly, the set of reason codes should align with established standards to maintain compatibility and avoid future migrations.

**Guideline:** Research industry standards and existing product terminology before naming domain concepts — alignment saves rework and ensures interoperability.

---

## 6. Verify Upsert Operations Don't Overwrite Immutable Fields

When a projection handler uses `upsert` to save data, check whether fields that should be write-once (like `createdByUser`) get overwritten on subsequent updates. An upsert that sets `createdByUser` to the current user on every call will silently replace the original creator with whoever triggers the next update. Either exclude immutable fields from the update clause or use `connectOrCreate` patterns.

**Guideline:** Audit upsert operations to ensure write-once fields (creator, creation timestamp) are only set in the create clause, not the update clause.

---

## 7. Get UX Copy Reviewed for Translations Early

User-facing strings, especially translations, benefit from early review. A Norwegian translation that reads like a "bad machine translation" creates a poor user experience and requires another round of changes late in the review cycle. Have native speakers review translated strings before marking the PR as ready.

**Guideline:** Include translated UI strings in the initial review request so copy issues surface early, not after approval.

---

## 8. Add Frontend Integration Tests for New Features

Backend integration tests alone don't validate the full user journey. When adding a new interactive feature (like marking data as absent), the frontend needs its own integration tests that verify the UI flow: clicking the button, selecting reasons, seeing the updated state. A reviewer asking "where are the frontend integration tests?" means the feature's test coverage has a gap.

**Guideline:** New user-facing features require frontend integration tests that exercise the complete interaction flow, not just backend service tests.
