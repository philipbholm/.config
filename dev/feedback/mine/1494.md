# Learnings from PR #1494
This PR removed multi-site registry functionality, consolidating all registry operations into a single registry type across 601 files with significant deletion of code.

---

## 1. Update All Hardcoded Route Strings When Restructuring URL Patterns

When removing or restructuring route segments (e.g., removing `single-site-registry` from paths), hardcoded navigation strings in components can silently break. A click handler was still navigating to `/${lang}/registries/single-site-registry/...` after the route had been changed to `/${lang}/registries/${registryId}/...`, resulting in a 404. These broken navigations do not cause compile-time errors and are easy to miss in large refactors.

**Guideline:** After changing route structures, search the entire codebase for hardcoded path segments being removed and replace them with the centralized route map.

---

## 2. Use Centralized Route Maps Instead of String Interpolation for Navigation

The PR had a centralized `REGISTRIES_ROUTE_MAP` utility for building paths, but at least one component bypassed it by constructing a route string manually. This divergence meant the manual path was not updated when routes changed. Centralized route maps act as a single source of truth and ensure all navigation paths update together.

**Guideline:** Always use the project's route map utility for navigation paths; never construct route strings by hand in components.

---

## 3. Update Error Messages and Developer-Facing Strings After Refactors

A `useFormId` hook still referenced `single-site-registry` in its error message after the route structure changed. While this does not break runtime behavior, stale error messages mislead developers during debugging by pointing them to paths that no longer exist. Developer-facing strings are easy to overlook because they do not affect user-visible behavior or tests.

**Guideline:** Treat developer-facing error messages as code -- search for and update all references to removed concepts, not just functional code.

---

## 4. Verify Hook Import Sources When Multiple Implementations Exist

When a codebase has multiple hooks with the same name (e.g., different implementations of a data-fetching hook in different modules), a large refactor that deletes or moves files can leave imports pointing at the wrong implementation. Reviewers flagged uncertainty about which hook implementation was being used, indicating the import was ambiguous.

**Guideline:** When multiple hooks share a name, use explicit re-exports or barrel files to make the canonical import path unambiguous, and verify imports after large-scale file moves or deletions.

---

## 5. Scope Large Deletion PRs to Reduce Review Burden

This PR touched 601 files with 13,000 lines deleted and 2,700 added. At that scale, reviewers cannot realistically inspect every file, and automated tools only covered 108 of 601 files. Critical issues like broken navigation paths can easily slip through. Splitting large removal PRs into phases (e.g., first redirect old routes, then delete dead code, then clean up references) makes each phase independently reviewable and testable.

**Guideline:** When a refactor exceeds roughly 100 files, consider splitting it into sequential PRs that each have a clear, independently verifiable scope.

---

## 6. Search for Stale References After Removing a Domain Concept

Removing a domain concept like "multi-site" touches more than just the feature's own files. References leak into error messages, comments, test fixtures, route constants, type discriminators, and navigation handlers across the codebase. A global search for the removed concept's keywords (e.g., `multi-site`, `single-site-registry`, `connection-invitation`) should be performed as a final check before marking the PR ready.

**Guideline:** Before submitting a PR that removes a feature or concept, run a codebase-wide search for all related terminology and verify every remaining reference is intentional.
