# Learnings from PR #1469
This PR added Claude Code MCP server and plugin configurations to the monorepo, including third-party integrations for documentation lookup, browser automation, and GitHub interaction.

---

## 1. Verify Data Retention Policies Before Adding Third-Party AI Plugins
When integrating third-party plugins or MCP servers into an AI coding assistant, reviewers will rightfully ask where source code is being sent and how it is retained. If a plugin forwards repository context to an external service, the data retention and privacy guarantees of that service must be understood and documented. Failing to do so creates ambiguity about whether proprietary code is being stored or processed by unknown third parties.

**Guideline:** Before adding any third-party AI plugin that processes source code, confirm its hosting provider and verify that data retention is covered by your existing vendor agreements.

---

## 2. Document the Trust Boundary for Each External Integration
A configuration file listing plugin IDs does not make it obvious which external services will receive source code at runtime. Reviewers should not have to guess whether a plugin is first-party (hosted by the AI vendor under existing agreements) or third-party (an independent service with its own terms). Making this distinction explicit in the PR description or in inline comments reduces review friction and prevents security concerns from blocking merges.

**Guideline:** When adding external tool integrations, explicitly state in the PR description which services handle source code and under what agreement.

---

## 3. Treat AI Tool Configuration as Security-Sensitive Infrastructure
Configuration files for AI coding assistants (like `.claude/settings.json` or `.mcp.json`) control what tools can read, modify, and transmit repository contents. These files deserve the same level of scrutiny as CI/CD pipeline configs or deployment secrets. The PR correctly included deny rules for destructive commands, but the review surfaced that the data-flow implications of plugins also need vetting.

**Guideline:** Review AI tool configuration changes with the same rigor as CI/CD or infrastructure changes, paying attention to both command permissions and data flow.

---

## 4. Separate Shared Team Tooling from Personal Developer Preferences
This PR was closed without merging, suggesting the team was not ready to adopt the configuration repo-wide. When introducing developer tooling that changes the workflow for all contributors, consider whether the configuration should live in the shared repository or in personal/local configuration. Pushing shared config prematurely can create friction, especially when not all team members use the same tools.

**Guideline:** Introduce shared developer tool configurations only after team consensus; keep personal tool preferences out of the repository until adoption is agreed upon.
