# Learnings from PR #1594

Updated sidebar design: removed name header, moved settings to bottom, added home tab, and persisted sidebar state across page refreshes.

---

## 1. Use Browser Storage for Persisting UI State Across Refreshes

The initial implementation used React state (`useState`) to track whether the sidebar was open or collapsed. This meant the sidebar reset to its default state on every page refresh, losing the user's preference. The fix was to use `localStorage` (via `useLocalStorage` from `@mantine/hooks`) instead of plain React state. When UI preferences need to survive page refreshes, cookies or in-memory state are insufficient or overcomplicated — `localStorage` or `sessionStorage` is the right tool.

**Guideline:** When a UI preference (sidebar state, theme, panel size) must persist across refreshes, use `localStorage` or `sessionStorage` from the start rather than plain React state.

---

## 2. Prefer Existing Utility Hooks Over Rolling Your Own

When switching from cookies to `localStorage`, the reviewer pointed to `useLocalStorage` from `@mantine/hooks` — a hook already available in the project's dependencies. Rolling a custom implementation (manual `localStorage.getItem`/`setItem` with `useEffect` syncing) introduces unnecessary boilerplate and edge-case bugs (SSR hydration, cross-tab sync). If the project already depends on a utility library, use its hooks.

**Guideline:** Before writing custom storage hooks, check if the project's existing dependencies (e.g., `@mantine/hooks`) already provide a battle-tested version.

---

## 3. Name Event Handlers by What They Do, Not Generically

A callback was named `handleClick` — a name that conveys nothing about its purpose. When reading `onClick={handleClick}` in JSX far from the definition, you cannot infer behavior. Naming it after the action (e.g., `toggleSidebarOnBackgroundClick`) makes the code self-documenting at every call site.

**Guideline:** Name event handlers after the action they perform (`toggleSidebar`, `closePanel`, `submitForm`) rather than the event they respond to (`handleClick`, `onPress`).

---

## 4. Forward Props in Wrapper Components

The `Link` component originally accepted only three explicit props (`className`, `href`, `ref`). When used as a child of Radix UI's `Slot` pattern (via `asChild`), Radix tried to merge additional props (data attributes, tooltip event handlers like `onMouseEnter`, `onFocus`) onto the element — but they were silently dropped because the component didn't forward them. This broke tooltip functionality entirely. The fix was to use `ComponentProps<typeof RouterLink>` and spread `...props` onto the underlying element.

```tsx
// Before: props are dropped, tooltips break
export const Link = ({ children, href, className, ref }: Props) => {
  return <RouterLink ref={ref} to={href} className={className}>{children}</RouterLink>;
};

// After: all props forwarded, Radix composition works
export const Link = ({ children, href, onClick, ...props }: Props) => {
  return <RouterLink to={href} {...props}>{children}</RouterLink>;
};
```

**Guideline:** Wrapper components should always forward unknown props to their underlying element via rest spreading, especially when used with composition patterns like Radix `asChild`.

---

## 5. Consider UX Impact Before Removing Navigation Affordances

The initial version removed the sidebar header (which contained the registry name and a collapse button) without ensuring mobile users could still access navigation. On mobile, the sidebar renders as a `Sheet` controlled by a trigger — removing the trigger left mobile users with no way to open the sidebar at all. Changes to navigation chrome must be evaluated across all viewport sizes.

**Guideline:** When removing or restructuring navigation UI, verify that all viewport sizes (mobile, tablet, desktop) still have a way to access every navigation path.

---

## 6. Be Cautious with Click-to-Dismiss Sidebar Patterns

Adding a click handler on the sidebar background to toggle/collapse the sidebar is a non-standard interaction. It risks accidental dismissal when users misclick, and it has unclear accessibility implications. The pattern was allowed to proceed with the understanding it would be reverted if it caused problems. Novel interactions should be treated as experiments — ship them, but have a clear rollback plan and monitor for user complaints.

**Guideline:** When introducing non-standard interactions (click-to-dismiss, gesture-based navigation), treat them as experiments with explicit rollback criteria rather than permanent features.

---

## 7. Write Tests for Shared Primitive Components

The `Link` component is a shared primitive used across the entire application, but it had no tests. The prop-forwarding bug (learning #4) would have been caught by a test verifying that arbitrary props are passed through to the rendered element. When a low-level component is used everywhere, even a small breakage (like swallowing props) cascades across the entire app.

**Guideline:** Shared primitive components (`Link`, `Button`, layout wrappers) that are used application-wide should have tests verifying their prop-forwarding and composition behavior.
