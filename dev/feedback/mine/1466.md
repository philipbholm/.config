# Learnings from PR #1466
This PR implemented PII (personally identifiable information) masking on patient detail pages, hiding sensitive fields by default with per-field reveal buttons and audit logging, to comply with the Norwegian Patient Medical Records Act.

---

## 1. Enforce Security-Sensitive Controls on the Backend, Not the Frontend
Client-side masking and logging can be trivially bypassed by a sophisticated user who modifies the JavaScript source. When a feature exists for regulatory compliance (such as audit logging of PII access), the enforcement must happen server-side. Frontend-only controls provide a false sense of security and do not satisfy legal requirements. If the API still returns PII data unconditionally, the masking is cosmetic.

**Guideline:** Any access control or audit logging required for compliance must be enforced at the API/backend layer, not in client-side code that users can modify.

---

## 2. Block the Protected Action When Its Audit Trail Fails
When a reveal-and-log flow encounters a logging failure (network error, auth error, backend outage), the UI should not proceed with revealing the data. Allowing the action to succeed while the audit write fails creates gaps in compliance logs and violates the requirement that "all reveal actions are logged." The action and its audit record must succeed or fail together.

**Guideline:** Treat the audit log write as a prerequisite for the protected action: only complete the action after the log mutation succeeds.

---

## 3. Use the Correct ARIA Roles for Loading States
Using `role="status"` for a loading indicator is semantically incorrect. Loading states should use `role="progressbar"` (or a spinner component) to convey the right meaning to assistive technologies. Additionally, showing a visual loading indicator (spinner) rather than plain text like "Loading..." provides better UX feedback during async operations.

**Guideline:** Use `role="progressbar"` and a visual spinner for loading states; reserve `role="status"` for polite live-region announcements of completed state changes.

---

## 4. Write Tests for Security-Critical Utility Functions
Utility functions that handle security-relevant behavior (such as clearing session storage on logout to remove cached PII visibility state) must have dedicated tests. If the logout cleanup silently breaks, users could see stale PII visibility state from a previous session, undermining the security model. These functions are small and easy to test, so there is no excuse for skipping coverage.

**Guideline:** Always add unit tests for utility functions involved in auth, session cleanup, or security state management.

---

## 5. Test Frontend Components That Have Security and Interaction Logic
When a component manages sensitive user interactions (masking/revealing PII, calling audit mutations, toggling aria attributes), it needs dedicated tests. The critical paths include: initial masked state, reveal triggering the API call, hide not triggering the API call, error handling on API failure, and correct accessibility attributes. Skipping these tests means the most user-facing part of the security feature has no automated verification.

**Guideline:** For components with security-sensitive interaction flows, write tests covering the happy path, error handling, and accessibility behavior before merging.

---

## 6. Prefer Strict Assertions Over Partial Matching in Database Verification Tests
Using `toMatchObject()` in tests that verify database records allows unexpected extra fields to pass silently. When testing that an audit log was stored correctly, use `toEqual()` with explicit matchers (e.g., `expect.any(String)` for generated IDs) to assert the full shape of the record. This catches schema drift and unintended data leaking into records.

**Guideline:** Use `toEqual()` with explicit matchers for database record assertions; reserve `toMatchObject()` for cases where you intentionally want partial matching.

---

## 7. Follow Codebase Naming Conventions in Test Files
Using lowercase in test `describe` blocks (e.g., `"log pii access"`) when the codebase convention is PascalCase (e.g., `"LogPiiAccess"`) creates inconsistency that makes tests harder to find and cross-reference with the source code. Small deviations compound across a growing test suite.

**Guideline:** Match test describe block naming to the codebase convention (typically PascalCase matching the module or class under test).

---

## 8. Evaluate Existing UX Patterns Before Inventing New Ones
Before building a new UI pattern for a feature (such as per-field reveal toggles), check whether the codebase already has an established pattern for similar functionality (e.g., a codelist-level visibility toggle). Reusing existing patterns is more user-friendly because users already understand them, and it reduces maintenance burden. Introducing a novel interaction when a familiar one exists creates unnecessary cognitive load.

**Guideline:** Search the codebase for existing UI patterns that solve the same problem before designing a new interaction; prefer consistency over novelty.
