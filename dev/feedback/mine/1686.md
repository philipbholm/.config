# Learnings from PR #1686

This PR fixed flaky integration tests for category structures by making assertions order-independent using `expect.arrayContaining()` combined with `toHaveLength()`.

---

## 1. Fix Flaky Ordering at the Source, Not Just in the Test

When a test is flaky because results come back in non-deterministic order, the first question should be whether the **production code** needs a stable ordering guarantee -- not just whether the test can be made lenient. If categories are created in a specific order and users expect that order, sorting by a random UUID in the projection layer silently shuffles domain data. Making the test order-independent masks the real bug: the read path should preserve insertion order (e.g., by sorting on `createdAt`). Only make tests order-independent when the domain genuinely has no ordering requirement.

**Guideline:** Before making a test order-independent, verify that the system under test has no ordering contract with its consumers; if it does, fix the ordering in the production code instead.

---

## 2. Pair `expect.arrayContaining` with a Length Check

Using `expect.arrayContaining()` alone is a subset matcher -- it passes even if the actual array has extra elements not in the expected list. Always pair it with a `.toHaveLength()` assertion to ensure exact membership without enforcing order.

```typescript
// Bad: passes even if categories has 20 extra items
expect(result.categories).toEqual(expect.arrayContaining([{ value: "A+" }, { value: "B+" }]));

// Good: order-independent but exact membership
expect(result.categories).toEqual(expect.arrayContaining([{ value: "A+" }, { value: "B+" }]));
expect(result.categories).toHaveLength(2);
```

**Guideline:** Every use of `expect.arrayContaining()` should be accompanied by a `toHaveLength()` check to prevent false-passing tests when extra elements appear.
