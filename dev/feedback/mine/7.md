# Learnings from PR #7
This PR added TFLint as a CI linter for Terraform files, but was closed before merging because the infrastructure was not yet mature enough to support the change.

---

## 1. Assess Blast Radius Before Renaming Shared Infrastructure Resources
Renaming an IAM role or other shared resource (such as the role assumed by GitHub Actions) has cascading effects across every workflow and pipeline that references it. A seemingly simple name change in one Terraform file can silently break all CI/CD jobs if the corresponding references are not updated in lockstep.

**Guideline:** Before renaming any shared infrastructure resource, audit every consumer of that resource and update all references in a single atomic change.

---

## 2. Apply Infrastructure Changes Through the Pipeline, Not Around It
When a CI pipeline depends on infrastructure that must be provisioned before the pipeline itself can run, there is a bootstrapping problem. Manually applying Terraform changes outside the pipeline to "unblock" it creates drift and bypasses the review and safety mechanisms the pipeline provides.

**Guideline:** Design infrastructure bootstrapping steps as explicit, documented prerequisites that are applied through a controlled process, not ad-hoc manual runs.

---

## 3. Do Not Introduce Tooling Before the Foundation Is Ready
Adding linting, formatting, or quality gates to a codebase that is still in early flux creates friction without proportional benefit. If the underlying infrastructure and workflows are not yet stable, the tooling PR will either be blocked by unrelated issues or require constant rebasing.

**Guideline:** Defer CI tooling additions until the foundational infrastructure and core workflows are stable enough that the tooling can be merged and run without special workarounds.

---

## 4. Scope PRs to Avoid Mixing Tooling With Infrastructure Changes
This PR combined the addition of a linting tool with modifications to shared infrastructure files (OIDC provider, IAM roles). Mixing concerns makes review harder and increases the chance of unintended side effects in unrelated systems.

**Guideline:** Keep tooling/CI PRs strictly separate from infrastructure resource changes so each can be reviewed, tested, and merged independently.
