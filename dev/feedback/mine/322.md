# Learnings from PR #322
A broad styling overhaul aligning UI components with the design system, updating Tailwind config, unifying table styles, fixing locale-dependent formatting bugs, and cleaning up icons.

---

## 1. Never Compare Domain Values Against Translated Strings
When checking entity types, status codes, or any logical identifiers, always compare against raw constants or enum values -- never against translated/localized strings. Translations change per locale and can be updated at any time, which means a comparison that works in English will silently break in Norwegian or any other language. This was flagged in the history page where `entityType` was being checked against translation output instead of a stable identifier.

**Guideline:** Always compare logical values (entity types, statuses, categories) against language-independent constants or enums, and only use translated strings for display purposes.

---

## 2. Keep Styling PRs Focused to Ease Review
This PR touched 85 files across many concerns: Tailwind configuration, table components, icons, translations, layout backgrounds, border radii, and a locale-formatting bug fix. Bundling unrelated changes into a single "styling" PR makes it harder for reviewers to give thorough feedback and increases the risk that bugs slip through. Smaller, scoped PRs (e.g., one for Tailwind config, one for table unification, one for icon cleanup) yield better review quality.

**Guideline:** Split large cosmetic overhauls into focused PRs grouped by concern so reviewers can evaluate each change in isolation.

---

## 3. Restrict Tailwind to Design System Tokens Only
When constraining a Tailwind config to only allow design-system values (by not extending the default theme), ensure that every usage site has been audited. Removing fallback defaults can silently break components that relied on built-in Tailwind classes. The PR description notes removing extended styles so "only design system styles are possible," which is a good enforcement pattern but requires thorough verification.

**Guideline:** When restricting a utility framework to a design-system subset, audit all existing class usage first to prevent regressions from removed defaults.

---

## 4. Fix Locale-Dependent Bugs Alongside Their Root Cause
The PR fixed a bug where history entries were not formatted correctly in Norwegian. Locale-sensitive formatting issues (dates, numbers, currency) tend to recur if the underlying approach is ad hoc. When fixing one locale bug, check whether the same pattern exists elsewhere and consider centralizing locale-aware formatting into shared utility functions.

**Guideline:** When fixing a locale-specific formatting bug, search for the same pattern across the codebase and centralize the formatting logic to prevent similar issues in other locales.

---

## 5. Unify Repeated UI Patterns Into Shared Components
The PR updated collaborators, groups, and history tables to use the same components as the dataset table. When multiple views implement the same visual pattern independently, they drift apart over time and require coordinated updates. Extracting a shared table component earlier would have avoided the need for this reconciliation pass.

**Guideline:** When you notice two or more views implementing the same UI pattern, extract it into a shared component immediately rather than allowing divergence.
