# Learnings from PR #235
This PR consolidated multiple database setup scripts into a single shell script for the mods service to simplify the local development setup process.

---

## 1. Use Safe Shell Scripting Practices
Shell scripts that silently continue after errors can corrupt database state, leave partial setups, or produce confusing downstream failures. Defensive options like `set -euo pipefail` cause scripts to fail fast on errors, undefined variables, and pipe failures, making problems immediately visible instead of silently propagating.

**Guideline:** Always start shell scripts with `set -euo pipefail` and review the safe shell scripting principles at sipb.mit.edu/doc/safe-shell before shipping production or setup scripts.

---

## 2. Survey Existing Implementations Before Creating New Scripts
When a codebase already contains a script or utility that does something similar, creating a new one leads to duplicated logic, divergent behavior, and confusion about which is canonical. Reviewers will question whether the new script was necessary or whether the existing one should have been extended.

**Guideline:** Before writing a new script or tool, search the repository for existing implementations that solve the same or a related problem, and prefer extending or replacing them explicitly over creating parallel solutions.

---

## 3. Prefer Typed Languages Over Raw Shell for Database Setup Scripts
Shell scripts that construct SQL or perform database operations have no compile-time safety: column names, table references, and data types are all unchecked strings. Using a typed language (e.g., TypeScript with an ORM or query builder like Prisma) catches schema mismatches at compile time and makes refactoring safer when the database schema evolves.

**Guideline:** Write database setup and seed scripts in a typed language with ORM/client support rather than raw shell or SQL, so that schema changes surface as type errors instead of runtime failures.
