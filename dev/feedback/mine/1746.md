# Learnings from PR #1746

Replaced hardcoded `localhost` URLs in E2E tests with configurable environment variables and removed redundant URL prefixes from `page.goto()` calls to support git worktree workflows.

---

## 1. Name Environment Variables After Their Specific Service

When introducing environment variables for URLs or configuration, use names that clearly identify the exact service they point to. A generic name like `BASE_URL` is ambiguous in a system with multiple services (frontend, API, database). The reviewer flagged `BASE_URL` as too vague and it was renamed to `FRONTEND_BASE_URL`. In a monorepo with many services, generic names lead to confusion about which service a variable configures, and they risk colliding with other variables as the system grows.

**Guideline:** Prefix environment variable names with the service or component they configure (e.g., `FRONTEND_BASE_URL`, `E2E_API_URL`, `POSTGRES_URL`) rather than using generic names.

---

## 2. Leverage Playwright's baseURL Instead of Hardcoding Host in page.goto()

When Playwright config already defines a `baseURL`, passing absolute URLs like `` `http://localhost:3001${path}` `` to `page.goto()` bypasses that configuration entirely. This means any environment-level URL override has no effect on the test navigation, creating a hidden coupling to a specific host. By passing only the path to `page.goto()`, Playwright automatically prepends the configured `baseURL`, making tests portable across environments with a single configuration change.

```typescript
// Bad: hardcoded host ignores Playwright's baseURL config
await page.goto(`http://localhost:3001${dashboardPath}`);

// Good: relies on baseURL from playwright.config.ts
await page.goto(dashboardPath);
```

**Guideline:** When using Playwright's `baseURL` config, pass only relative paths to `page.goto()` so all test navigation respects the centralized URL configuration.
