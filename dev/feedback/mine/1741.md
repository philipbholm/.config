# Learnings from PR #1741

Updated the NOBAREV registry template with restructured episodes, forms, and modular file organization, plus test data generation and E2E tests.

---

## 1. Make Test Data Deterministic by Seeding Randomness

When tests rely on randomly generated data, assertions like "length greater than 0" or approximate value checks become non-deterministic. Even if they pass most of the time, they create a maintenance burden because any failure requires investigating whether the test is flaky or genuinely broken. Seed your random generators so that test data is reproducible across runs.

**Guideline:** Always seed random data generators in tests so that every assertion can be exact and every failure is meaningful.

---

## 2. Declare Dependent Types After Their Dependencies

When type B depends on type A, declare A first. This lets readers understand the code top-to-bottom without jumping around. The same principle applies to constants and derived values -- the code should read like an article where each concept builds on the previous one.

```typescript
// Good: dependency first, then derived type
const DICTIONARY = { ... } as const;
type Translation = typeof DICTIONARY[Locale];

// Bad: derived type appears before the thing it derives from
type Translation = typeof DICTIONARY[Locale];
const DICTIONARY = { ... } as const;
```

**Guideline:** Order type and constant declarations so that dependencies appear before the things that depend on them.

---

## 3. Place Helper Code Below the React Component

In React files, keep the component itself as the first major block after imports and its props interface. Move helper types, constants, dictionaries, and utility functions below the component. This lets readers find the exported component immediately without scrolling past implementation details.

**Guideline:** Structure React files as: imports, props interface, component, then all supporting code below.

---

## 4. Use `as const` and Derive Types from Data Objects

Instead of manually defining a type that mirrors a data object's shape, use `as const` on the object and derive the type with `typeof`. This eliminates duplication between the data and its type, and ensures the type always stays in sync with the actual values. Use consistent naming: the full object is a "dictionary", a per-language entry is a "translation".

```typescript
// Good: single source of truth
const DICTIONARY = {
  en: { title: 'Create Registry' },
  nb: { title: 'Opprett register' },
} as const;

type Translation = (typeof DICTIONARY)[Locale];
```

**Guideline:** Prefer `as const` with `typeof` over manually duplicated type definitions for static data objects.

---

## 5. Throw on Invalid States Instead of Failing Silently

When code encounters a state that should be impossible (e.g., an unrecognized enum value in a configurator), throw an error rather than silently returning or skipping. Silent failures hide bugs and make them surface far from the root cause, turning a simple fix into a debugging session.

**Guideline:** Use explicit throws (or exhaustive checks) for invalid states so problems are caught immediately at their source.

---

## 6. Remove Development Artifacts Before Merging

Files created as scaffolding or AI prompts during development (e.g., a `DESCRIPTION.md` used as Claude input) become confusing once the real implementation exists alongside them. They create ambiguity about which is the source of truth and add maintenance burden. Remove them before the PR is merged.

**Guideline:** Audit the changeset for files that served a development-time purpose but are redundant in the final codebase.

---

## 7. Justify Configuration Changes in the PR

When changing configuration values (timeouts, feature flags, test settings), include the rationale in the PR or a comment. Reviewers cannot evaluate whether a change is correct without understanding why it was made. In this case, a test timeout increase turned out to be unnecessary once the underlying issue was resolved.

**Guideline:** Always state the reason for configuration changes so reviewers can assess necessity and future maintainers understand intent.
