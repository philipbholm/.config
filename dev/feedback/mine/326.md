# Learnings from PR #326
Styling improvements to a component library, including Button API changes, icon handling, focus/error states, hover animations, and design token usage.

---

## 1. Keep Specialized Components Separate Instead of Adding Conditionals
When a component like Button needs to support an icon-only variant, resist the urge to merge it into the base component with conditionals. Each conditional added to a shared component is a step toward maintenance hell -- the abstraction becomes harder to reason about, test, and extend. If two variants have meaningfully different rendering logic, they deserve separate components (e.g., Button and IconButton).

**Guideline:** If adding a feature to a shared component requires a conditional branch that changes its rendering shape, extract a separate component instead.

---

## 2. Preserve Existing Functionality During Refactors
When refactoring a component's API or internal structure, audit every existing feature (loading states, disabled behavior, accessibility attributes) and ensure none are silently dropped. Reviewers should not have to ask "what is the plan for loading now?" -- the PR should either preserve it or explicitly document the removal.

**Guideline:** Before submitting a component refactor, diff the old and new prop interfaces and verify every previously supported feature is either retained or intentionally removed with justification.

---

## 3. Use Props Over Children for Design System Consistency
Passing icons and text as explicit props (rather than as arbitrary children) lets the component enforce consistent spacing, ordering, and sizing. Children-based APIs offer flexibility but sacrifice the guardrails that prevent visual inconsistency across a design system.

**Guideline:** In design system components, prefer typed props for structured content (icons, labels) and reserve children for truly freeform content areas.

---

## 4. Use PascalCase for React Node Props
Props that accept React nodes (JSX elements) should use PascalCase naming (e.g., `TrailingIcon` rather than `trailingIcon`). This convention signals to consumers that the prop expects a component or rendered element rather than a primitive value, making the API self-documenting.

**Guideline:** Name any prop that accepts `React.ReactNode` or `React.ComponentType` with a leading capital letter to distinguish it from primitive props.

---

## 5. Ensure Embedded Sub-Components Inherit Parent Color Tokens
When a component embeds another component (e.g., a Button with a loading spinner), the embedded component must inherit or reference the parent's color tokens. A loader hardcoded to blue inside a button themed with a baseline/dark color creates a jarring visual mismatch that is easy to miss in code but obvious in the UI.

**Guideline:** Use `currentColor` or the parent's design token variables for any embedded sub-component's colors rather than hardcoding separate color values.

---

## 6. Make Styling PRs Reviewable Beyond the Diff
Large styling PRs that touch many files are inherently hard to review from diffs alone -- reviewers cannot visualize the result. Include before/after screenshots, Storybook links, or short screen recordings so reviewers can evaluate the actual visual outcome rather than guessing from CSS changes.

**Guideline:** For any PR that primarily changes visual styling, attach visual evidence (screenshots or Storybook deploy links) to the PR description.

---

## 7. Do Not Adopt External Library Patterns Without Evaluating Fit
Copying an API pattern from a popular library (e.g., shadcn) does not automatically make it the right choice for your codebase. External libraries optimize for different constraints (generality, tree-shaking, framework agnosticism) than an internal design system. Evaluate whether the pattern serves your team's specific consistency and usability goals before adopting it.

**Guideline:** When borrowing an API design from an external library, document why it fits your use case -- "shadcn does it" is not sufficient justification.
