# Learnings from PR #1106
This PR formatted ICD-10 medical codes for correct display (e.g., inserting dots) and added a database field for tracking unspecified diagnosis codes.

---

## 1. Always Disconnect Database Clients in Scripts
Scripts that establish database connections (e.g., Prisma clients in seed scripts) must explicitly disconnect before exiting. If the connection is not closed, the Node.js process will hang indefinitely because the open connection keeps the event loop alive. This wastes resources, blocks CI pipelines, and can lead to connection pool exhaustion in environments with connection limits.

**Guideline:** Every script that creates a database client must call its disconnect/close method in a finally block before the process exits.

---

## 2. Add Test Assertions When Introducing New Data Fields
When adding a new field or changing data representation, existing tests should be updated with assertions that verify the new behavior works as intended. Simply running old tests without new assertions gives false confidence -- the tests pass, but they never actually check that the new field is populated correctly or that the formatting change produces the expected output. If the feature "matters enough to ship," it matters enough to assert on.

**Guideline:** For every new or modified data field, add at least one test assertion that explicitly validates the new expected value or format.
