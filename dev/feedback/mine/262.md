# Learnings from PR #262
Replaced a projects table view with a card grid layout, introducing a Card component and utility functions inspired by shadcn/ui.

---

## 1. Delay Abstractions Until Multiple Consumers Exist
Creating sub-components (e.g., CardHeader, CardContent, CardFooter) before there are multiple distinct card layouts leads to speculative abstractions that may not fit future needs. When only one usage exists, inline the styling directly and extract shared components only after a second or third consumer reveals the real abstraction boundary.

**Guideline:** Do not split a component into composable sub-components until at least two concrete use cases exist to validate the abstraction.

---

## 2. Avoid Adding Utility Libraries Preemptively
Introducing a class-merging utility like `cn` (wrapping `tailwind-merge` and `clsx`) adds a dependency and conceptual overhead. If the codebase convention is to not override component styles via `className`, the utility solves a problem that does not yet exist. Add such utilities only when style conflict resolution is a demonstrated, recurring pain point.

**Guideline:** Do not add utility libraries or wrapper functions until the problem they solve has occurred more than once in practice.

---

## 3. Understand When displayName Is Actually Required
Setting `displayName` on a React component is only necessary when `forwardRef` is used, because `forwardRef` wraps the component and hides its name in DevTools. For standard function components, React infers the display name from the function declaration automatically. Adding it unnecessarily is dead code that signals a misunderstanding of the API.

**Guideline:** Only set `displayName` on components wrapped with `forwardRef` or created via higher-order functions that obscure the original name.

---

## 4. Use Skeleton Screens Instead of Spinners for Content Loading
When a page loads content like a grid of cards, a generic spinner or blank area provides no spatial hint about what is coming. Skeleton screens (placeholder shapes that mirror the layout of the real content) reduce perceived wait time and prevent layout shifts. Existing patterns in the codebase should be referenced for consistency.

**Guideline:** Use skeleton loading states that approximate the shape of the final content, and follow the loading pattern already established in other pages of the application.

---

## 5. Prefer Design System Spacing Tokens Over Arbitrary Values
Using arbitrary pixel values (e.g., `h-[150px]`) in Tailwind defeats the purpose of a constrained spacing scale and increases CSS bundle size because each unique arbitrary value generates a new utility class. Standard spacing tokens like `h-40` keep the design consistent and the output smaller.

**Guideline:** Always use Tailwind's built-in spacing scale instead of arbitrary bracket values, and verify dimensions against the design source (e.g., Figma) before committing.

---

## 6. Remove Unused Dependencies Before Merging
Leaving dependencies in `package.json` that were added during experimentation but are no longer used inflates install times, increases potential attack surface, and confuses future developers about what the project actually depends on. This is easy to miss during iterative development.

**Guideline:** Before opening a PR for review, audit `package.json` for any dependencies that were added during development but are no longer imported anywhere in the final code.

---

## 7. Reuse Existing Components Before Creating New Ones
Before building a new loading indicator or any UI primitive, check if the codebase already provides one. Duplicating existing components leads to visual inconsistency and maintenance burden. When in doubt, search the component directory or ask teammates.

**Guideline:** Search the existing component library for a matching primitive before creating a new one, and only diverge if the existing component genuinely cannot serve the use case.
