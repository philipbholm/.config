# Learnings from PR #243
This PR added a `description` field to the Project model and a new `ProjectMetadata` table for key-value project information, with frontend forms and backend validation.

---

## 1. Wrap Related Database Writes in a Transaction
When creating a parent record and its associated child records (e.g., a project and its metadata rows), performing them as separate database calls risks partial writes -- the parent could be created while the child inserts fail, leaving the data in an inconsistent state. Use Prisma's nested create or an explicit transaction to ensure atomicity.

**Guideline:** Always wrap multi-table writes that must succeed or fail together in a single database transaction or nested create operation.

---

## 2. Validate on Both Frontend and Backend
Frontend validation (e.g., Zod schemas in forms) only protects users who go through the UI. Anyone calling the API directly bypasses those checks entirely. If the backend does not independently validate inputs, malformed or malicious data can be persisted.

**Guideline:** Duplicate critical validation rules on the backend so that API consumers who bypass the UI are still subject to the same data integrity constraints.

---

## 3. Reuse Shared Type Definitions Instead of Reinventing Them
When a utility type (like `PageProps`) already exists in the codebase, creating ad-hoc inline types for the same purpose leads to inconsistency and makes refactoring harder. Discoverability of existing types is a team responsibility.

**Guideline:** Before defining a new type, search the codebase for existing shared types that cover the same shape, and extend or reuse them.

---

## 4. Keep Optional Attributes in the Metadata Layer
When a data model has a core set of always-present fields and a variable set of optional fields, placing optional attributes directly on the core table blurs the boundary. Reviewers suggested that truly optional project attributes belong in the metadata key-value table, not as nullable columns on the main model.

**Guideline:** Use a metadata or extension table for optional, template-driven fields rather than adding nullable columns to the primary entity table -- unless the field is universally displayed.

---

## 5. Make GraphQL Nullability Match Actual Usage
If the only client always sends an array (even if empty), but the schema allows null, the contract is misleading. Conversely, if external API consumers should not be forced to send an empty array, the field should be nullable. The schema nullability should reflect the intended usage pattern.

**Guideline:** Align GraphQL field nullability (`[Type!]` vs `[Type!]!`) with how callers are actually expected to use the API, and document the rationale.

---

## 6. Prefer Union Types Over Enums in TypeScript
TypeScript union types (`type FieldType = "STRING" | "DATE"`) integrate more naturally with the type system than `enum` declarations. Unions are simpler to use in type narrowing, do not produce runtime artifacts, and avoid the mapping boilerplate that enums require when crossing service boundaries (e.g., gRPC to internal models).

**Guideline:** Use string union types instead of TypeScript enums for fixed sets of values, especially when the values must be mapped across service layers.

---

## 7. Avoid Unsafe Type Casting -- Let the Type System Prove Correctness
Using `as` type casts to force a value into a type bypasses the compiler's safety checks. If the value actually comes from a validated source (e.g., a GraphQL enum), remove the cast and let the types flow through. If it does not come from a validated source, add a runtime type guard instead of casting.

**Guideline:** Never use `as` casts to silence type errors; instead, either fix the upstream type so it flows correctly or add a runtime type guard that narrows the type safely.

---

## 8. Add Tests Whenever Repository Methods Change
Modifying repository methods (e.g., changing what `getProject` or `getMany` returns) without updating or adding corresponding tests means regressions can slip through undetected. The review explicitly flagged missing tests for changed repository code.

**Guideline:** Every change to a repository or data-access method must be accompanied by tests that verify the new behavior, even if the change seems small.

---

## 9. Use JSDoc Comments for Inline Documentation
Placing format descriptions or usage notes in plain comments above a function makes them invisible to IDE tooling. Using JSDoc syntax (`/** ... */`) instead causes the documentation to appear in hover tooltips and autocomplete, making it far more discoverable.

**Guideline:** Document function contracts, expected formats, and constraints using JSDoc (`/** */`) so they surface in IDE tooltips rather than being buried in plain comments.

---

## 10. Minimize Cross-Service Domain Duplication
When a service (e.g., mods) duplicates domain models from another service (e.g., projects), every future change to the source model requires a parallel update in the consuming service. This coupling grows over time and makes refactoring expensive. Prefer delegating requests to the owning service rather than duplicating its domain concepts.

**Guideline:** When two services share domain concepts, delegate operations to the owning service rather than duplicating its models, so changes propagate from a single source of truth.
