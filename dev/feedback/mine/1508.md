# Learnings from PR #1508
Large-scale rename/refactor of domain concepts across 277 files in the monorepo, including frontend, backend, and Prisma schema changes.

---

## 1. Separate Formatting Changes from Logical Changes
When a tool (e.g., Prisma formatter, Prettier) reformats an entire file, the resulting diff drowns out the meaningful changes in the PR. Reviewers cannot easily distinguish reformatting noise from intentional code changes, which slows review and increases the risk of missing real issues.

**Guideline:** If a formatter produces a whole-file diff, extract that formatting into its own dedicated PR merged before or after the substantive changes.

---

## 2. Avoid Redundant Type Annotations in TypeScript
When TypeScript can infer a type from context (e.g., return types of arrow functions passed to well-typed higher-order functions, or variables initialized with typed values), explicitly annotating the type adds visual clutter without improving safety. It also creates a maintenance burden because two sources of truth must be kept in sync.

**Guideline:** Omit explicit type annotations where TypeScript inference already provides the correct type, and rely on strict compiler settings to catch any gaps.

---

## 3. Understand Why a Tool Produces Unexpected Changes Before Committing Them
When a formatter or code generator suddenly produces large diffs in files you did not intend to modify, pause and investigate the root cause. Blindly committing tool-generated changes makes it harder for reviewers to trust the diff and may introduce unintended side effects.

**Guideline:** Before committing unexpected tool-generated changes, identify why they appeared and document the cause in the PR description so reviewers have context.

---

## 4. Annotate Renames Clearly in the PR Description
GitHub does not always detect renames, especially when the file content has also changed. In a large rename PR, reviewers may waste time examining files they believe are entirely new when they are actually moved or renamed. Calling this out upfront saves review cycles.

**Guideline:** When a PR involves file renames that the diff tool may misrepresent, state which files are renames (not new) in the PR description so reviewers can adjust their review strategy.

---

## 5. Keep Large Refactors Minimal and Reviewable
A 277-file PR is inherently difficult to review thoroughly. The more unrelated changes (formatting, renames, logic tweaks) that are bundled together, the higher the chance that a real bug slips through. Splitting by concern -- pure renames in one PR, formatting fixes in another, logic changes in a third -- makes each PR focused and reviewable.

**Guideline:** When a refactor touches many files, break it into the smallest PRs that can each be merged independently, grouping by the type of change (renames, formatting, logic).
