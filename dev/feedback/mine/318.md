# Learnings from PR #318
Adding int, float, date, and datetime value types with frontend formatting (Luxon), backend business logic restructuring, and dataset table UI improvements.

---

## 1. Place Type Packages in devDependencies, Not dependencies
TypeScript type packages (e.g., `@types/*`) are only needed at build/compile time, not at runtime. Placing them in `dependencies` bloats the production install and signals to other developers that these packages are required at runtime, which is misleading. The distinction between `dependencies` (runtime) and `devDependencies` (build/dev time) should be maintained precisely.

**Guideline:** Always install type-only packages as devDependencies since they are stripped during compilation and have no runtime effect.

---

## 2. Do Not Expose UI Options That Have No Effect
If a user-facing control (e.g., a timezone selector) does not actually propagate its value to application state or the backend, it should not be shown to the user. Displaying non-functional controls creates false expectations, leads to bug reports, and erodes user trust. It is better to omit the feature entirely and add it later when fully wired up.

**Guideline:** Remove or hide any UI control whose selected value is not consumed by the application until the full round-trip is implemented.

---

## 3. Validate Shared Component Changes Across All Consumers
When modifying a shared/reusable component (e.g., a `Select` component used in multiple places), the change can have unintended side effects in other parts of the application. Each consumer may rely on specific styling, behavior, or prop defaults. Failing to check all usage sites can introduce visual regressions or broken interactions elsewhere.

**Guideline:** Before merging changes to a shared component, search for all import sites and manually verify that each consumer still renders and behaves correctly.

---

## 4. Discuss and Review Data Access Patterns Before Merging
When the sequence of database operations in a data repository or adapter changes (e.g., how records and cells are inserted), this can affect data integrity, transaction boundaries, and error handling. These patterns deserve explicit discussion because they form the foundation of data consistency and are expensive to fix retroactively.

**Guideline:** Flag any changes to the order or structure of database write operations for explicit team review before merging.

---

## 5. Decide on a Canonical Date/Time Representation Early
When introducing date and datetime types, a key architectural question is whether to store and transmit dates as ISO strings, Unix timestamps, or library-specific objects. Mixing representations across the frontend and backend creates parsing ambiguity and timezone bugs. Settling on a canonical format (e.g., ISO 8601 strings) prevents a class of serialization and comparison issues.

**Guideline:** Establish and document a single canonical date/datetime string format for storage and API transport before implementing date-related features.

---

## 6. Extract Complex Object Construction into Helper Functions
When constructing complex objects (e.g., building a data record from form inputs with type coercion and validation), inline construction in a use-case function makes the code harder to read, test, and reuse. Extracting this into a dedicated helper function isolates the transformation logic, makes unit testing straightforward, and keeps the orchestration layer clean.

**Guideline:** Move non-trivial object construction or data transformation logic into named helper functions that can be tested independently.

---

## 7. Use Consistent Numeric Formatting in Table Columns
In tabular data displays, showing different numbers of decimal places per row (e.g., "3" vs "3.14" vs "3.100") makes visual comparison across rows difficult. Users scanning a column expect uniform formatting. Right-aligning numbers and using a fixed number of decimal places per column creates a scannable, professional data presentation.

**Guideline:** Format all numeric values in a table column to the same number of decimal places and use right-alignment to enable easy visual comparison across rows.

---

## 8. Avoid O(n^2) Lookups -- Use Maps for Repeated Access by ID
When rendering table cells, doing a `.find()` inside a loop to look up a variable by ID for each cell creates O(n*m) complexity. As datasets grow to hundreds or thousands of rows and columns, this causes noticeable rendering lag. Pre-building a `Map` keyed by ID before the loop converts the operation to O(n+m), which remains fast at scale.

**Guideline:** When you need to look up items by ID inside a loop, build a `Map` (or object keyed by ID) before the loop instead of using `.find()` or `.filter()` repeatedly.

---

## 9. Revisit Complex Insert Patterns Collaboratively
When a data insertion flow involves multiple steps (e.g., creating a record, then inserting cells one by one), there may be a simpler or more performant approach such as batch inserts or a single structured write. Complex multi-step inserts are harder to make transactional and more prone to partial-failure states. These patterns benefit from collaborative design review.

**Guideline:** When implementing multi-step data insertion flows, schedule a brief design discussion to evaluate whether batch operations or simpler patterns could replace sequential inserts.
