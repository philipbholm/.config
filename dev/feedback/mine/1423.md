# Learnings from PR #1423
This PR added a new registry template (Nobarev) to the registries service, including template definitions, test-data generation, and updates to the template index.

---

## 1. Clean Up Legacy Naming When Touching Adjacent Code
When modifying a file that contains outdated naming conventions (e.g., redundant suffixes like `-registry` in a module already scoped to registries), take the opportunity to remove the cruft rather than leaving it in place. Stale naming accumulates as tech debt and makes future readers question whether the inconsistency is intentional.

**Guideline:** If you are already changing a file, fix any nearby naming inconsistencies so the codebase stays internally consistent.

---

## 2. Write Unit Tests for Data-Generation Logic
Test-data generators may look straightforward, but they encode domain assumptions (field formats, value ranges, relational constraints) that can silently break downstream consumers if they drift. Even when the generator is "just" producing fake data, unit tests lock in those assumptions and catch regressions early.

**Guideline:** Every module that programmatically produces structured data -- including test fixtures and seed generators -- should have its own unit tests validating shape, constraints, and edge cases.
